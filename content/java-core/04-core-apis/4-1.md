## Title
String & String Pool

## Summary
`String` is an **immutable**, **final** class backed by a `byte[]` + coder flag (Java 9+, Compact Strings). The **String Pool** is a special area in the heap holding unique string literals (Flyweight pattern). Literals go into the pool automatically; `new String(...)` is **outside** the pool; `.intern()` explicitly pools a string. `==` compares references, `.equals()` compares content.

## Deep Dive
## Immutability — why and what it gives

Once created, a String **never changes**. Any apparently-modifying method (`concat`, `replace`, `substring`, `toUpperCase`) returns a **new** object. Benefits:

- **Thread safety** — shareable across threads without synchronisation.
- **Security** — a string used for a class name, URL, or file path cannot be mutated after a check.
- **Cached hashCode** — `hashCode()` is computed once and stored in a field. That's why `String` is the ideal `HashMap` key.
- **String Pool** — the JVM can safely reuse instances.
- **Correct class loading** — a class name won't change between verification and use.

## String Pool (Flyweight)

A special heap area (pre-Java-7 it was in **PermGen**; since Java 7 in the main heap). Stores unique strings.

**What the pool receives automatically**:
- String literals (`"hello"`).
- Compile-time constants (`final String x = "a" + "b"`).

**What it does NOT**:
- `new String("hello")` — always creates a **new** heap object outside the pool.
- Runtime concatenation with a variable — another fresh object.

**`.intern()`** — explicitly pools a string or returns the existing pooled reference:
```java
String s = new String("hello").intern();  // from the pool
```

> [!gotcha]
> `"hello" == "hello"` → **true** (same pooled object).
> `new String("hello") == "hello"` → **false** (different objects).
> `"Ja" + "va" == "Java"` → **true** (compile-time constant folding).
> `var x = "Ja"; x + "va" == "Java"` → **false** (runtime concatenation).

## Compact Strings (JEP 254, Java 9)

Pre-Java-9: internal storage was `char[]` (2 bytes per char, always).

Java 9+: `byte[]` + a **coder flag**:
- **LATIN1 (0)** — one byte per char, for ASCII and Latin-1 strings.
- **UTF16 (1)** — two bytes per char, for strings containing non-Latin-1 characters.

For most apps (ASCII-dominant) — **memory is halved**. The API is unchanged, the optimisation is transparent.

## Concatenation under the hood

**Pre-Java-9**: `javac` compiled `"a" + b + "c"` into a `new StringBuilder().append().append().toString()` chain. Rigid and slow for complex cases.

**Java 9+ (JEP 280)**: `+` compiles to a **single `invokedynamic`** → `StringConcatFactory.makeConcatWithConstants`. At runtime the JVM picks the optimal strategy (one byte[] allocation of the exact size).

> [!tip]
> In a single expression — use `+`, the JVM optimises better than manual StringBuilder. **In a loop** — use `StringBuilder`, because the compiler can't optimise across iterations.

## Core methods

- `length()`, `charAt(i)`, `isEmpty()`, `isBlank()` (Java 11+).
- Comparison: `equals`, `equalsIgnoreCase`, `compareTo`, `compareToIgnoreCase`.
- Search: `indexOf`, `lastIndexOf`, `contains`, `startsWith`, `endsWith`.
- Extract: `substring`, `split`, `chars()` (IntStream).
- Transform: `toUpperCase`, `toLowerCase`, `trim`, `strip` (Java 11+, Unicode-aware), `replace`, `replaceAll` (regex), `replaceFirst`.
- Format: `format`, `formatted` (Java 15+), `String.join`.
- Java 15+: **text blocks** `""" ... """` for multi-line literals.

## Risks of excessive `intern()`

> [!production]
> The pool is a native hashtable inside the JVM (`-XX:StringTableSize`, default 65536 buckets in modern JVMs).
>
> Risks:
> - **Hash collisions** degrade lookup to O(n) per bucket if the table is too small.
> - **Strong references** — pooled strings live until JVM exit (GC roots). Interning millions of unique strings → memory leak.
> - **Lock contention** — under heavy concurrent interning the pool can become a bottleneck.
>
> Apply `intern()` only to a **bounded, frequently repeated** set of values (enum-like, parsed identifiers).

## `switch` on Strings (Java 7+)

Works via `hashCode() + equals()` internally. The JVM compiles it into a `tableswitch` on hashes with an equals fallback for collision cases.

## String vs StringBuilder vs StringBuffer

| | Mutable | Thread-safe | When |
|--|--------|-------------|------|
| `String` | No | Yes (immutable) | Default |
| `StringBuilder` | Yes | No | Single-threaded building |
| `StringBuffer` | Yes | Yes (synchronized) | Legacy; rarely needed |

## Code
```java
public class StringPoolDemo {
    public static void main(String[] args) {
        // Literal goes to the String Pool
        String a = "Java";
        String b = "Java";
        System.out.println(a == b);        // true (same pool reference)

        // 'new' always creates a separate heap object
        String c = new String("Java");
        System.out.println(a == c);        // false (different objects)
        System.out.println(a.equals(c));   // true (same content)

        // intern() returns the pool reference
        String d = c.intern();
        System.out.println(a == d);        // true

        // Concatenation of literals is resolved at compile time
        String e = "Ja" + "va";
        System.out.println(a == e);        // true (compiler folds constants)

        // Concatenation with a variable happens at runtime -> new object
        String prefix = "Ja";
        String f = prefix + "va";
        System.out.println(a == f);        // false

        // Compact Strings: ASCII uses 1 byte per char internally
        String ascii = "hello";             // Latin-1 encoded (5 bytes)
        String unicode = "\u4F60\u597D";  // UTF-16 encoded (4 bytes)

        // Immutability: every "modification" returns a new String
        String original = "immutable";
        String upper = original.toUpperCase();
        System.out.println(original);       // "immutable" (unchanged)
        System.out.println(upper);          // "IMMUTABLE" (new object)
    }
}
```

## Tip
Never use `==` for String comparison unless explicitly testing reference identity (e.g. checking whether a value is interned). In interviews, immediately correcting `==` to `.equals()` signals awareness of a bug that plagues production code.

### explanation
Spring resolves `@Value("${property}")` placeholders into `String`s. Thanks to immutability, injected configuration values are automatically thread-safe.

`Environment` and the `PropertySource` chain always return Strings. Internally, Spring uses `String.intern()` for bean names and scope identifiers — a bounded set of known names with frequent identity comparisons, the ideal interning use case.

Keys in `ApplicationContext.getBean(name)`, name lookup in the registry — immutability + pooling pay off everywhere.

## Interview
### [4-1-q0 | junior] Why are Strings immutable in Java?
Three key benefits:

1. **Thread safety** — Strings can be shared across threads without synchronisation.
2. **Security** — Strings used in class loading, URLs, file paths cannot be tampered with after a check.
3. **Performance** — `hashCode()` is cached (computed once, stored in a field), and the **String Pool** can safely reuse instances because no one can modify a pooled String.

The immutable design also enables aggressive JVM optimisations — `substring()` in Java 6 shared the backing array (changed since to prevent leaks).

### [4-1-q1 | junior] What's the difference between string literals, `new String(...)`, and `.intern()`?
- **Literal** (`"hello"`) — goes into the **String Pool** automatically on class load. `"hello" == "hello"` → true (same pooled reference).
- **`new String("hello")`** — always creates a **new** heap object **outside** the pool, even if an identical literal already exists. `new String("hello") == "hello"` → false.
- **`.intern()`** — explicitly pools the string or returns the existing pooled reference. `new String("hello").intern() == "hello"` → true.

Compile-time constants (`"a" + "b"`) also enter the pool, but runtime concatenation with a variable (`prefix + "b"`) doesn't.

**Rule**: `.equals()` for content comparison, `==` only if you're explicitly checking identity.

### [4-1-q2 | mid] How does String concatenation work internally since Java 9, and when should you still use StringBuilder?
Since Java 9 (JEP 280) the `+` operator compiles to a **single `invokedynamic`** → `StringConcatFactory.makeConcatWithConstants`. At runtime the bootstrap method generates an optimal strategy (usually a single byte[] allocation of the exact size).

**For a single expression** — `+` is more efficient than manual `StringBuilder`: the JVM can pre-compute the buffer size and do one allocation.

**When `StringBuilder` is still needed**:
- **In loops** — each iteration is a separate `invokedynamic` call that creates a new String. `StringBuilder` accumulates into a single buffer:
```java
StringBuilder sb = new StringBuilder();
for (String s : list) sb.append(s).append(",");
String result = sb.toString();
```
- **Building across method calls or conditional branches** — the compiler can't optimise one expression.
- **Dynamic size** — when you don't know the number of pieces up front.

`StringBuffer` is the legacy synchronised version of `StringBuilder`. Rarely needed; `StringBuilder` is almost always the right tool.

### [4-1-q3 | senior] Explain Compact Strings (JEP 254), the String Pool's internal data structure, and the risks of excessive interning.
**Compact Strings (Java 9, JEP 254)** — the internal `char[]` was replaced by a `byte[]` + `coder` flag:
- **LATIN1 (0)** — 1 byte/char (ASCII and Latin-1).
- **UTF16 (1)** — 2 bytes/char (strings with non-Latin-1).

For a typical ASCII-dominant app — **memory is halved** with no API change.

**The String Pool** is a native hashtable inside the JVM (not a `HashMap`!), sized by `-XX:StringTableSize` (default 65536 buckets). Each bucket is a linked list.

**Risks of `intern()`**:

1. **Collision degradation** — with a too-small table, lookup degrades to O(n) per bucket.
2. **Strong references** — pooled strings are GC roots, **not collected until JVM exit**. Interning millions of unique strings → memory leak.
3. **Lock contention** — under heavy concurrent interning the pool can become a bottleneck.

**Rule**: use `intern()` only for a **bounded, frequently repeated** set of values (enum-like, parsed identifiers from a fixed domain).

Monitoring: `jcmd <pid> VM.stringtable` shows size and collisions.
