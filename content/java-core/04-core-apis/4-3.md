## Title
equals(), hashCode(), toString()

## Summary
`Object` is the base class of every Java object. `equals`, `hashCode`, `toString` handle identity, hashing, and debugging. **The contract**: equals is reflexive, symmetric, transitive, consistent, never equal to null; **equal objects must return the same hashCode**. Breaking it destroys `HashMap`, `HashSet`, and every hash-based collection.

## Deep Dive
## Methods every Java object inherits

| Method | Purpose | Default |
|--------|---------|---------|
| `equals(Object)` | Value comparison | `==` (identity) |
| `hashCode()` | Hash for HashMap/Set | native (identity hash) |
| `toString()` | Readable representation | `Class@hexHash` |
| `getClass()` | Runtime class | final, can't override |
| `clone()` | Shallow copy | protected, requires `Cloneable` |
| `wait` / `notify` / `notifyAll` | Sync primitives | final |
| `finalize()` | Pre-GC cleanup | **deprecated since Java 9** |

## The `equals()` contract

From the JLS / `Object` javadoc:

1. **Reflexive**: `x.equals(x) == true`.
2. **Symmetric**: `x.equals(y) == y.equals(x)`.
3. **Transitive**: `x.equals(y) && y.equals(z)` Ôćĺ `x.equals(z)`.
4. **Consistent**: same result absent modifications.
5. **Null-safety**: `x.equals(null) == false`.

**Standard recipe**:

```java
@Override public boolean equals(Object obj) {
    if (this == obj) return true;              // identity ÔÇö fast path
    if (!(obj instanceof Money other)) return false;  // type check + pattern (Java 16+)
    return amount == other.amount
        && Objects.equals(currency, other.currency);
}
```

- `this == obj` ÔÇö **free fast path**, interviewers expect this line.
- `instanceof` with pattern binding (Java 16+) replaces the two-step old style (`instanceof` + cast).
- For reference fields ÔÇö `Objects.equals(...)` (null-safe).
- For `double`/`float` ÔÇö `Double.compare` / `Float.compare`, correctly handling NaN.

## The `hashCode()` contract

1. **Stable**: within one JVM run and unchanged state ÔÇö always the same value.
2. **Consistent with equals**: if `x.equals(y)`, then `x.hashCode() == y.hashCode()`. **Mandatory**.
3. **Reverse is not required**: unequal objects may share a hash (collisions allowed).

> [!gotcha]
> **Overrode `equals` but forgot `hashCode`** Ôćĺ contract broken. `HashMap`/`HashSet` put the object in one bucket; at lookup time with an equal object, they look in a different bucket (different hashes) ÔÇö object **not found**. This is the most common production bug for Java developers.

**Standard implementations**:

- **Manual (Effective Java)**: `int result = 17; result = 31 * result + field.hashCode();` ÔÇö prime 31 distributes well, and the JIT optimises `31 * i` into `(i << 5) - i`.
- **`Objects.hash(field1, field2, ...)`** (Java 7+) ÔÇö one-liner, but creates a varargs array (minor GC overhead in hot paths).
- **IDE / Lombok `@EqualsAndHashCode`** ÔÇö for production code.
- **Records** (Java 16+) ÔÇö auto-generate equals + hashCode over all components.

## Mutable keys ÔÇö the classic leak

> [!production]
> Put an object in a `HashSet` Ôćĺ change a field used in `hashCode()` Ôćĺ the object "disappears": it's still in the set, but `contains`/`remove` can't find it. Only iteration works. This is a memory leak that's **impossible to debug** from the symptom.
> 
> **Rule**: `HashMap` keys / `HashSet` elements must be **immutable** (or at minimum, their hash-relevant fields must not change while in the collection). Typical safe keys: `String`, `Integer`, `UUID`, `enum`, records.

## toString() ÔÇö for debugging and logs

The default `Class@hexHash` is useless. Override with the class name and significant fields:

```java
@Override public String toString() {
    return "Money[amount=" + amount + ", currency=" + currency + "]";
}
```

> [!gotcha]
> **Never** put sensitive data in `toString()`: passwords, tokens, secrets. They'll instantly end up in logs, crash reports, stack traces. Lombok `@ToString(exclude = "password")` / `@ToString.Exclude` is must-have in production.

## Inheritance and the symmetry problem

The classic trap:

```java
class Point { int x, y; ... }
class ColorPoint extends Point { Color color; ... }
```

If `Point.equals` uses `instanceof Point` and compares x, y ÔÇö and `ColorPoint.equals` also compares color, then:
- `point.equals(colorPoint)` Ôćĺ **true** (Point doesn't know about color).
- `colorPoint.equals(point)` Ôćĺ **false** (fails instanceof ColorPoint or missing color).

**Symmetry is broken**.

**Solutions**:
1. **`getClass()` instead of `instanceof`** ÔÇö but breaks Liskov Substitution and doesn't work with proxies/subclasses.
2. **Composition over inheritance** ÔÇö `ColorPoint` holds a `Point` field. Effective Java recommendation for value types.
3. **`canEqual` pattern** (Scala-style) ÔÇö an extra method checking mutual comparability. Lombok implements it.
4. **Records (Java 16+)** ÔÇö implicitly `final`, the problem disappears.

## Records ÔÇö the proper path for value types

```java
public record Money(int amount, String currency) {}
```

Automatically:
- `equals` ÔÇö component-wise comparison.
- `hashCode` ÔÇö across all components.
- `toString` ÔÇö `Money[amount=100, currency=USD]`.
- Final class ÔÇö no inheritance problem.
- Shallowly immutable ÔÇö components are `final`.

For value types in modern Java ÔÇö **records by default**.

## Code
```java
import java.util.Objects;
import java.util.HashSet;

public class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        this.amount = amount;
        this.currency = Objects.requireNonNull(currency);
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;                     // identity check
        if (!(obj instanceof Money other)) return false;  // type check (Java 16 pattern)
        return amount == other.amount
            && currency.equals(other.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
        // equivalent to: 31 * Integer.hashCode(amount) + currency.hashCode()
    }

    @Override
    public String toString() {
        return "Money{amount=" + amount + ", currency='" + currency + "'}";
    }

    public static void main(String[] args) {
        Money a = new Money(100, "USD");
        Money b = new Money(100, "USD");
        Money c = new Money(200, "EUR");

        // equals works by value
        System.out.println(a.equals(b)); // true
        System.out.println(a.equals(c)); // false
        System.out.println(a == b);      // false (different heap objects)

        // hashCode contract: equal objects -> same hash
        System.out.println(a.hashCode() == b.hashCode()); // true

        // HashSet relies on equals + hashCode
        HashSet<Money> set = new HashSet<>();
        set.add(a);
        System.out.println(set.contains(b)); // true (works because equals+hashCode)

        // toString for debugging
        System.out.println(a); // Money{amount=100, currency='USD'}

        // --- Mutable key pitfall ---
        // (Don't do this: if key fields change, the object is lost in the set)
    }
}

// --- Modern Java: record does all this automatically ---
record MoneyRecord(int amount, String currency) {}

```

## Tip
In any `equals()` implementation, always start with `if (this == obj) return true`. It's a free fast-path optimisation and every interviewer expects to see it.

## Spring
### concept
equals() & hashCode()

### springFeature
Spring Data / JPA Entity Identity

### explanation
JPA entities have a **tricky** relationship with `equals`/`hashCode`.

- The **primary key** (ID) may be `null` before persist (auto-generated). Using the ID in equals/hashCode breaks the contract when an entity transitions from transient to managed (ID flips from `null` to a value Ôćĺ `hashCode` changes Ôćĺ the object "vanishes" from Sets).
- **Hibernate lazy proxies** aren't direct instances of the target class ÔÇö `instanceof Entity` returns true, but `getClass() == Entity.class` is false.

**Recommended pattern** (Vlad Mihalcea):
- A natural business key if one exists (email, SKU).
- Or a **UUID** assigned in the constructor (before persist).
- Or primary-key-only but null-safe: if both IDs are null ÔÇö identity, else compare IDs.

Getting this wrong Ôćĺ duplicates in `Set<Entity>`, broken lookups in Hibernate's persistence context, weird cascade bugs.

## Interview
### [4-3-q0 | junior] Why must you override `hashCode()` when you override `equals()`?
The contract: **if two objects are equal via `equals()`, they must return the same `hashCode()`**.

`HashMap`/`HashSet` work in two phases:
1. `hashCode()` determines the **bucket** (slot in the array).
2. Inside the bucket, `equals()` finds the exact match.

If `equals` says "objects are equal" but `hashCode` returns different values, the lookup searches the **wrong bucket** and can't find the object. The entire hash-based collection mechanism breaks.

Example bug:
```java
class Bad { int x; public boolean equals(Object o) { return ((Bad)o).x == x; } /* hashCode not overridden */ }
Set<Bad> set = new HashSet<>();
set.add(new Bad(42));
set.contains(new Bad(42));  // false! (different identity hashes)
```

### [4-3-q1 | mid] What happens if you use a mutable object as a HashMap key and then modify fields used in `hashCode()`?
The object was placed in a bucket based on its **original** `hashCode`. After mutation, `hashCode()` returns a different value, so `get()` looks in a **different** bucket and can't find the entry.

The object becomes a **ghost entry**: physically in the map but:
- `get(key)` returns `null`.
- `containsKey(key)` Ôćĺ `false`.
- `remove(key)` does nothing.
- Only reachable by iterating `entrySet()`.

It's a **memory leak impossible to debug from symptoms** ÔÇö the object is invisible via the API but still occupies memory and can pin a large object graph.

**Fixes**:
- Use **immutable** keys: `String`, `Integer`, `UUID`, `enum`, records.
- If a mutable key is unavoidable ÔÇö don't modify hash-relevant fields while the object is in a collection.
- For entities with auto-generated IDs (JPA): use a natural business key, a UUID, or implement `equals`/`hashCode` consistently across transient and managed states.

### [4-3-q2 | senior] Explain the symmetry problem with `equals()` in inheritance hierarchies and how to solve it.
If `Point.equals()` uses `instanceof Point` and compares `x, y`, while `ColorPoint.equals()` also checks `instanceof ColorPoint` and compares `color`:

```java
Point p = new Point(1, 2);
ColorPoint cp = new ColorPoint(1, 2, RED);
p.equals(cp);   // true  (Point.equals ignores color)
cp.equals(p);   // false (ColorPoint fails the instanceof ColorPoint check)
```

**Symmetry is broken** ÔÇö the `equals` contract is violated.

**Solutions**:

1. **`getClass()` instead of `instanceof`**:
```java
if (getClass() != obj.getClass()) return false;
```
Works, but **breaks Liskov Substitution** (Point and ColorPoint are no longer interchangeable) and doesn't work with proxies/subclasses.

2. **Composition over inheritance** (Effective Java recommendation for value types):
```java
class ColorPoint { Point point; Color color; }  // has-a, not is-a
```

3. **`canEqual` pattern** (Scala-style): an extra method checking mutual comparability. Lombok `@EqualsAndHashCode(callSuper = true)` implements it.

4. **Records (Java 16+)** ÔÇö implicitly `final`, the problem disappears. The **recommended modern-Java approach** for value types.
