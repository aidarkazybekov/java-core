## Title
equals(), hashCode(), toString()

## Summary
`Object` — базовый класс всех объектов Java. Методы `equals`, `hashCode`, `toString` — для идентичности, хэширования, отладки. **Контракт**: equals рефлексивно, симметрично, транзитивно, консистентно, никогда не равен null; **равные объекты обязаны иметь одинаковый hashCode**. Нарушение ломает `HashMap`, `HashSet` и любые hash-based коллекции.

## Deep Dive
## Методы, которые наследует каждый Java-объект

| Метод | Назначение | Дефолт |
|-------|------------|--------|
| `equals(Object)` | Сравнение по значению | `==` (identity) |
| `hashCode()` | Хэш для HashMap/Set | native (identity hash) |
| `toString()` | Читаемое представление | `Class@hexHash` |
| `getClass()` | Runtime-класс | final, нельзя override |
| `clone()` | Поверхностное копирование | protected, требует `Cloneable` |
| `wait` / `notify` / `notifyAll` | Примитивы синхронизации | final |
| `finalize()` | Cleanup перед GC | **deprecated с Java 9** |

## Контракт `equals()`

Из JLS / `Object` javadoc:

1. **Рефлексивность**: `x.equals(x) == true`.
2. **Симметричность**: `x.equals(y) == y.equals(x)`.
3. **Транзитивность**: `x.equals(y) && y.equals(z)` → `x.equals(z)`.
4. **Консистентность**: результат не должен меняться при отсутствии модификаций.
5. **Null-safety**: `x.equals(null) == false`.

**Стандартный рецепт**:

```java
@Override public boolean equals(Object obj) {
    if (this == obj) return true;              // identity — fast path
    if (!(obj instanceof Money other)) return false;  // type check + pattern (Java 16+)
    return amount == other.amount
        && Objects.equals(currency, other.currency);
}
```

- `this == obj` — **free fast path**, interviewers ожидают эту строку.
- `instanceof` с pattern binding (Java 16+) — заменяет два шага старого стиля (`instanceof` + cast).
- Для ссылочных полей — `Objects.equals(...)` (null-safe).
- Для `double`/`float` — `Double.compare` / `Float.compare`, корректно обрабатывают NaN.

## Контракт `hashCode()`

1. **Стабильность**: в рамках одного запуска JVM и неизменного состояния — всегда одно и то же значение.
2. **Консистентность с equals**: если `x.equals(y)`, то `x.hashCode() == y.hashCode()`. **Обязательное требование**.
3. **Обратное не требуется**: неравные объекты могут иметь одинаковый hash (коллизии допустимы).

> [!gotcha]
> **Переопределили `equals`, но забыли `hashCode`** → контракт нарушен. `HashMap`/`HashSet` положат объект в один bucket, а при lookup'е по равному объекту будут искать в другом (разные хэши) — объект **не найдётся**. Это самый частый production-баг Java-разработчиков.

**Стандартные реализации**:

- **Вручную (Effective Java)**: `int result = 17; result = 31 * result + field.hashCode();` — простое число 31 даёт хорошее распределение, JIT оптимизирует `31 * i` в `(i << 5) - i`.
- **`Objects.hash(field1, field2, ...)`** (Java 7+) — one-liner, но создаёт varargs-массив (minor GC overhead в hot paths).
- **IDE / Lombok `@EqualsAndHashCode`** — для production-кода.
- **Records** (Java 16+) — авто-генерируют equals + hashCode по всем компонентам.

## Mutable key — классическая утечка

> [!production]
> Положили объект в `HashSet` → изменили поле, используемое в `hashCode()` → объект «потерялся»: он есть в set'е, но `contains`/`remove` его не находят. Остаётся только итерацией. Это memory leak, который **невозможно отладить**.
> 
> **Правило**: ключи `HashMap`/элементы `HashSet` должны быть **immutable** (или, как минимум, не менять hash-relevant поля, пока они в коллекции). Типовые безопасные ключи: `String`, `Integer`, `UUID`, `enum`, record.

## toString() — для отладки и логов

Дефолт `Class@hexHash` бесполезен. Переопределяйте с именем класса и значимыми полями:

```java
@Override public String toString() {
    return "Money[amount=" + amount + ", currency=" + currency + "]";
}
```

> [!gotcha]
> **Никогда** не включайте в `toString()` чувствительные данные: пароли, токены, секреты. Они мгновенно окажутся в логах, crash-reports, stack trace'ах. Lombok `@ToString(exclude = "password")` / `@ToString.Exclude` — must-have в production.

## Наследование и проблема симметрии

Классическая ловушка:

```java
class Point { int x, y; ... }
class ColorPoint extends Point { Color color; ... }
```

Если `Point.equals` использует `instanceof Point` и сравнивает x, y — а `ColorPoint.equals` ещё и color, то:
- `point.equals(colorPoint)` → **true** (Point не знает про color).
- `colorPoint.equals(point)` → **false** (fail instanceof ColorPoint или отсутствие color).

**Симметрия нарушена**.

**Решения**:
1. **`getClass()` вместо `instanceof`** — но ломает Liskov Substitution и не работает с прокси/наследниками.
2. **Композиция вместо наследования** — `ColorPoint` содержит `Point` как поле. Рекомендация Effective Java для value types.
3. **`canEqual` паттерн** (Scala-style) — дополнительный метод, проверяющий взаимную совместимость. Lombok его реализует.
4. **Records (Java 16+)** — неявно `final`, проблема отпадает.

## Records — правильный путь для value types

```java
public record Money(int amount, String currency) {}
```

Автоматически:
- `equals` — сравнение всех компонентов.
- `hashCode` — по всем компонентам.
- `toString` — `Money[amount=100, currency=USD]`.
- Финальный класс — проблема наследования отсутствует.
- Shallowly immutable — компоненты `final`.

Для value types в современной Java — **records по умолчанию**.

## Code
```java
import java.util.Objects;
import java.util.HashSet;

public class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        this.amount = amount;
        this.currency = Objects.requireNonNull(currency);
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;                     // identity check
        if (!(obj instanceof Money other)) return false;  // type check (Java 16 pattern)
        return amount == other.amount
            && currency.equals(other.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
        // equivalent to: 31 * Integer.hashCode(amount) + currency.hashCode()
    }

    @Override
    public String toString() {
        return "Money{amount=" + amount + ", currency='" + currency + "'}";
    }

    public static void main(String[] args) {
        Money a = new Money(100, "USD");
        Money b = new Money(100, "USD");
        Money c = new Money(200, "EUR");

        // equals works by value
        System.out.println(a.equals(b)); // true
        System.out.println(a.equals(c)); // false
        System.out.println(a == b);      // false (different heap objects)

        // hashCode contract: equal objects -> same hash
        System.out.println(a.hashCode() == b.hashCode()); // true

        // HashSet relies on equals + hashCode
        HashSet<Money> set = new HashSet<>();
        set.add(a);
        System.out.println(set.contains(b)); // true (works because equals+hashCode)

        // toString for debugging
        System.out.println(a); // Money{amount=100, currency='USD'}

        // --- Mutable key pitfall ---
        // (Don't do this: if key fields change, the object is lost in the set)
    }
}

// --- Modern Java: record does all this automatically ---
record MoneyRecord(int amount, String currency) {}

```

## Tip
В любой реализации `equals()` всегда начинайте с `if (this == obj) return true`. Это бесплатная fast-path оптимизация, и каждый интервьюер ожидает увидеть эту строку.

## Spring
### concept
equals() & hashCode()

### springFeature
Spring Data / JPA Entity Identity

### explanation
JPA-сущности имеют **сложные** отношения с `equals`/`hashCode`.

- **Primary key** (ID) может быть `null` до persist (auto-generated). Использование ID в equals/hashCode ломает контракт, когда entity переходит из transient в managed (ID меняется с `null` на значение → `hashCode` меняется → объект «теряется» в Set'ах).
- **Lazy proxies Hibernate** не являются instance целевого класса напрямую — `instanceof Entity` возвращает true, но `getClass() == Entity.class` — false.

**Рекомендованный паттерн** (Vlad Mihalcea):
- Natural business key, если есть (email, SKU).
- Или **UUID**, присвоенный в конструкторе (до persist).
- Или только по primary key, но с защитой от `null`: если оба ID null — identity, иначе по ID.

Ошибка здесь = дубликаты в `Set<Entity>`, сломанные lookup'ы в persistence context'е Hibernate, непонятные баги с cascade.

## Interview
### [4-3-q0 | junior] Почему при переопределении `equals()` обязательно переопределять `hashCode()`?
Контракт: **если два объекта равны по `equals()`, они обязаны возвращать одинаковый `hashCode()`**.

`HashMap`/`HashSet` работают в две фазы:
1. По `hashCode()` определяют **bucket** (слот в массиве).
2. Внутри bucket'а через `equals()` ищут точное совпадение.

Если `equals` говорит «объекты равны», но `hashCode` возвращает разные значения — lookup ищет в **неправильном bucket'е** и не находит объект. Ломается весь механизм hash-based коллекций.

Пример бага:
```java
class Bad { int x; public boolean equals(Object o) { return ((Bad)o).x == x; } /* hashCode не переопределён */ }
Set<Bad> set = new HashSet<>();
set.add(new Bad(42));
set.contains(new Bad(42));  // false! (разные identity-hash)
```

### [4-3-q1 | mid] Что произойдёт, если использовать мутабельный объект как ключ `HashMap` и изменить поля, участвующие в `hashCode()`?
Объект был помещён в bucket на основе **оригинального** `hashCode`. После мутации `hashCode()` возвращает другое значение, поэтому `get()` ищет в **другом** bucket'е и **не находит** запись.

Объект становится **ghost entry**: он физически в map, но:
- `get(key)` возвращает `null`.
- `containsKey(key)` → `false`.
- `remove(key)` ничего не удаляет.
- Найти можно только итерацией по `entrySet()`.

Это **memory leak, который невозможно отладить** — объект невидим через API, но занимает память и держит ссылку (возможно на большой object graph).

**Решения**:
- Используйте **immutable** объекты как ключи: `String`, `Integer`, `UUID`, `enum`, records.
- Если ключ с мутабельными полями неизбежен — не изменяйте hash-relevant поля, пока объект в коллекции.
- Для entity с auto-generated ID (JPA): используйте natural business key, UUID или реализуйте `equals`/`hashCode` consistently для transient и managed состояний.

### [4-3-q2 | senior] Объясните проблему симметрии `equals()` в иерархиях наследования и как её решать.
Если `Point.equals()` использует `instanceof Point` и сравнивает `x, y`, а `ColorPoint.equals()` дополнительно проверяет `instanceof ColorPoint` и сравнивает `color`:

```java
Point p = new Point(1, 2);
ColorPoint cp = new ColorPoint(1, 2, RED);
p.equals(cp);   // true  (Point.equals игнорирует color)
cp.equals(p);   // false (ColorPoint fail на instanceof ColorPoint)
```

**Симметрия нарушена** — контракт `equals` сломан.

**Решения**:

1. **`getClass()` вместо `instanceof`**:
```java
if (getClass() != obj.getClass()) return false;
```
Работает, но **ломает Liskov Substitution** (Point и ColorPoint больше не взаимозаменяемы) и не работает с прокси/subclass'ами.

2. **Композиция вместо наследования** (рекомендация Effective Java для value types):
```java
class ColorPoint { Point point; Color color; }  // has-a, не is-a
```

3. **`canEqual` паттерн** (Scala-style): дополнительный метод, проверяющий взаимную совместимость. Lombok `@EqualsAndHashCode(callSuper = true)` его реализует.

4. **Records (Java 16+)** — неявно `final`, проблема отсутствует. Для value types — **рекомендованный путь в современной Java**.
