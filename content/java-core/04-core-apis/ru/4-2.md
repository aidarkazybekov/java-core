## Title
StringBuilder vs StringBuffer

## Summary
**`StringBuilder`** (Java 5+) — мутабельный буфер для построения строк, **не синхронизирован**, быстрый. **`StringBuffer`** (Java 1.0) — то же самое, но все методы `synchronized`, thread-safe. Оба наследуют `AbstractStringBuilder` с `byte[]`-буфером (Java 9+) и стратегией удвоения емкости. `String` concat в цикле — O(n²), StringBuilder.append — амортизированно O(1).

## Deep Dive
## Три класса — три роли

| | Mutable | Thread-safe | Появился | Когда |
|--|--------|-------------|----------|-------|
| `String` | **No** | Yes (immutable) | 1.0 | По умолчанию для значений-строк |
| `StringBuffer` | Yes | Yes (`synchronized`) | 1.0 | Legacy; при многопоточном построении — и то редко |
| `StringBuilder` | Yes | **No** | 5.0 | Single-thread построение — почти всегда |

Оба `StringBuilder` и `StringBuffer` наследуют `AbstractStringBuilder`, который хранит мутабельный `byte[]`-буфер и count использованных символов. API идентичны.

## Почему `+` в цикле O(n²)

```java
String s = "";
for (int i = 0; i < n; i++) s += "x";  // O(n²)!
```

Каждый `+=` создаёт **новый `String`**, копирует все существующие символы в него и добавляет новый. На итерации k копируется k символов. Суммарно: `1 + 2 + ... + n = O(n²)`.

Для n = 100 000 — это 10 миллиардов операций копирования. Типичный микро-бенчмарк показывает **1000× разницу** с `StringBuilder`.

## Почему `StringBuilder.append` амортизированно O(1)

Буфер хранится в `byte[]` определённой capacity. При `append`:
1. Если места хватает — просто копируются байты. O(k) для k новых символов.
2. Если нет — буфер **удваивается** (`(oldCapacity * 2) + 2`), старое содержимое копируется в новый массив через `System.arraycopy`.

Амортизационно: общая работа на n appends = O(n), потому что resize случается O(log n) раз и суммарно копируется 2n символов. Отсюда O(1) на append **в среднем**.

## Pre-sizing — важная оптимизация

Default capacity — 16 символов. Если знаете приблизительный финальный размер, **передайте в конструктор**:

```java
// BAD — несколько resize-циклов
StringBuilder sb = new StringBuilder();
for (User u : users) sb.append(u.name()).append(",");  // n users → log n resizes

// GOOD — одна аллокация
StringBuilder sb = new StringBuilder(users.size() * 20);
```

Для очень больших строк (мегабайты) pre-sizing заметно снижает GC pressure.

## StringBuffer vs StringBuilder — синхронизация

Единственное отличие — каждый публичный метод `StringBuffer` объявлен `synchronized`. Для single-thread построения — ненужный overhead (захват монитора).

> [!production]
> **Практика**: `StringBuffer` почти никогда не нужен. Построение строки редко когда идёт параллельно из нескольких потоков; когда идёт — лучше внешняя синхронизация или thread-local буфер. `StringBuffer` — legacy с Java 1.0, `StringBuilder` — его правильная замена с Java 5.

## Ловушка `equals()`

> [!gotcha]
> `StringBuilder.equals()` **не переопределён** — используется `Object.equals()`, сравнение по ссылке. Два StringBuilder с одинаковым содержимым **не равны** через `.equals()`!
> 
> Правильно: `sb1.toString().equals(sb2.toString())` или, с Java 11+, `sb1.compareTo(sb2) == 0` (`StringBuilder` implements `Comparable`).

## Compact Strings и ловушка inflation

С Java 9 `StringBuilder` тоже использует Compact Strings — хранит Latin-1 контент в `byte[]` с одним байтом на символ.

> [!gotcha]
> Когда вы добавляете **не-Latin-1** символ (китайский, emoji) к большому Latin-1 буферу, StringBuilder вынужден **инфлейтировать весь буфер** в UTF-16: выделить новый `byte[]` двойного размера, расширить каждый существующий байт в два, и только потом добавить новый символ. Одноразовая O(n) стоимость, сюрприз в performance-чувствительном коде с mixed-encoding входом.
> 
> Workaround: если точно знаете, что будет не-Latin-1 контент, начните с `sb.append('\u00ff').deleteCharAt(0)` (force UTF-16 mode) — хак, но работает.

## Основные методы

- `append(...)` — добавить (overload для всех примитивов, `String`, `CharSequence`, `Object`).
- `insert(int offset, ...)` — вставить в позицию.
- `delete(int start, int end)`, `deleteCharAt(int i)`.
- `replace(int start, int end, String s)`.
- `reverse()` — in-place.
- `setLength(int n)` — обрезать или расширить (новые позиции `\u0000`).
- `charAt`, `indexOf`, `substring` (возвращает новый `String`).
- `length()` — число использованных символов.
- `capacity()` — размер буфера.
- `ensureCapacity(int n)` — гарантировать минимум.
- `trimToSize()` — сжать буфер до длины (редко нужно).

Все `append`/`insert`/`replace` возвращают `this` → **fluent API**.

## Когда использовать `+`, когда `StringBuilder`

- **Одиночное выражение** (`"x=" + x + ", y=" + y`) — `+`. С Java 9+ JVM оптимизирует через `StringConcatFactory` лучше ручного StringBuilder.
- **Цикл** — `StringBuilder`. Компилятор не может оптимизировать across iterations.
- **Построение через условия/методы** — `StringBuilder`, особенно если порядок и состав частей зависит от логики.
- **Streams**: `stream.collect(Collectors.joining(","))` использует StringBuilder внутри — часто удобнее и читаемее.

## Code
```java
public class StringBuilderDemo {
    public static void main(String[] args) {
        // Pre-size when you know the approximate length
        StringBuilder sb = new StringBuilder(256);

        // Efficient append chain (fluent API, returns 'this')
        sb.append("SELECT * FROM users")
          .append(" WHERE active = true")
          .append(" AND created > '2024-01-01'")
          .append(" ORDER BY name")
          .append(" LIMIT 100");

        String sql = sb.toString();
        System.out.println(sql);

        // Common operations
        sb.insert(0, "-- Query\n");    // insert at position
        sb.delete(0, 11);              // delete range
        sb.replace(0, 6, "SELECT");    // replace range
        sb.reverse();                   // reverse in-place
        System.out.println(sb.length());       // chars used
        System.out.println(sb.capacity());     // buffer size

        // --- Performance comparison ---
        int iterations = 100_000;

        // BAD: String concatenation in loop -- O(n^2)
        long start = System.nanoTime();
        String result = "";
        for (int i = 0; i < iterations; i++) {
            result += "a";  // new String + copy each iteration
        }
        long stringTime = System.nanoTime() - start;

        // GOOD: StringBuilder -- O(n) amortized
        start = System.nanoTime();
        StringBuilder sb2 = new StringBuilder(iterations);
        for (int i = 0; i < iterations; i++) {
            sb2.append("a");
        }
        String result2 = sb2.toString();
        long builderTime = System.nanoTime() - start;

        System.out.printf("String concat: %d ms%n", stringTime / 1_000_000);
        System.out.printf("StringBuilder: %d ms%n", builderTime / 1_000_000);
        // Typical ratio: 1000x+ faster with StringBuilder

        // equals() pitfall
        StringBuilder x = new StringBuilder("hello");
        StringBuilder y = new StringBuilder("hello");
        System.out.println(x.equals(y));              // false!
        System.out.println(x.toString().equals(y.toString())); // true
    }
}
```

## Tip
На интервью всегда упомяните pre-sizing `StringBuilder`, когда знаете ожидаемую длину. `new StringBuilder(expectedSize)` показывает понимание buffer management и GC pressure — отличает от junior-кандидата.

## Interview
### [4-2-q0 | junior] В чём разница между StringBuilder и StringBuffer?
- **`StringBuilder`** — **не синхронизирован**. Быстрый, но не thread-safe.
- **`StringBuffer`** — все публичные методы `synchronized`. Thread-safe, но overhead на захвате монитора.

Оба наследуют `AbstractStringBuilder` и имеют идентичный API. `StringBuilder` добавлен в Java 5 как замена `StringBuffer` для single-thread-кейсов (99%+).

Используйте **`StringBuilder`** почти всегда. `StringBuffer` — legacy с Java 1.0; ситуации реального многопоточного построения встречаются крайне редко, и там обычно лучше внешняя синхронизация или thread-local buffer.

### [4-2-q1 | mid] Почему `String` конкатенация в цикле O(n²), а `StringBuilder.append()` — O(n)?
**`String +=`**: каждая итерация создаёт **новый `String`**, копирует все существующие символы и добавляет новый. На итерации k копируется k символов. Суммарно: `1 + 2 + ... + n = O(n²)`.

**`StringBuilder.append`**: буфер `byte[]` фиксированной capacity. При переполнении capacity **удваивается**. Амортизационно resize случается O(log n) раз и копирует суммарно 2n символов. Итого O(n) для n appends — O(1) на append в среднем.

Для n = 100 000 разница — ~1000× в реальных бенчмарках.

Нюанс Java 9+: **одиночное выражение** `a + b + c` компилируется в один `invokedynamic` → эффективно, как StringBuilder. **В цикле** каждая итерация — отдельный `invokedynamic`, и оптимизация across iterations невозможна — вот почему вручную StringBuilder нужен именно в циклах.

### [4-2-q2 | senior] Объясните, как Compact Strings влияют на StringBuilder и опишите сценарий, в котором append одного символа вызывает полную реаллокацию.
С Java 9 `StringBuilder` использует Compact Strings: Latin-1 контент хранится в `byte[]` с **одним** байтом на символ (coder=LATIN1). Пока весь контент Latin-1, буфер остаётся компактным.

Когда добавляется **не-Latin-1 символ** (китайский, emoji, кириллица за пределами Latin-1), StringBuilder вынужден **инфлейтировать весь буфер** из Latin-1 в UTF-16:
1. Аллоцировать новый `byte[]` **двойного размера** (2 байта на символ).
2. **Расширить каждый существующий байт** до двух байт.
3. Скопировать в новый массив.
4. Только потом добавить новый символ.

Это **одноразовая O(n) стоимость**, сюрприз в performance-чувствительном коде, обрабатывающем mixed-encoding входные данные (пользовательские имена, логи с эмодзи).

**Workaround**: если знаете, что будет не-Latin-1 контент, принудительно включите UTF-16 mode с самого начала:
```java
StringBuilder sb = new StringBuilder(expectedSize);
sb.append('\u00ff');  // выше Latin-1
sb.deleteCharAt(0);    // удалить, но coder остаётся UTF16
```

### [4-2-q3 | mid] Какая ловушка с `.equals()` у StringBuilder?
`StringBuilder.equals()` **не переопределён** — наследуется от `Object`, сравнение по identity (ссылке).

```java
StringBuilder x = new StringBuilder("hello");
StringBuilder y = new StringBuilder("hello");
x.equals(y);                  // false!
x.toString().equals(y.toString());  // true
```

Для сравнения содержимого:
- `sb1.toString().equals(sb2.toString())`.
- С Java 11+: `sb1.compareTo(sb2) == 0` (StringBuilder implements Comparable).

Причина: StringBuilder — mutable. Равные «сейчас» инстансы могут стать неравными при следующем append. Такой объект — плохой кандидат для HashMap ключа, поэтому `equals()` и `hashCode()` оставили как identity-сравнение, чтобы не создавать иллюзию корректности.
