## Title
String & String Pool

## Summary
`String` — **immutable** и **final** класс, внутри `byte[]` + coder flag (Java 9+, Compact Strings). **String Pool** — специальная область в Heap, хранящая уникальные строковые литералы (Flyweight). Литералы попадают в пул автоматически; `new String(...)` — **вне** пула; `.intern()` — явно помещает в пул. `==` сравнивает ссылки, `.equals()` — содержимое.

## Deep Dive
## Immutability — зачем и что даёт

Однажды созданная строка **никогда не меняется**. Любой метод, кажущийся модифицирующим (`concat`, `replace`, `substring`, `toUpperCase`), возвращает **новый** объект. Что это даёт:

- **Thread safety** — можно шарить между потоками без синхронизации.
- **Безопасность** — строка, использованная для имени класса, URL, file path, не может быть изменена после проверки.
- **Кэш hashCode** — `hashCode()` вычисляется один раз и хранится в поле. Поэтому `String` идеальный ключ для `HashMap`.
- **String Pool** — JVM может безопасно переиспользовать инстансы.
- **Корректная загрузка классов** — имя класса не изменится между проверкой и использованием.

## String Pool (Flyweight)

Специальная область в Heap (до Java 7 была в **PermGen**, с Java 7 — в main heap). Хранит уникальные строки.

**Что попадает в пул автоматически**:
- Строковые литералы (`"hello"`).
- Компиляционные константы (`final String x = "a" + "b"`).

**Что НЕ попадает**:
- `new String("hello")` — всегда создаёт **новый** объект в heap вне пула.
- Рантайм-конкатенация с переменной — тоже новый объект.

**`.intern()`** — явно кладёт строку в пул или возвращает существующую pooled-ссылку:
```java
String s = new String("hello").intern();  // из пула
```

> [!gotcha]
> `"hello" == "hello"` → **true** (один pooled-объект).
> `new String("hello") == "hello"` → **false** (разные объекты).
> `"Ja" + "va" == "Java"` → **true** (compile-time constant folding).
> `var x = "Ja"; x + "va" == "Java"` → **false** (runtime concatenation).

## Compact Strings (JEP 254, Java 9)

До Java 9: внутреннее хранилище — `char[]` (2 байта на символ всегда).

Java 9+: `byte[]` + **coder flag**:
- **LATIN1 (0)** — один байт на символ, для ASCII и Latin-1 строк.
- **UTF16 (1)** — два байта на символ, для строк с не-Latin-1 символами.

Для большинства приложений (ASCII-доминантных) — **экономия памяти в 2×**. API не изменился, всё прозрачно.

## Конкатенация — что под капотом

**До Java 9**: `javac` компилировал `"a" + b + "c"` в цепочку `new StringBuilder().append().append().toString()`. Жёстко и медленно для сложных случаев.

**Java 9+ (JEP 280)**: `+` компилируется в **одиночный `invokedynamic`** к `StringConcatFactory.makeConcatWithConstants`. JVM в рантайме генерирует оптимальную стратегию (одна аллокация byte[] с точным размером).

> [!tip]
> В одном выражении — пользуйтесь `+`, JVM оптимизирует лучше ручного StringBuilder. **В цикле** — `StringBuilder`, потому что компилятор не может оптимизировать across iterations.

## Основные методы

- `length()`, `charAt(i)`, `isEmpty()`, `isBlank()` (Java 11+).
- Сравнение: `equals`, `equalsIgnoreCase`, `compareTo`, `compareToIgnoreCase`.
- Поиск: `indexOf`, `lastIndexOf`, `contains`, `startsWith`, `endsWith`.
- Извлечение: `substring`, `split`, `chars()` (IntStream).
- Преобразование: `toUpperCase`, `toLowerCase`, `trim`, `strip` (Java 11+, Unicode-aware), `replace`, `replaceAll` (regex), `replaceFirst`.
- Форматирование: `format`, `formatted` (Java 15+), `String.join`.
- Java 15+: **text blocks** `""" ... """` для многострочных литералов.

## Риски чрезмерного `intern()`

> [!production]
> Пул внутри — native hashtable (`-XX:StringTableSize`, по умолчанию 65536 bucket'ов в современных JVM).
>
> Риски:
> - **Hash-коллизии** деградируют lookup до O(n) на bucket, если таблица мала.
> - **Сильные ссылки** — pooled-строки живут до выхода JVM (GC roots). Интернирование миллионов уникальных строк → утечка памяти.
> - **Lock contention** — под высокой конкурентной нагрузкой пул может стать узким местом.
>
> Применяйте `intern()` только к **ограниченному, часто повторяющемуся** множеству значений (enum-like, parsed identifiers).

## `switch` на строках (Java 7+)

Работает через `hashCode() + equals()` внутри. JVM компилирует в tableswitch по хэшам с fallback проверкой equals для collision-случаев.

## String vs StringBuilder vs StringBuffer

| | Mutability | Thread-safe | Когда |
|--|-----------|-------------|-------|
| `String` | No | Yes (immutable) | По умолчанию |
| `StringBuilder` | Yes | No | Single-thread построение |
| `StringBuffer` | Yes | Yes (synchronized) | Legacy; редко нужен |

## Code
```java
public class StringPoolDemo {
    public static void main(String[] args) {
        // Literal goes to the String Pool
        String a = "Java";
        String b = "Java";
        System.out.println(a == b);        // true (same pool reference)

        // 'new' always creates a separate heap object
        String c = new String("Java");
        System.out.println(a == c);        // false (different objects)
        System.out.println(a.equals(c));   // true (same content)

        // intern() returns the pool reference
        String d = c.intern();
        System.out.println(a == d);        // true

        // Concatenation of literals is resolved at compile time
        String e = "Ja" + "va";
        System.out.println(a == e);        // true (compiler folds constants)

        // Concatenation with a variable happens at runtime -> new object
        String prefix = "Ja";
        String f = prefix + "va";
        System.out.println(a == f);        // false

        // Compact Strings: ASCII uses 1 byte per char internally
        String ascii = "hello";             // Latin-1 encoded (5 bytes)
        String unicode = "\u4F60\u597D";  // UTF-16 encoded (4 bytes)

        // Immutability: every "modification" returns a new String
        String original = "immutable";
        String upper = original.toUpperCase();
        System.out.println(original);       // "immutable" (unchanged)
        System.out.println(upper);          // "IMMUTABLE" (new object)
    }
}
```

## Tip
Никогда не используйте `==` для сравнения String, если не тестируете явно identity (например, проверяете, что значение интерновано). На интервью немедленное исправление `==` на `.equals()` показывает внимание к частому production-багу.

### explanation
Spring резолвит `@Value("${property}")`-плейсхолдеры в `String`. Благодаря immutability инжектированные конфигурационные значения автоматически thread-safe.

`Environment` и цепочка `PropertySource` всегда возвращают строки. Spring внутри использует `String.intern()` для имён бинов и scope-идентификаторов — ограниченный набор известных имён, частое сравнение по identity, идеальный кейс для interning.

Ключи `ApplicationContext.getBean(name)`, lookup по имени в registry — везде выгоду даёт immutability + pooling.

## Interview
### [4-1-q0 | junior] Почему строки в Java immutable?
Три ключевых преимущества:

1. **Thread safety** — строки можно безопасно шарить между потоками без синхронизации.
2. **Безопасность** — строки, используемые для загрузки классов, URL, file path, не могут быть изменены после проверки.
3. **Производительность** — `hashCode()` кэшируется (вычисляется один раз, хранится в поле), **String Pool** может безопасно переиспользовать инстансы, потому что никто не может модифицировать pooled-строку.

Immutable-дизайн также позволяет JVM применять aggressive оптимизации — `substring()` в Java 6 шарил backing-массив (с тех пор изменено для предотвращения утечек).

### [4-1-q1 | junior] В чём разница между строковыми литералами, `new String(...)` и `.intern()`?
- **Литерал** (`"hello"`) — попадает в **String Pool** автоматически при загрузке класса. `"hello" == "hello"` → true (одна pooled-ссылка).
- **`new String("hello")`** — всегда создаёт **новый** объект в heap **вне** пула, даже если идентичный литерал уже в пуле. `new String("hello") == "hello"` → false.
- **`.intern()`** — явно кладёт строку в пул или возвращает существующую pooled-ссылку. `new String("hello").intern() == "hello"` → true.

Компиляционные константы (`"a" + "b"`) тоже попадают в пул, но рантайм-конкатенация с переменной (`prefix + "b"`) — нет.

**Правило**: `.equals()` для содержимого, `==` только если явно проверяете identity.

### [4-1-q2 | mid] Как работает конкатенация строк начиная с Java 9 и когда всё равно использовать StringBuilder?
С Java 9 (JEP 280) оператор `+` компилируется в **одиночный `invokedynamic`** → `StringConcatFactory.makeConcatWithConstants`. В рантайме bootstrap-метод генерирует оптимальную стратегию (обычно одна byte[]-аллокация точного размера).

**Для одиночного выражения** — `+` эффективнее ручного `StringBuilder`: JVM может предварительно вычислить размер буфера и сделать одну аллокацию.

**Когда `StringBuilder` всё равно нужен**:
- **В циклах** — каждая итерация = отдельный `invokedynamic`-вызов, создающий новый String. `StringBuilder` аккумулирует в единый буфер:
```java
StringBuilder sb = new StringBuilder();
for (String s : list) sb.append(s).append(",");
String result = sb.toString();
```
- **Построение across method calls или conditional branches** — компилятор не может соптимизировать одно выражение.
- **Динамический размер** — когда вы не знаете количество частей заранее.

`StringBuffer` — legacy-версия `StringBuilder` с synchronized. Редко нужна, практически всегда подходит `StringBuilder`.

### [4-1-q3 | senior] Объясните Compact Strings (JEP 254), внутреннюю структуру String Pool и риски чрезмерного interning.
**Compact Strings (Java 9, JEP 254)** — внутренний `char[]` заменён на `byte[]` + `coder`-флаг:
- **LATIN1 (0)** — 1 байт/символ (ASCII и Latin-1).
- **UTF16 (1)** — 2 байта/символ (строки с не-Latin-1).

Для типичного ASCII-доминантного приложения — **экономия памяти в 2×**, без изменений в API.

**String Pool** — нативная hashtable в JVM (не `HashMap`!), размер регулируется `-XX:StringTableSize` (по умолчанию 65536 bucket'ов). Каждый bucket — связанный список.

**Риски `intern()`**:

1. **Collision degradation** — при малом размере таблицы lookup становится O(n) по bucket.
2. **Strong references** — pooled-строки являются GC-roots, **не собираются до выхода JVM**. Interning миллионов уникальных строк → утечка памяти.
3. **Lock contention** — под высокой конкурентной нагрузкой пул может стать bottleneck'ом.

**Правило**: применять `intern()` только к **ограниченному, часто повторяющемуся** множеству значений (enum-подобные, parsed identifiers из фиксированного домена).

Мониторинг: `jcmd <pid> VM.stringtable` покажет размер и коллизии.
