## Title
Operators & Casting

## Summary
Java is strongly typed but supports casting: identity, widening, narrowing (for primitives and objects), and string conversion. Arithmetic **always** promotes `byte`/`short`/`char` to `int`. `instanceof` checks castability at runtime; `ClassCastException` is the failure mode.

## Deep Dive
## Kinds of conversions

- **Identity** — to the same type, free.
- **Widening (primitive)** — smaller to larger: `byte → short → int → long → float → double`. Implicit.
- **Narrowing (primitive)** — explicit cast required, can lose data.
- **Upcasting (object)** — to a parent, implicit.
- **Downcasting (object)** — to a subtype, explicit cast. Mismatch → `ClassCastException`.
- **String conversion** — `"x=" + 42` — anything is converted via `String.valueOf(...)`.

## Widening with precision loss

> [!gotcha]
> `int → float` and `long → double` are widening (no cast needed) but **can lose precision**. `int` has 32 bits of precision; `float` only has 24 bits of mantissa. Values above 2²⁴ (16 777 216) stop fitting exactly. Same for `long → double` above 2⁵³. The JLS candidly calls these "widening primitive conversions that may lose information".

## Integer promotion — the big arithmetic pitfall

In arithmetic expressions `byte`, `short`, `char` are **always** promoted to `int` before the operation:

```java
byte a = 1, b = 2;
byte c = a + b;     // COMPILE ERROR: a + b is an int
byte c = (byte)(a + b);  // OK
```

But compound assignment (`+=`, `-=`, ...) includes an **implicit cast** per the JLS:

```java
byte b = 127;
b += 1;  // compiles, equivalent to b = (byte)(b + 1). b is now -128 (overflow!)
```

> [!gotcha]
> A favourite trick question: compound operators silently overflow with no compiler warning.

## == vs equals()

- On primitives, `==` compares values.
- On objects, `==` compares **references** (identity).
- With String literals `==` often "seems to work" because of the **string pool**: `"hello" == "hello"` → true. But `new String("hello") == "hello"` → false.
- With wrappers — same story via the -128..127 cache.

**Always use `.equals()` for objects.**

## Pattern matching for instanceof (Java 16+)

Old code:

```java
if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}
```

New — shorter and type-safe:

```java
if (obj instanceof String s) {
    System.out.println(s.length());
}
```

**Flow scoping** — `s` is only in scope where the compiler proved the instanceof matched. Works even with negation:

```java
if (!(obj instanceof String s)) return;
// s IS in scope here — the only way to reach this point is if instanceof succeeded
System.out.println(s.length());
```

Java 21 extends pattern matching to `switch` and records — you can deconstruct record components in a single expression.

## Bitwise and shift

- `&`, `|`, `^` — bitwise (no short-circuit on booleans).
- `&&`, `||` — short-circuit (second side not evaluated if unnecessary).
- `<<` — left shift.
- `>>` — signed right shift (preserves sign).
- `>>>` — unsigned right shift (fills with zeros).

## Modulo — sign follows the dividend

`-7 % 3 == -1` (not `2`). For a positive result use `Math.floorMod(-7, 3) == 2`.

## Code
```java
public class OperatorsAndCastingDemo {
    public static void main(String[] args) {
        // === Widening with precision loss ===
        int bigInt = 16_777_217; // 2^24 + 1
        float asFloat = bigInt;  // widening conversion — no cast needed
        System.out.println("int -> float precision loss:");
        System.out.println("  int:   " + bigInt);             // 16777217
        System.out.println("  float: " + (int) asFloat);      // 16777216 (lost!)

        // === Integer promotion trap ===
        byte b1 = 50;
        byte b2 = 60;
        // byte b3 = b1 + b2; // COMPILE ERROR: int cannot be assigned to byte
        byte b3 = (byte)(b1 + b2); // compiles, but: 110 fits, what about overflow?

        byte b4 = 127;
        b4 += 1; // compiles! compound assignment has implicit cast. b4 is now -128
        System.out.println("\nbyte overflow via +=: " + b4); // -128

        // === == vs equals() ===
        String s1 = "hello";
        String s2 = "hello";
        String s3 = new String("hello");
        System.out.println("\nString == comparison:");
        System.out.println("  literal == literal: " + (s1 == s2));    // true (pool)
        System.out.println("  literal == new:     " + (s1 == s3));    // false
        System.out.println("  literal.equals(new):" + s1.equals(s3)); // true

        // === Ternary type promotion ===
        // Ternary promotes to common type
        char c = 'A';
        int result = true ? c : 0; // char promoted to int
        System.out.println("\nTernary char + int: " + result); // 65, not 'A'

        // === Pattern Matching instanceof (Java 16+) ===
        Object obj = "Hello, World!";
        if (obj instanceof String s && s.length() > 5) {
            System.out.println("\nPattern match: " + s.toUpperCase());
        }

        // Flow scoping — s is in scope after a negative check too
        if (!(obj instanceof String s)) {
            return; // s is not in scope here
        }
        // s IS in scope here because we know instanceof succeeded
        System.out.println("Flow scoping: " + s.length());

        // === Modulo sign behavior ===
        System.out.println("\nModulo sign:");
        System.out.println("  -7 % 3 = " + (-7 % 3));              // -1 (not 2!)
        System.out.println("  Math.floorMod(-7, 3) = " +
            Math.floorMod(-7, 3));                                   // 2

        // === Short-circuit vs non-short-circuit ===
        int x = 0;
        boolean shortCircuit = (x != 0) && (10 / x > 1);  // safe: second not evaluated
        // boolean noShortCircuit = (x != 0) & (10 / x > 1); // ArithmeticException!
        System.out.println("\nShort-circuit saved us from division by zero");
    }
}
```

## Tip
Compound assignment (`+=`, `*=`, ...) carries a **hidden cast**. `short s = 32767; s += 1;` compiles and silently overflows. A favourite trick question.

## Spring
### concept
Type casting and instanceof

### springFeature
Spring ConversionService

### explanation
Spring's `ConversionService` handles conversions that go well beyond Java's built-in casting: `String → enum`, `String → Date`, `String → List`, any user-defined types.

When you write `@Value("${server.port}") int port`, Spring's type conversion parses the String property into an `int`. Same for `@RequestParam`, `@PathVariable`, and form data binding.

Understanding the limits of native Java casting makes clear why Spring needs a separate conversion framework for property binding, request mapping, and serialisation.

## Interview
### [2-3-q0 | junior] What kinds of type conversion does Java support?
- **Identity** — same type.
- **Primitive widening** — implicit: `byte → short → int → long → float → double`. Can lose precision (`int → float`, `long → double`).
- **Primitive narrowing** — explicit cast, can lose data.
- **Object upcasting** — to a parent, implicit.
- **Object downcasting** — to a subtype, explicit cast. Mismatch → `ClassCastException`.
- **String conversion** — `"x=" + 42`, via `String.valueOf(...)`.

Guard a downcast with `instanceof`; since Java 16+ use pattern matching: `if (obj instanceof String s) { ... }`.

### [2-3-q1 | junior] Why does `byte b = 1; byte c = b + 1;` not compile, but `b += 1;` does?
In `b + 1` the `byte` is promoted to `int` (integer promotion — Java widens `byte`/`short`/`char` to `int` in any arithmetic), and the literal `1` is already an `int`. The result is `int`, which can't be assigned to `byte` without an explicit cast. Hence `byte c = (byte)(b + 1);`.

But compound assignment operators (`+=`, `-=`, ...) per the JLS carry an **implicit narrowing cast**. So `b += 1` is equivalent to `b = (byte)(b + 1)`. That means compound assignment can silently overflow: `byte b = 127; b += 1;` gives `b = -128` with no warning.

### [2-3-q2 | mid] Can a widening conversion lose data? Give an example.
Yes. `int → float` and `long → double` are widening (no cast required) but **can lose precision**.

`int` has 32 bits of integer precision; `float` only has 24 bits of mantissa. Any value above 2²⁴ = 16 777 216 can lose precision:

```java
int n = 16_777_217;
float f = n;            // 16777216.0f — we lost 1
```

Similarly `long → double` loses precision above 2⁵³. A real source of bugs in financial maths — the JLS honestly calls these "widening primitive conversions that may lose information".

### [2-3-q3 | senior] How does pattern matching for `instanceof` and flow scoping work in Java 16+?
Pattern matching `instanceof` introduces a **binding variable** whose scope is determined by flow analysis — it's only visible where the compiler proved the pattern matched.

```java
if (obj instanceof String s) {
    // s is in scope here
}
```

With negation:
```java
if (!(obj instanceof String s)) return;
// s IS in scope here — the only way to reach this point is if instanceof succeeded
```

In boolean expressions:
- `obj instanceof String s && s.length() > 5` — OK, `s` is available on the RHS of `&&`, since we only get there when the match succeeded.
- `obj instanceof String s || s.isEmpty()` — **illegal**: on the RHS of `||` the pattern might not have matched.

Java 21 extends pattern matching to `switch` and records — you can deconstruct record components in a single expression.
