## Title
Primitive Types & Wrappers

## Summary
В Java 8 примитивных типов: `boolean` (1 бит), `byte` (1 байт, -128..127), `short` (2 байта), `int` (4 байта), `long` (8 байт), `float` (4 байта), `double` (8 байт), `char` (2 байта, UTF-16). Autoboxing — неявная упаковка примитива в класс-обёртку. Целочисленные обёртки в диапазоне -128..+127 кэшируются JVM.

## Deep Dive
## Восемь примитивов

| Тип       | Размер     | Диапазон / значения                 |
|-----------|------------|-------------------------------------|
| `boolean` | 1 бит\*    | true / false                        |
| `byte`    | 1 байт     | -128..127                           |
| `short`   | 2 байта    | -32768..32767                       |
| `int`     | 4 байта    | ±2.1 × 10⁹                          |
| `long`    | 8 байт     | ±9.2 × 10¹⁸                         |
| `float`   | 4 байта    | IEEE 754 одинарная точность         |
| `double`  | 8 байт     | IEEE 754 двойная точность           |
| `char`    | 2 байта    | UTF-16 code unit (0..65535)         |

\* На самом деле `boolean` занимает байт или int-слот в зависимости от контекста (JVM spec не фиксирует размер).

Примитивы лежат на **стеке** или inline внутри объектов. Обёртки (`Integer`, `Long` и т.д.) — полноценные объекты в **куче** с ~16 байтами overhead каждый.

## Autoboxing и его ловушки

Autoboxing добавлен в Java 5. Компилятор молча вставляет `Integer.valueOf(n)` при упаковке и `intObj.intValue()` при распаковке.

> [!gotcha]
> Распаковка `null` бросает `NullPointerException`. `Integer x = null; int y = x;` — компилируется чисто, падает в рантайме. Это источник #1 auto­boxing-багов.

Менее очевидная ловушка — тернарный оператор:

```java
Integer x = null;
Integer result = cond ? x : 0;  // NPE! x распакуется в int, потом упакуется обратно
```

Смешение `Integer` и `int` в тернарнике заставляет компилятор распаковать **обе ветки** до общего типа `int`.

## Кэш Integer [-128..127]

`Integer.valueOf(n)` кэширует экземпляры для значений **-128..127** (верхнюю границу можно поднять через `-XX:AutoBoxCacheMax`):

```java
Integer.valueOf(100) == Integer.valueOf(100)  // true — один объект из кэша
Integer.valueOf(200) == Integer.valueOf(200)  // false — разные объекты!
```

Аналогичное кэширование у `Byte`, `Short`, `Long` (-128..127), `Character` (0..127) и `Boolean` (всегда кэшируется).

> [!tip]
> **Всегда** сравнивайте обёртки через `.equals()`, никогда через `==`. Это настолько важное правило, что современные IDE подсвечивают `==` на `Integer` как warning.

## Производительность и память

`int[]` хранит 4 байта на элемент подряд в памяти. `List<Integer>` — это массив ссылок (~4-8 байт) на отдельные объекты `Integer` (~16 байт каждый). Получается в 5-6× больше памяти и плохая кэш-локальность из-за pointer chasing.

> [!production]
> Для hot-path кода и больших коллекций чисел: используйте `IntStream`, примитивные массивы или библиотеки типа Eclipse Collections (`IntArrayList`, `IntIntHashMap`), Koloboke, HPPC. Project Valhalla со временем принесёт настоящие value types — `List<int>` без боксинга.

## Ловушки плавающей точки

- `0.1 + 0.2 == 0.3` → **false** (IEEE 754 не представляет точно).
- Никогда не используйте `float`/`double` для денег. `BigDecimal` — правильный инструмент.
- `Double.NaN != Double.NaN` → **true** (NaN не равен ничему, даже себе).
- Но `Double.valueOf(Double.NaN).equals(Double.valueOf(Double.NaN))` → **true** — `equals()` специально обрабатывает NaN для консистентности в коллекциях.

## Code
```java
public class PrimitivesAndWrappersDemo {
    public static void main(String[] args) {
        // === Integer Cache Trap ===
        Integer a = 127;
        Integer b = 127;
        Integer c = 128;
        Integer d = 128;
        System.out.println("127 == 127: " + (a == b));   // true (cached)
        System.out.println("128 == 128: " + (c == d));   // false (not cached!)
        System.out.println("128.equals(128): " + c.equals(d)); // true (correct way)

        // === Autoboxing NPE Trap ===
        Integer nullInteger = null;
        try {
            int boom = nullInteger; // Compiles fine. NPE at runtime!
        } catch (NullPointerException e) {
            System.out.println("\nAutoboxing NPE: unboxing null Integer");
        }

        // === Ternary Operator Unboxing Trap ===
        Integer x = null;
        // This throws NPE because the ternary unboxes to int then reboxes!
        try {
            Integer result = true ? x : 0; // x gets unboxed -> NPE
        } catch (NullPointerException e) {
            System.out.println("Ternary unboxing NPE: mixing Integer and int");
        }

        // === Floating Point ===
        System.out.println("\n0.1 + 0.2 == 0.3: " + (0.1 + 0.2 == 0.3)); // false!
        System.out.println("0.1 + 0.2 = " + (0.1 + 0.2)); // 0.30000000000000004

        // Use BigDecimal for exact arithmetic
        var bd1 = new java.math.BigDecimal("0.1");
        var bd2 = new java.math.BigDecimal("0.2");
        var bd3 = new java.math.BigDecimal("0.3");
        System.out.println("BigDecimal 0.1 + 0.2 == 0.3: " +
            (bd1.add(bd2).compareTo(bd3) == 0)); // true

        // === NaN Weirdness ===
        System.out.println("\nNaN == NaN: " + (Double.NaN == Double.NaN)); // false!
        System.out.println("NaN equals NaN: " +
            Double.valueOf(Double.NaN).equals(Double.valueOf(Double.NaN))); // true!

        // === Performance: autoboxing in loops ===
        long start = System.nanoTime();
        Long sum = 0L; // Long, not long — creates ~2 billion temporary objects!
        for (int i = 0; i < 1_000_000; i++) {
            sum += i; // unbox, add, rebox on every iteration
        }
        long boxedTime = System.nanoTime() - start;

        start = System.nanoTime();
        long sum2 = 0L; // primitive — no boxing
        for (int i = 0; i < 1_000_000; i++) {
            sum2 += i;
        }
        long primitiveTime = System.nanoTime() - start;
        System.out.printf("\nBoxed loop:     %,d ns%n", boxedTime);
        System.out.printf("Primitive loop: %,d ns%n", primitiveTime);
        System.out.printf("Slowdown:       %.1fx%n", (double) boxedTime / primitiveTime);
    }
}
```

## Tip
Если видите `Long`, `Integer` или `Double` как переменную-аккумулятор в цикле — это почти всегда ошибка. Используйте примитив (`long`, `int`, `double`), чтобы не создавать миллионы мусорных объектов-обёрток.

## Spring
### concept
Autoboxing and null primitives

### springFeature
Spring Data JPA entity mapping

### explanation
JPA-поля, маппящиеся на nullable-колонки БД, должны использовать обёртки (`Integer`, `Long`), а не примитивы.

Примитив `int` по умолчанию равен 0, что неотличимо от реального хранимого нуля — JPA не может сказать, была ли колонка NULL или 0. `Integer` сохраняет семантику null.

Обратная сторона: сервисный слой должен обрабатывать возможные null при работе с полями сущностей, и autoboxing-NPE становится реальным риском в вычислениях над данными из БД.

## Interview
### [2-1-q0 | junior] Перечислите примитивные типы Java и их размеры.
**8 примитивов:**

- `boolean` — true/false (размер не фиксирован спецификацией JVM)
- `byte` — 1 байт, -128..127
- `short` — 2 байта, -32768..32767
- `int` — 4 байта, ±2.1 × 10⁹
- `long` — 8 байт, ±9.2 × 10¹⁸
- `float` — 4 байта, IEEE 754
- `double` — 8 байт, IEEE 754
- `char` — 2 байта, UTF-16 code unit

Все примитивы лежат на стеке или inline в объектах; overhead заголовка объекта у них отсутствует.

### [2-1-q1 | junior] Почему `Integer == Integer` иногда даёт `true`, а иногда `false`?
`==` на объектах сравнивает **ссылки**, не значения. `Integer.valueOf()` кэширует экземпляры для значений **-128..127**.

```java
Integer a = 127, b = 127;  // один кэшированный объект → a == b (true)
Integer a = 128, b = 128;  // два разных объекта → a == b (false)
```

Верхнюю границу можно поднять через `-XX:AutoBoxCacheMax`. Аналогичное кэширование у `Byte`, `Short`, `Long` (-128..127), `Character` (0..127), `Boolean` (всегда).

Правильный способ — `.equals()`. Современные IDE подсвечивают `==` на обёртках как предупреждение.

### [2-1-q2 | mid] Опишите сценарий, в котором autoboxing вызывает `NullPointerException`, который трудно найти.
Классика — тернарный оператор:

```java
Integer x = null;
Integer result = cond ? x : 0;  // NPE!
```

Тернарник видит `Integer` и `int`, и распаковывает `x` в `int` (NPE на null) перед обратным боксингом.

Другой частый случай — `Map<String, Integer>.get(key)` возвращает `null`, и результат передаётся в метод, ожидающий `int`. Неявная распаковка падает, а трейс указывает на строку без видимого null-dereference.

Третий — стримы: `list.stream().mapToInt(Integer::intValue)` на списке с `null`-элементами.

### [2-1-q3 | senior] Какие последствия по памяти и производительности у `List<Integer>` против `int[]`, и какие есть альтернативы?
- `int[]` — 4 байта на элемент подряд в памяти, один заголовок объекта (~16 байт overhead).
- `List<Integer>` — массив ссылок (4-8 байт каждая) на объекты `Integer` (~16 байт заголовок + 4 байта значение + padding). Итого ~5-6× больше памяти + плохая кэш-локальность из-за pointer chasing.

На бенчмарках примитивные массивы обходят боксированные коллекции в 3-10×.

**Альтернативы:**
- Eclipse Collections: `IntArrayList`, `IntIntHashMap`
- Koloboke, HPPC — специализированные коллекции
- `IntStream` и примитивные массивы в JDK покрывают большинство кейсов
- Project Valhalla в будущих версиях Java даст value types и `List<int>` без боксинга
