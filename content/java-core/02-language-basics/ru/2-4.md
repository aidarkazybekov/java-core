## Title
Control Flow

## Summary
Классика — `if/else`, `for`, `while`, `do-while`, `switch`. Современная Java добавила switch-выражения (Java 14+) со стрелочным синтаксисом и pattern matching в `switch` (Java 21+). Знание enhanced for-loop, labeled break и поведения `finally` отличает зрелого разработчика.

## Deep Dive
## Виды циклов

- **`while`** — цикл с предусловием. Условие проверяется **до** первой итерации. Используется, когда число итераций заранее неизвестно.
- **`do-while`** — цикл с постусловием. Тело выполняется **минимум один раз**.
- **`for`** — цикл со счётчиком. Начальное и конечное значения заданы в заголовке.
- **Enhanced for-loop (for-each)** — `for (String s : collection)`. Компилируется в `Iterator.hasNext()` / `next()`.

> [!gotcha]
> Нельзя удалять элементы из коллекции во время enhanced-for — получите `ConcurrentModificationException`. Итератор отслеживает `modCount`; прямой `collection.remove()` меняет его и fail-fast проверка срабатывает. Используйте `Iterator.remove()` или `Collection.removeIf(...)`.

## Switch — эволюция

**Классический** switch имеет fall-through по умолчанию (каждый `case` требует `break`). Пропуск `break` — распространённый баг.

**Switch-выражение (Java 14+)** со стрелочным синтаксисом — без fall-through, возвращает значение:

```java
int result = switch (x) {
    case 1 -> 10;
    case 2 -> 20;
    default -> 0;
};
```

Для многострочного case используется `yield`:

```java
String msg = switch (code) {
    case 200 -> "OK";
    case 500 -> {
        log.error("Server error");
        yield "Internal Error";
    }
    default -> "Unknown";
};
```

Switch-выражение должно быть **exhaustive** — покрывать все варианты или иметь `default`.

## Pattern matching в switch (Java 21+)

```java
String desc = switch (obj) {
    case Integer i when i > 0 -> "positive " + i;
    case Integer i            -> "non-positive " + i;
    case String s             -> "string, length=" + s.length();
    case null                 -> "null";
    default                   -> "other";
};
```

**Guarded patterns** (`when`) фильтруют после type-match. Guarded-версия должна идти **до** unguarded.

**Sealed types** + switch — компилятор проверяет exhaustiveness без `default`:

```java
sealed interface Shape permits Circle, Square, Triangle {}
double area = switch (shape) {
    case Circle c   -> Math.PI * c.r() * c.r();
    case Square s   -> s.side() * s.side();
    case Triangle t -> t.base() * t.height() / 2;
};  // default не нужен
```

Добавление нового permitted-подтипа — compile error во всех switch над этим типом. Пропуск case невозможен.

## Labeled break / continue

Недооценённый инструмент для вложенных циклов:

```java
outer:
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        if (matrix[i][j] == target) {
            break outer;  // выходит из ВНЕШНЕГО цикла
        }
    }
}
```

Без метки `break` выходит только из ближайшего цикла. Метки работают и на блоках, иногда используются как структурированная альтернатива early-return в сложной валидации.

## Short-circuit — гарантия корректности, не оптимизация

- `&&` не вычисляет правую часть, если левая = `false`.
- `||` не вычисляет правую часть, если левая = `true`.

Именно поэтому безопасно: `if (obj != null && obj.getValue() > 0)`. Это не просто ускорение — это корректность.

Операторы `&` и `|` на booleans вычисляют **обе стороны всегда** — для логики не применять.

## finally — два сюрприза

`finally` выполняется **всегда**, кроме двух случаев: `System.exit(...)` и крах JVM.

> [!gotcha]
> `try { return 1; } finally { return 2; }` возвращает **2**. Return в finally **перезаписывает** сохранённое возвращаемое значение try. Если try бросает исключение и в finally есть return — исключение молча проглатывается. Статические анализаторы флагают этот паттерн — никогда не возвращайте из finally.

`try-with-resources` заменяет большинство `finally`-блоков и корректно обрабатывает «исключение в `close()`» через suppressed exceptions.

## Code
```java
public class ControlFlowDemo {
    public static void main(String[] args) {
        // === Switch Expression (Java 14+) ===
        int statusCode = 404;
        String message = switch (statusCode) {
            case 200 -> "OK";
            case 301 -> "Moved Permanently";
            case 404 -> "Not Found";
            case 500 -> "Internal Server Error";
            default -> {
                System.out.println("Unknown code: " + statusCode);
                yield "Unknown"; // 'yield' for multi-line switch expression
            }
        };
        System.out.println(statusCode + ": " + message);

        // === Pattern Matching Switch (Java 21+) ===
        Object obj = 42;
        String description = switch (obj) {
            case Integer i when i > 0  -> "Positive int: " + i;
            case Integer i             -> "Non-positive int: " + i;
            case String s              -> "String of length " + s.length();
            case null                  -> "null value";
            default                    -> "Other: " + obj.getClass().getSimpleName();
        };
        System.out.println("\nPattern switch: " + description);

        // === Labeled Break — escaping nested loops ===
        System.out.println("\nLabeled break:");
        int[][] matrix = {{1, 2, 3}, {4, 5, 6}, {7, 8, 9}};
        int target = 5;
        search:
        for (int row = 0; row < matrix.length; row++) {
            for (int col = 0; col < matrix[row].length; col++) {
                if (matrix[row][col] == target) {
                    System.out.println("Found " + target + " at [" + row + "][" + col + "]");
                    break search;
                }
            }
        }

        // === Enhanced for-loop ConcurrentModificationException ===
        var list = new java.util.ArrayList<>(java.util.List.of("a", "b", "c", "d"));
        System.out.println("\nBefore removal: " + list);

        // WRONG: throws ConcurrentModificationException
        // for (String s : list) { if (s.equals("b")) list.remove(s); }

        // RIGHT: use removeIf
        list.removeIf(s -> s.equals("b"));
        System.out.println("After removeIf: " + list);

        // === Finally block return override ===
        System.out.println("\nFinally return override: " + finallyReturns()); // 2!

        // === Short-circuit matters for correctness ===
        String nullStr = null;
        if (nullStr != null && nullStr.length() > 0) {
            System.out.println("not reached");
        }
        System.out.println("Short-circuit prevented NPE on null string");
    }

    // This returns 2, not 1 — finally overrides the try return
    static int finallyReturns() {
        try {
            return 1;
        } finally {
            return 2; // WARNING: this swallows the try return!
        }
    }
}
```

## Tip
Когда switch-выражение не покрывает все варианты, компилятор **требует** `default`. Это одна из причин предпочитать switch-выражения цепочкам `if-else` — компилятор гарантирует обработку всех случаев.

## Interview
### [2-4-q0 | junior] Какие виды циклов есть в Java, когда применять каждый?
- **`while`** — цикл с предусловием. Условие проверяется до итерации. Для случаев, когда число итераций неизвестно.
- **`do-while`** — цикл с постусловием. Тело выполняется хотя бы раз. Подходит для меню, валидации ввода.
- **`for`** — со счётчиком. Известное число итераций.
- **Enhanced for-loop (`for-each`)** — для итерации по коллекциям и массивам. Внутри компилируется в `Iterator.hasNext()/next()`.

### [2-4-q1 | junior] Почему нельзя удалять элементы коллекции внутри enhanced for-loop?
Enhanced for-loop компилируется в вызовы `Iterator.hasNext()` / `next()`. Большинство коллекций имеют fail-fast итератор, который отслеживает `modCount` (счётчик модификаций).

Когда вы вызываете `collection.remove(...)` напрямую (не через итератор), `modCount` увеличивается, а ожидаемый счётчик итератора — нет. На следующем `hasNext()`/`next()` срабатывает проверка и кидается `ConcurrentModificationException`.

**Решения:**
- `Iterator.remove()` — обновляет ожидаемый счётчик.
- `Collection.removeIf(Predicate)` (Java 8+) — использует итератор внутри.
- `CopyOnWriteArrayList` / `ConcurrentHashMap` — weak-consistent итераторы, CME не кидают.

### [2-4-q2 | mid] Что вернёт `try { return 1; } finally { return 2; }` и почему?
Вернёт **2**. Когда `try` выполняет `return 1`, значение 1 сохраняется, но перед реальным возвратом метода выполняется `finally`. `return 2` в `finally` **перезаписывает** сохранённое значение.

То же с исключениями: если `try` бросает исключение, а `finally` имеет `return`, исключение **молча проглатывается**. Если оба — try и finally — бросают исключение, исключение из finally заменяет исключение из try (оригинал теряется, если не использовать `addSuppressed`).

Статические анализаторы и IDE флагают return из finally. **Никогда не возвращайте из finally.**

### [2-4-q3 | senior] Объясните pattern matching в `switch` с guarded-паттернами и sealed-типами. Как работает exhaustiveness?
`switch` с pattern matching (Java 21) позволяет type-patterns (`case String s ->`) и guarded patterns (`case String s when s.length() > 5 ->`). `when` — фильтр после type-match. Guarded-версия должна идти **до** unguarded (компилятор идёт сверху вниз, первый match побеждает).

С **sealed types** компилятор гарантирует exhaustiveness без `default`:

```java
sealed interface Shape permits Circle, Square, Triangle {}
double area = switch (shape) {
    case Circle c   -> Math.PI * c.r() * c.r();
    case Square s   -> s.side() * s.side();
    case Triangle t -> t.base() * t.height() / 2;
};
```

Добавление нового permitted-подтипа — compile error во всех switch над этим типом. Это надёжная замена Visitor pattern.

Null обрабатывается явно: `case null ->` или `switch` по умолчанию бросит NPE.
