## Title
Classes & Objects

## Summary
A class is a compile-time blueprint with fields and methods; an object is a runtime instance allocated on the heap. The reference variable lives on the stack; the object lives on the heap. Java **always** passes by value — for objects, the reference is copied, not the object itself. The four OOP pillars: abstraction, encapsulation, inheritance, polymorphism.

## Deep Dive
## Class vs object vs instance

- **Class** — a compile-time construct. The `.class` file contains the constant pool, field descriptors, method bytecode, access flags. The JVM loads a class via a ClassLoader, verifies and links it **once**.
- **Object** — a runtime instance. On `new MyClass()` the JVM allocates heap memory for instance fields plus the object header.
- **Instance** — same as object. The reference to the instance lives on the stack (or inside another object, if it is a field).

## Object layout on HotSpot (64-bit, compressed oops)

| Part | Size | Description |
|------|------|-------------|
| Mark word | 8 bytes | hashCode, GC age, lock state, biased locking |
| Klass pointer | 4 bytes | compressed, points to Metaspace metadata |
| Instance fields | N bytes | fields with reordering and padding |
| Array length | 4 bytes | arrays only, after the header |
| Padding | 0-7 bytes | aligned to 8-byte boundaries |

Even an empty `new Object()` consumes **16 bytes** (12 header + 4 padding).

Turning off compressed oops (`-XX:-UseCompressedOops`) makes the klass pointer 8 bytes → overhead grows.

## The four OOP pillars

- **Abstraction** — identify the essentials, hide implementation details. Through `abstract` classes and interfaces.
- **Encapsulation** — bundle data with methods and protect state through access modifiers. `private` fields, access via getters/setters (or none at all, for immutables).
- **Inheritance** — new classes built on existing ones. Java supports only **single** class inheritance (but multiple interface implementation).
- **Polymorphism** — one entity, many forms: overloading (compile-time, by signature) and overriding (runtime, virtual dispatch).

## Identity vs equality

- `==` — compares **references** (pointers).
- `.equals()` — compares **value** (if overridden).

The default `Object.equals()` is equivalent to `==` — it checks identity. That is **almost never** what domain code wants.

> [!gotcha]
> The `equals()` / `hashCode()` contract: equal objects **must** have the same `hashCode`. Violating this causes the object to be "lost" inside a HashMap — you put it under a key, and `get` returns null. Always override both together.

## Pass-by-value — always

Java **does not have** pass-by-reference. For reference types, the **pointer is copied** (the pointer itself is a value):

```java
void tryReassign(Employee e) {
    e = new Employee("Ghost", 0);   // rebinding the local copy — caller doesn't see
}

void mutate(Employee e) {
    e.salary = 999_999;              // mutating the object — caller sees
}
```

**Reassigning the parameter** has no effect on the caller. **Mutating the object** through the copied reference is visible to everyone holding a reference to the same object.

## Object lifecycle

1. **Allocation** on the heap (typically Eden in Young Gen).
2. **Initialisation**: fields → init blocks → constructor, parent to child.
3. **Use** — through reachable references.
4. **Unreachable** — when no strong references remain, the object is eligible for GC.
5. **Reclaim** — GC frees the memory (in a future cycle).

Between "unreachable" and "reclaim" an object can be soft/weak/phantom-reachable:

- **SoftReference** — GC reclaims only under memory pressure (caches).
- **WeakReference** — GC reclaims at the next cycle (WeakHashMap).
- **PhantomReference** — the object is already collected but not yet freed; used for cleanup actions.

> [!production]
> `finalize()` is **deprecated since Java 9**. Use `try-with-resources` and `java.lang.ref.Cleaner`. Finalizers run in non-deterministic order, can "resurrect" an object (re-stash its reference into a field), and run on a separate thread — a source of leaks and unpredictable behaviour.

## Nested and inner classes

- **Static nested class** — holds no reference to the enclosing instance. The preferred form.
- **Inner class** (non-static) — **implicitly holds** a reference to the enclosing instance. If it outlives the outer (listener, callback on another thread) → **memory leak**.
- **Local class** — declared inside a method. Can capture effectively-final locals.
- **Anonymous class** — a local with a one-off implementation. Pre-Java-8 it was the de-facto lambda; nowadays lambdas win for functional interfaces.

## Code
```java
// Demonstrates class structure, object creation, and reference semantics
public class Employee {
    // Instance fields (stored on the heap per object)
    private final String name;
    private double salary;

    // Static field (stored in metaspace, shared across all instances)
    private static int headcount = 0;

    public Employee(String name, double salary) {
        this.name = name;
        this.salary = salary;
        headcount++;
    }

    // Passing references by value demo
    public static void tryReassign(Employee e) {
        e = new Employee("Ghost", 0); // caller's reference is unchanged
    }

    public static void mutate(Employee e) {
        e.salary = 999_999;           // caller sees the mutation
    }

    public static void main(String[] args) {
        Employee a = new Employee("Alice", 80_000);
        Employee b = a;               // both point to the same heap object

        System.out.println(a == b);   // true  (same reference)

        tryReassign(a);
        System.out.println(a.name);   // "Alice" (unchanged)

        mutate(a);
        System.out.println(a.salary); // 999999.0 (mutated via copied ref)
        System.out.println(b.salary); // 999999.0 (b points to same object)
    }
}
```

## Tip
When an interviewer asks about pass-by-value, draw a stack/heap diagram with two reference variables pointing to the same heap object — it instantly proves you understand the mechanics.

## Spring
### concept
Classes & Objects

### springFeature
Spring Bean Lifecycle

### explanation
Spring manages object creation through an IoC container. Instead of calling `new`, you declare beans (classes) and Spring instantiates, configures and wires them.

Understanding how objects are created and referenced is essential to scopes:
- **singleton** — one instance per ApplicationContext; every injection gets the same reference.
- **prototype** — a new heap object per injection; GC reclaims once references drop.
- **request / session** — lifecycle tied to HTTP boundaries.

A classic bug: a singleton depends directly on a request-scoped bean — it captures one copy forever. Fixed with a scoped proxy.

## Interview
### [3-1-q0 | junior] What's the difference between a class and an object in Java?
- **Class** — a compile-time template, a `.class` file with field definitions, method bytecode, and metadata. Loaded once by a ClassLoader.
- **Object** — a runtime instance allocated on the heap at `new`. Many objects can come from one class.
- The reference variable (stack) points to the object (heap).

The class lives in Metaspace; the object lives in the Heap; the reference lives on the stack or inside another object.

### [3-1-q1 | junior] What are the four pillars of OOP? Explain each.
- **Abstraction** — pick the essentials, hide the details. Interfaces and `abstract` classes let you code against a contract without knowing the implementation.
- **Encapsulation** — bundle data with methods and protect state. `private` fields, access via methods. Taken to its extreme — immutable objects with no setters at all.
- **Inheritance** — a new class reuses an existing one. Java supports **single** class inheritance, multiple interface implementation.
- **Polymorphism** — one entity in many forms: overloading (by signature, compile-time) and overriding (virtual dispatch, runtime).

### [3-1-q2 | mid] Is Java pass-by-reference or pass-by-value? Explain with an example.
Java is **always pass-by-value**. For primitives, the value is copied. For objects, the **reference** is copied (a pointer is also a value).

Inside a method, reassigning the parameter (`e = new Employee(...)`) has no effect on the caller. But mutating the object through the copied reference (`e.salary = 999_999`) is visible to everyone holding a reference to the same object.

```java
Employee a = new Employee("Alice", 80_000);
tryReassign(a);      // inside: e = new Employee(...) — a unchanged
mutate(a);           // inside: e.salary = ... — a.salary changes
```

A common confusion: "passing a reference" ≠ pass-by-reference. Java passes the reference's value, not the variable itself.

### [3-1-q3 | senior] Explain the layout of a Java object on HotSpot and why an empty `new Object()` takes 16 bytes.
On HotSpot 64-bit with compressed oops an object has:

- **Mark word** — 8 bytes (hashCode, GC age, lock state, biased locking).
- **Klass pointer** — 4 bytes (compressed; points to Metaspace metadata).
- **Instance fields** — N bytes with reordering and padding.
- **Array length** — 4 bytes, arrays only.
- **Padding** — aligned to 8-byte boundaries.

The header is 8 + 4 = **12 bytes**. `new Object()` has no fields but is padded up to 16 (8-byte alignment). With compressed oops off (`-XX:-UseCompressedOops`) the klass pointer is 8 bytes — the object becomes ~24 bytes.

For an array, add 4 more bytes for `length`, plus content.

This is why `List<Integer>` is so fat compared to `int[]` — every `Integer` is a 16-byte object for 4 bytes of data.

### [3-1-q4 | senior] What's the difference between a static nested class and an inner class, and what's the risk of inner classes?
- **Static nested class** — a regular class nested inside another. Does **not** hold a reference to the enclosing instance. Can be instantiated without an outer-class object.
- **Inner (non-static) class** — implicitly holds a reference to the enclosing instance through a hidden `this$0` field. Can only be instantiated via an outer-class object.

**The risk of an inner class** is a memory leak. If the inner class is passed as a listener/callback and outlives the outer, the `this$0` reference keeps GC from reclaiming the outer → everything the outer transitively holds leaks with it.

Classic: an inner `Runnable` submitted to a long-lived `ExecutorService`. The outer might be an Activity / ApplicationContext / service — and the entire object graph it holds stays alive.

**Rule**: default to `static nested class`. Make it inner only when you genuinely need access to enclosing-instance state.
