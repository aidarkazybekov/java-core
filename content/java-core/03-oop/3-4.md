## Title
Polymorphism

## Summary
Polymorphism is the ability of one entity to take many forms. Three kinds in Java: **ad-hoc (static)** — method overloading; **parametric** — generics; **dynamic (subtype)** — method overriding via virtual dispatch through the vtable.

## Deep Dive
## Three kinds of polymorphism

- **Ad-hoc (compile-time)** — method **overloading**. Several methods with the same name, different parameters. The compiler picks the right one by argument types.
- **Parametric** — generics. `List<T>` is a single class specialised per type at compile time.
- **Subtype / Dynamic** — **overriding**. Code works against a parent reference, the JVM picks the implementation by the object's runtime type.

## Overloading vs Overriding

| | Overload | Override |
|--|----------|----------|
| When | Compile-time | Runtime |
| Where | Same class (or inherited) | In the subclass |
| What changes | Parameters | Implementation (same signature) |
| Dispatch | Static | Dynamic (vtable) |
| `@Override` | No | Yes |

**Method signature** = name + parameter list (types, order). **Not** in the signature: return type, checked exceptions, modifiers.

## Virtual method dispatch — how the JVM does it

Every class has a **vtable** built at class load. It's an array of pointers to method implementations. An override replaces the vtable entry with the subclass's implementation.

At the call site:
1. The JVM reads the klass pointer from the object header.
2. Locates the vtable via the class metadata in Metaspace.
3. Jumps to the method at a fixed index.

HotSpot optimises further:
- **Monomorphic inline cache** — call always sees one type → method inlined directly.
- **Bimorphic** — two types → an if-branch.
- **Megamorphic** — three or more → full vtable lookup.

Deoptimization happens if a new subclass loads that breaks the monomorphic assumption.

## Overload resolution rules

The compiler picks the **most specific** applicable method. Priority (Java 5+ with autoboxing):

1. **Exact match / widening** (no boxing, no varargs).
2. **Autoboxing** (if widening didn't apply).
3. **Varargs** (last resort).

```java
static void f(long x)     {}
static void f(Integer x)  {}
static void f(Object... x) {}

int i = 42;
f(i);  // calls f(long) — widening beats boxing
```

> [!gotcha]
> Adding an overload can silently change dispatch for existing call sites. Ask whether you're introducing a more-specific signature that may win over the one you used to rely on.

## Covariant return types (Java 5+)

An override can return a **narrower** type than the parent:

```java
class Shape { Shape copy() { ... } }
class Circle extends Shape {
    @Override Circle copy() { ... }  // OK, covariant return
}
```

**Contravariant parameters** aren't supported: changing a parameter type creates an overload, not an override — a common mistake.

## Bridge methods

The compiler generates **synthetic bridge methods** to preserve polymorphism after generic **type erasure** and for covariant returns.

Example:
```java
class StringList extends ArrayList<String> {
    @Override public String get(int i) { ... }  // returns String
}
```

After erasure the parent's `get` returns `Object`. The compiler emits a bridge method `Object get(int)` that delegates to `String get(int)` — preserving the vtable contract.

Bridge methods are visible to reflection: `Method.isBridge()`. Frameworks that enumerate declared methods must handle them.

## Liskov Substitution Principle

LSP: any code that works with `Parent` must keep working if you substitute a `Child`. A subclass **cannot**:
- Strengthen preconditions (accept fewer inputs).
- Weaken postconditions (guarantee less output).
- Throw unexpected exceptions.

LSP violations are a frequent bug in inheritance hierarchies (`Square extends Rectangle` is the canonical example).

## Code
```java
public abstract class Shape {
    public abstract double area();

    @Override
    public String toString() {
        return getClass().getSimpleName() + " area=" + area();
    }
}

public class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle extends Shape {
    private final double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override
    public double area() { return w * h; }
}

// --- Overloading resolution demo ---
public class OverloadDemo {
    static String describe(Object o)  { return "Object";  }
    static String describe(String s)  { return "String";  }
    static String describe(long l)    { return "long";    }
    static String describe(Integer i) { return "Integer"; }

    public static void main(String[] args) {
        // Runtime polymorphism
        Shape s = new Circle(5);
        System.out.println(s);         // "Circle area=78.539..."
        s = new Rectangle(3, 4);
        System.out.println(s);         // "Rectangle area=12.0"

        // Overloading resolution
        int x = 42;
        System.out.println(describe(x));          // "long" (widening beats boxing)
        System.out.println(describe("hello"));    // "String" (most specific)
        System.out.println(describe((Object)"a"));// "Object" (explicit cast)
    }
}
```

## Tip
When explaining polymorphism in an interview, always give a concrete example: declare a variable with a parent type, assign a child instance, call an overridden method. Then explain which version runs and why.

## Spring
### concept
Polymorphism

### springFeature
Spring Interface-Based Design

### explanation
Spring leans heavily on polymorphism. You code against **interfaces** (`UserService`, `UserRepository`) and Spring injects the concrete implementation at runtime.

Pure runtime polymorphism. Consequences:
- Implementations are swappable (`JpaUserRepository` vs `MongoUserRepository`) without changing consumers.
- Mocks in tests work because consumers depend on the interface, not the implementation.
- JDK dynamic proxy AOP only works for beans that implement an interface (the rest use CGLIB subclassing).

## Interview
### [3-4-q0 | junior] What's the difference between method overloading and method overriding?
- **Overloading** — compile-time polymorphism. Same method name, different parameter lists. The compiler resolves the call by argument types.
- **Overriding** — runtime polymorphism. A subclass redefines a parent's method with the **same signature**. The JVM resolves the call by the object's runtime type via the vtable.

Overloading — within one class (or inherited). Overriding — between parent and child.

A method signature includes name and parameters; return type, exceptions, and modifiers are **not** part of it. So you can't overload by return type or exception list.

### [3-4-q1 | mid] How does the JVM internally perform dynamic method dispatch?
Each class has a **vtable** (virtual method table) built at class load — an array of pointers to method implementations. When a subclass overrides a method, its vtable entry points to the subclass's implementation.

At the call site:
1. The JVM reads the **klass pointer** from the object header.
2. Finds the vtable via class metadata (Metaspace).
3. Jumps to the method at a fixed index.

**HotSpot optimisations**:
- **Monomorphic inline cache** — call site always sees one type → the method is inlined directly.
- **Bimorphic** — two types → an `if` branch.
- **Megamorphic** — 3+ types → full vtable lookup.

If a new subclass loads that breaks the monomorphic assumption, **deoptimization** triggers — fallback to the interpreter and recompilation without that optimisation. Shows up as a latency spike in dashboards.

### [3-4-q2 | senior] What are bridge methods and how do they relate to polymorphism?
Bridge methods are **synthetic** methods the compiler generates to preserve polymorphism after generic **type erasure** and for **covariant return types**.

Erasure example:
```java
class StringList extends ArrayList<String> {
    @Override public String get(int i) { ... }
}
```

After erasure the parent's `get` returns `Object`. To maintain the vtable contract the compiler synthesises:
```java
// synthetic, added by the compiler
public Object get(int i) { return this.<real>get(i); }
```

Detect them with `Method.isBridge()`. Frameworks that iterate `Class.getDeclaredMethods()` must skip them — otherwise you get duplicates when building proxies or serialisers.

They also appear with covariant returns (a subclass narrows the return type).

### [3-4-q3 | senior] What is the Liskov Substitution Principle — give an example of a violation.
LSP: any code that works with `Parent` must work correctly when a `Child` is substituted.

A subclass must not:
- Strengthen a precondition (require more from input).
- Weaken a postcondition (guarantee less in output).
- Throw unexpected exceptions.

**The canonical example — Square extends Rectangle:**
```java
class Rectangle { int w, h; void setW(int w){this.w=w;} void setH(int h){this.h=h;} }
class Square extends Rectangle { void setW(int w){this.w=w; this.h=w;} void setH(int h){this.h=h; this.w=h;} }

void test(Rectangle r) { r.setW(5); r.setH(10); assert r.area() == 50; }
test(new Square());  // FAIL: area == 100, not 50
```

Code assuming `setW` and `setH` are independent (natural for Rectangle) breaks when a Square is passed.

**Correct**: Rectangle and Square are parallel classes sharing a `Shape` interface, not inheritance. The geometric IS-A exists, but **behavioural** IS-A does not.
