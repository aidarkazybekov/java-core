## Title
Constructors & this

## Summary
A constructor is a special method with no return type whose name matches the class. It's invoked on `new` and initialises the object. `this` is a reference to the current instance: disambiguates fields from parameters, delegates to another constructor (`this(...)`), passes the current object as an argument. Constructors are not inherited.

## Deep Dive
## What a constructor is

- Name matches the class name.
- **No** return type (not even `void`).
- Invoked exactly once per `new`.
- Not inherited.
- Can be overloaded by parameters.

If you declare no constructor, the compiler generates a **default no-arg** constructor that calls `super()`. **As soon as** you declare any constructor, the default is gone.

> [!gotcha]
> A common mistake: you declared a parameterised constructor and forgot the no-arg one. If a framework (JPA, Jackson) requires a no-arg constructor, code compiles but fails at runtime. That's why JPA entities need an explicit `protected` / `public` no-arg constructor.

## Constructor chaining — `this(...)` and `super(...)`

```java
public Account(String owner, double balance) {
    this.owner = owner;
    this.balance = balance;
}
public Account(String owner) {
    this(owner, 0.0);  // delegation — MUST be the first statement
}
```

**Rule**: `this(...)` and `super(...)` must be the **first statement** of the constructor. You can't use both in one constructor. (Loosened slightly in a Java 22 preview feature.)

## Initialisation order

On **first** class load:
1. Parent static fields and static blocks (declaration order).
2. Child static fields and static blocks.

On **every** `new`:
1. Implicit / explicit `super(...)` — the parent part is constructed recursively.
2. Parent field initialisers and instance-init blocks.
3. Parent constructor body.
4. Child field initialisers and instance-init blocks.
5. Child constructor body.

**Instance initializer block** (`{ ... }` in class body) is copied by the compiler into every constructor — handy when multiple constructors share initialisation.

## Three roles of `this`

1. **Disambiguate shadowing** — `this.name = name` when the parameter name matches the field.
2. **Delegate** — `this(...)` calls another constructor in the same class.
3. **Pass the current object** — `listener.register(this)`, `someMethod(this)`.

`this` is not available in static methods — static belongs to the class, not any instance.

## Leaking `this`

> [!gotcha]
> Passing `this` out of a constructor is a **frequent bug**, especially in concurrent code. Another thread may see the object half-constructed: some fields assigned, some not. The Java Memory Model does not guarantee safe publication of `this` during construction **except** for `final` fields.

Anti-patterns:
- Registering `this` as a listener inside the constructor.
- Starting a background thread from the constructor.
- Passing `this` to constructors of other objects that may leak it.

**Fix**: a static factory method that constructs the object fully, then registers it.

## Copy constructor vs `clone()`

A copy constructor takes an instance of its own class:

```java
public Account(Account other) {
    this(other.owner, other.balance);
}
```

**Advantages over `clone()`**:
- Type-safe — no cast required.
- No `Cloneable` needed.
- Full control over deep vs shallow copying.
- Can be `final` and overridden in a subclass with its own copy constructor.

Josh Bloch recommends avoiding `Cloneable`/`clone()` in favour of copy constructors or static factory methods.

## Records — compact syntax

Since Java 16+ **records** auto-generate the canonical constructor:

```java
public record Point(int x, int y) {
    // Compact constructor — validation before fields are assigned
    public Point {
        if (x < 0 || y < 0) throw new IllegalArgumentException();
    }
}
```

The compact form allows validation without boilerplate. Fields are assigned automatically after its body.

## Code
```java
public class Account {
    private final String owner;
    private double balance;
    private final long createdAt;

    // Primary constructor
    public Account(String owner, double balance) {
        if (balance < 0) throw new IllegalArgumentException("Negative balance");
        this.owner = owner;       // 'this' disambiguates field from param
        this.balance = balance;
        this.createdAt = System.currentTimeMillis();
    }

    // Chained constructor -- delegates to the primary one
    public Account(String owner) {
        this(owner, 0.0);         // must be the first statement
    }

    // Copy constructor -- safer alternative to clone()
    public Account(Account other) {
        this(other.owner, other.balance);
    }

    // Instance initializer block -- runs before every constructor body
    {
        System.out.println("Instance initializer: object being created");
    }

    public void deposit(double amount) {
        this.balance += amount;   // 'this' is optional here but adds clarity
    }

    public Account transferTo(Account target, double amount) {
        this.balance -= amount;
        target.deposit(amount);
        return this;              // returning 'this' enables method chaining
    }

    public static void main(String[] args) {
        Account a = new Account("Alice", 1000);
        Account b = new Account("Bob");
        a.transferTo(b, 200).deposit(50); // chained via 'return this'
        System.out.println(a.balance);    // 850.0
        System.out.println(b.balance);    // 200.0
    }
}
```

## Tip
With multiple constructors, define a single 'primary' one with all parameters and chain others to it via `this(...)`. This ensures all validation runs in one place.

## Spring
### concept
Constructors

### springFeature
Constructor Injection

### explanation
Spring's preferred DI style is **constructor injection**: the container calls the constructor with already-resolved dependencies.

Since Spring 4.3, a single-constructor class doesn't even need `@Autowired`. Constructor injection:
- Guarantees required dependencies are **non-null** (unlike field/setter injection).
- Makes the bean immutable-friendly (fields can be `final`).
- Eases unit testing — every dependency is visible in the signature.
- Avoids cyclic-dependency traps at startup — Spring raises a clear error rather than letting you hit NPE on first call.

## Interview
### [3-2-q0 | junior] What happens if you don't define any constructor in a class?
The compiler generates a **default no-arg** constructor that calls `super()` (the parent's no-arg). If the parent has no accessible no-arg constructor, the code won't compile.

As soon as you declare **any** constructor, the compiler stops generating the default. A common bug: a parameterised constructor without an explicit no-arg breaks JPA/Jackson/serialisers that require one.

### [3-2-q1 | mid] In what order are static blocks, instance initialisers, and constructors executed? What about inheritance?
**On first class load** (once):
1. Parent static initialisers and blocks.
2. Child static initialisers and blocks.

**On every `new`**:
1. `super(...)` — the parent part is constructed recursively.
2. Parent field initialisers + instance-init blocks.
3. Parent constructor body.
4. Child field initialisers + instance-init blocks.
5. Child constructor body.

The compiler **copies** instance-init blocks into every constructor, placed before the body and after `super()`.

### [3-2-q2 | senior] What is the 'leaking this' problem and why is it dangerous in concurrent code?
Leaking `this` means passing the current instance out before the constructor finishes. In a multithreaded environment, another thread may see the object **half-constructed** — some fields assigned, some not.

The Java Memory Model does **not** guarantee safe publication of `this` during construction — except for `final` fields (which have special freeze rules).

**Examples of leakage**:
- Registering `this` as a listener: `this.bus.subscribe(this)`.
- Starting a background thread: `new Thread(this).start()`.
- Passing `this` into inner classes that escape the constructor scope.

**Fix**: a static factory method:

```java
public static MyClass create(...) {
    MyClass obj = new MyClass(...);   // fully constructed
    obj.register(bus);                 // register afterwards
    return obj;
}
```

### [3-2-q3 | mid] Why is a copy constructor preferable to `clone()`?
**`clone()`** is inherited from `Object`, returns `Object`, requires `Cloneable` (a marker interface without methods), and does a shallow copy by default. Forget `super.clone()` and it throws `CloneNotSupportedException`.

**A copy constructor** takes an instance of its own class:

```java
public Account(Account other) {
    this(other.owner, new ArrayList<>(other.history));  // control over deep/shallow
}
```

**Advantages**:
- Type-safe (no cast).
- No `Cloneable`.
- Full control over deep vs shallow.
- Can be `final`.
- Subclasses write their own copy constructor.

Josh Bloch (Effective Java): **avoid `Cloneable` in favour of copy constructors** or static factory methods.
