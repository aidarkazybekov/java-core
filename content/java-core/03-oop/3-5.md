## Title
Interfaces vs Abstract Classes

## Summary
An **interface** is a contract — a set of methods a class must implement. Since Java 8 it supports `default` and `static` methods, since Java 9 — `private`. All fields are `public static final`. An **abstract class** can have state (fields), constructors, and methods with any visibility. A class implements **many** interfaces but extends **one** class.

## Deep Dive
## Interface

- A **contract**: method signatures an implementing class must provide.
- All **fields** are implicitly `public static final` — constants, not state.
- Methods are `public abstract` by default.
- An `abstract` method cannot be `final` or `private`.
- **No constructor**.

**What Java 8+ added:**
- `default` methods — implementation directly in the interface. Enables API evolution without breaking existing implementations (how `stream()` was added to `Collection`).
- `static` methods — utilities that belong with the interface.
- (Java 9) `private` methods — sharing code between defaults.

## Abstract class

- Holds a **partial implementation**.
- Can have **state** (instance fields), constructors, static and instance methods.
- Methods can have any visibility (`private`, `protected`, `public`).
- Can be `abstract` (uninstantiable) or contain abstract methods without fully covering them.
- Implements several interfaces but extends only one class.

## Key differences

| | Interface | Abstract class |
|--|-----------|----------------|
| Multiple inheritance | Yes (implements many) | No (extends one) |
| State (instance fields) | No | Yes |
| Constructor | No | Yes |
| Method modifiers | `public` default | Any |
| When to use | Contract, capability | Shared implementation + state |

## When to choose which

**Interface** — when:
- Defining a capability that cuts across hierarchies: `Comparable`, `Serializable`, `AutoCloseable`, `Iterable`.
- You need easy mocking/stubbing in tests.
- Multiple behaviour inheritance is required.
- In Spring — inject by interface = easier testing, freer to swap implementations.

**Abstract class** — when:
- Subclasses share significant implementation.
- Common state (fields) and constructors are needed.
- Template method pattern — you provide the algorithm skeleton, subclasses fill in pieces.

## Diamond conflict for default methods

If two interfaces provide conflicting defaults, the compiler **forces** an override:

```java
interface A { default String hi() { return "A"; } }
interface B { default String hi() { return "B"; } }

class C implements A, B {
    @Override public String hi() { return A.super.hi(); }  // explicit resolution
}
```

**Priority rules** (when not obvious):
1. **Class wins over interface** — a method from the class hierarchy beats a default.
2. **More specific interface wins** — sub-interface > super-interface.
3. If ambiguous — the class **must** override.

## Functional interfaces and lambdas

A **functional interface** has **exactly one** abstract method (SAM, Single Abstract Method). Defaults and static don't count. The foundation of lambdas:

```java
@FunctionalInterface
interface Loggable {
    String toLogString();
    default void log() { System.out.println(toLogString()); }
}

Loggable l = () -> "hello";  // a lambda — an instance of a functional interface
```

`@FunctionalInterface` is optional but useful: the compiler verifies there's exactly one abstract method — preventing accidental addition of a second one that would break lambda use.

From `java.util.function`: `Function<T,R>`, `Predicate<T>`, `Consumer<T>`, `Supplier<T>`, `UnaryOperator<T>`, `BinaryOperator<T>`, `BiFunction`, `BiPredicate`, and more.

## Interface + Skeletal implementation

> [!tip]
> Effective Java Item 20: declare an **interface** + an **abstract class** implementing a skeleton.

JDK examples: `Collection` + `AbstractCollection`, `List` + `AbstractList`, `Map` + `AbstractMap`.

**Clients** depend on the interface. **Implementors** extend the abstract class and get iterator, equals, hashCode "for free", overriding just a few base methods (`get`, `size` for `AbstractList`).

If a class already extends another and can't extend the skeletal class, it implements the interface directly and internally uses a private inner class delegating to the skeletal (emulating multiple inheritance).

## Code
```java
// Interface with default and static methods
public interface Loggable {
    String toLogString();

    default void log() {
        System.out.println("[LOG] " + toLogString());
    }

    static Loggable of(String message) {
        return () -> message;  // lambda since Loggable is functional
    }
}

// Abstract class with shared state and template method
public abstract class AbstractEntity implements Loggable {
    private final long id;
    private final java.time.Instant createdAt;

    protected AbstractEntity(long id) {
        this.id = id;
        this.createdAt = java.time.Instant.now();
    }

    // Template method — subclasses fill in details
    @Override
    public final String toLogString() {
        return getClass().getSimpleName()
            + "[id=" + id + ", created=" + createdAt + "] " + details();
    }

    protected abstract String details();

    public long getId() { return id; }
}

public class User extends AbstractEntity {
    private final String email;

    public User(long id, String email) {
        super(id);
        this.email = email;
    }

    @Override
    protected String details() {
        return "email=" + email;
    }

    public static void main(String[] args) {
        User u = new User(1, "alice@example.com");
        u.log();                           // default from Loggable + template from AbstractEntity

        Loggable quick = Loggable.of("quick message");
        quick.log();
    }
}
```

## Tip
In interviews, mentioning the 'interface + skeletal implementation' pattern from Effective Java (Item 20) instantly signals deep understanding and lifts your answer above generic 'interfaces are for contracts'.

## Spring
### concept
Interfaces

### springFeature
Spring JDK Dynamic Proxies

### explanation
When a Spring bean implements at least one interface, Spring defaults to **JDK dynamic proxies** for AOP — creating a proxy that implements the same interfaces.

Consequences:
- The proxy is type-compatible only with the **interface**, not the concrete class.
- Injecting a bean as `@Autowired MyServiceImpl service;` (by concrete class) won't match the proxy → `ClassCastException` or `NoSuchBeanDefinitionException`.
- If class-based injection is needed, Spring falls back to **CGLIB** proxies that subclass the target.

Takeaway: **code to interfaces** in Spring — it's both good OO practice and aligns with AOP proxy semantics.

## Interview
### [3-5-q0 | junior] When should you use an interface versus an abstract class?
**Interface** — when you need to define a **capability** or contract for potentially unrelated classes (`Comparable`, `AutoCloseable`, `Serializable`), or to get multiple behaviour inheritance.

**Abstract class** — when subclasses share substantial implementation or common mutable state. The template method pattern is the ideal case.

Because a class can implement many interfaces but extend only one class, interfaces are more flexible for cross-cutting concerns. Modern practice: **start with an interface**, and if you find shared code among implementations, add an abstract skeletal class implementing the interface.

### [3-5-q1 | mid] How does Java resolve conflicts when a class implements two interfaces with the same default method?
If both defaults remain, the compiler **forces** the implementer to override — else a compile error. Inside the override you can delegate to a specific interface: `InterfaceA.super.method()`.

**Resolution rules**:
1. **Class / abstract class wins** — if the method is defined in the class hierarchy (not only in interfaces), it beats any default.
2. **More specific interface wins** — if one interface `extends` another, the default in the sub-interface wins.
3. **Otherwise** — the class must override.

```java
interface A { default String hi() { return "A"; } }
interface B { default String hi() { return "B"; } }

class C implements A, B {
    @Override public String hi() { return B.super.hi(); }  // pick B
}
```

### [3-5-q2 | senior] Explain the 'interface + skeletal implementation' pattern and how the JDK uses it.
The pattern (Effective Java Item 20): a **public interface** + an **abstract class** providing a skeletal implementation.

**Clients** depend on the interface for maximum flexibility. **Implementors** extend the abstract class, get boilerplate for free (iterator, equals, hashCode) and only override 2-3 primitive methods.

**Examples in the JDK**:
- `List` + `AbstractList` — `AbstractList` implements iterator, equals, hashCode, subList. A subclass only needs `get(int)` and `size()`.
- `Set` + `AbstractSet`, `Map` + `AbstractMap`, `Queue` + `AbstractQueue`.
- `Collection` + `AbstractCollection`.

**What if a class already extends another** and can't extend the skeletal? It implements the interface directly + a private inner class delegating to the skeletal — emulating multiple inheritance.

Advantages:
- Contract decoupled from implementation.
- Minimal code duplication.
- Flexibility for non-standard implementers.
