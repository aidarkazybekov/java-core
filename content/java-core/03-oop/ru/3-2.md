## Title
Constructors & this

## Summary
Конструктор — специальный метод без возвращаемого типа с именем класса. Вызывается при создании нового объекта и инициализирует его. `this` — ссылка на текущий экземпляр: устраняет неоднозначность полей и параметров, делегирует другому конструктору (`this(...)`), передаёт текущий объект как аргумент. Конструкторы не наследуются.

## Deep Dive
## Что такое конструктор

- Имя совпадает с именем класса.
- **Нет** возвращаемого типа (даже `void`).
- Вызывается ровно один раз на `new`.
- Не наследуется.
- Может быть перегружен по параметрам.

Если вы не объявили конструктор — компилятор генерирует **default no-arg** конструктор, вызывающий `super()`. **Как только** вы объявили любой конструктор, default исчезает.

> [!gotcha]
> Распространённая ошибка: объявили параметризованный конструктор и забыли про no-arg. Если фреймворк (JPA, Jackson) требует no-arg конструктор — код компилируется, но падает в рантайме. Поэтому JPA-сущностям нужен явный `protected` / `public` no-arg конструктор.

## Constructor chaining — `this(...)` и `super(...)`

```java
public Account(String owner, double balance) {
    this.owner = owner;
    this.balance = balance;
}
public Account(String owner) {
    this(owner, 0.0);  // делегирование — ДОЛЖНО быть первым оператором
}
```

**Правило**: `this(...)` и `super(...)` должны быть **первым оператором** конструктора. Нельзя вызвать оба в одном конструкторе. (Ограничение немного ослаблено preview-фичей в Java 22.)

## Порядок инициализации

При **первой** загрузке класса:
1. Static-поля и static-блоки родителя (в порядке объявления).
2. Static-поля и static-блоки потомка.

При **каждом** `new`:
1. Неявный / явный `super(...)` — рекурсивно создаётся parent-часть.
2. Инициализаторы полей и instance-init блоки родителя.
3. Тело конструктора родителя.
4. Инициализаторы полей и instance-init блоки потомка.
5. Тело конструктора потомка.

**Instance initializer block** (`{ ... }` в теле класса) копируется компилятором в каждый конструктор — удобно, когда несколько конструкторов делят общую инициализацию.

## Три роли `this`

1. **Устранение тени** — `this.name = name` когда имя поля совпадает с параметром.
2. **Делегирование** — `this(...)` вызов другого конструктора того же класса.
3. **Передача текущего объекта** — `listener.register(this)`, `someMethod(this)`.

`this` недоступен в static-методах — static принадлежит классу, а не экземпляру.

## Leaking `this`

> [!gotcha]
> Передача `this` из конструктора наружу — **частая ошибка**, особенно в concurrent-коде. Другой поток может увидеть объект в полу-сконструированном состоянии: часть полей уже присвоена, часть — ещё нет. Java Memory Model не гарантирует безопасную публикацию `this` во время конструирования, **кроме** `final` полей.

Анти-паттерны:
- Регистрация `this` как listener внутри конструктора.
- Запуск фонового потока из конструктора.
- Передача `this` в конструкторы других объектов, которые могут её «утечь».

**Решение**: статический factory-метод, который сначала полностью сконструирует объект, а потом регистрирует его.

## Copy constructor vs `clone()`

Copy-конструктор принимает экземпляр своего же класса:

```java
public Account(Account other) {
    this(other.owner, other.balance);
}
```

**Преимущества перед `clone()`**:
- Типобезопасен — не нужен каст после `clone()`.
- Не требует `Cloneable`.
- Полный контроль над deep vs shallow копированием.
- Может быть `final` и переопределён в subclass через свой copy-конструктор.

Josh Bloch рекомендует избегать `Cloneable`/`clone()` в пользу copy-конструкторов или static factory methods.

## Records — компактный синтаксис

С Java 16+ **рекорды** генерируют canonical-конструктор автоматически:

```java
public record Point(int x, int y) {
    // Compact constructor — для валидации до присвоения полей
    public Point {
        if (x < 0 || y < 0) throw new IllegalArgumentException();
    }
}
```

Компактный конструктор позволяет валидацию без boilerplate. Поля присваиваются автоматически после его тела.

## Code
```java
public class Account {
    private final String owner;
    private double balance;
    private final long createdAt;

    // Primary constructor
    public Account(String owner, double balance) {
        if (balance < 0) throw new IllegalArgumentException("Negative balance");
        this.owner = owner;       // 'this' disambiguates field from param
        this.balance = balance;
        this.createdAt = System.currentTimeMillis();
    }

    // Chained constructor -- delegates to the primary one
    public Account(String owner) {
        this(owner, 0.0);         // must be the first statement
    }

    // Copy constructor -- safer alternative to clone()
    public Account(Account other) {
        this(other.owner, other.balance);
    }

    // Instance initializer block -- runs before every constructor body
    {
        System.out.println("Instance initializer: object being created");
    }

    public void deposit(double amount) {
        this.balance += amount;   // 'this' is optional here but adds clarity
    }

    public Account transferTo(Account target, double amount) {
        this.balance -= amount;
        target.deposit(amount);
        return this;              // returning 'this' enables method chaining
    }

    public static void main(String[] args) {
        Account a = new Account("Alice", 1000);
        Account b = new Account("Bob");
        a.transferTo(b, 200).deposit(50); // chained via 'return this'
        System.out.println(a.balance);    // 850.0
        System.out.println(b.balance);    // 200.0
    }
}
```

## Tip
Если у класса несколько конструкторов — заведите один «первичный» с полным набором параметров и делегируйте к нему через `this(...)`. Это гарантирует, что вся валидация выполнится в одном месте.

## Spring
### concept
Constructors

### springFeature
Constructor Injection

### explanation
Предпочтительный стиль DI в Spring — **constructor injection**: контейнер вызывает конструктор с уже разрешёнными зависимостями.

С Spring 4.3+ класс с одним конструктором даже не требует `@Autowired`. Constructor injection:
- Гарантирует, что обязательные зависимости — **non-null** (в отличие от field/setter injection).
- Делает бин immutable-friendly (поля можно объявить `final`).
- Облегчает unit-тестирование — видны все зависимости по сигнатуре.
- Избегает циклических зависимостей на этапе запуска — Spring бросит понятную ошибку, а не `NullPointerException` при первом вызове.

## Interview
### [3-2-q0 | junior] Что произойдёт, если не объявить ни одного конструктора в классе?
Компилятор сгенерирует **default no-arg** конструктор, который вызывает `super()` (no-arg конструктор родителя). Если у родителя нет доступного no-arg конструктора — код не скомпилируется.

Как только вы объявите **любой** свой конструктор, компилятор перестаёт генерировать default. Частая ошибка: параметризованный конструктор без явного no-arg ломает JPA/Jackson/сериализаторы, которым нужен no-arg.

### [3-2-q1 | mid] Каков порядок инициализации static-блоков, instance-инициализаторов и конструкторов с учётом наследования?
**При первой загрузке класса** (по одному разу):
1. Static-инициализаторы и блоки родителя.
2. Static-инициализаторы и блоки потомка.

**При каждом `new`**:
1. `super(...)` — рекурсивно создаётся parent-часть.
2. Инициализаторы полей + instance-init блоки родителя.
3. Тело конструктора родителя.
4. Инициализаторы полей + instance-init блоки потомка.
5. Тело конструктора потомка.

Компилятор **копирует** instance-init блоки в каждый конструктор перед его телом, после `super()`.

### [3-2-q2 | senior] Что такое «leaking this» и почему это опасно в многопоточном коде?
Leaking `this` — передача текущего экземпляра наружу до того, как конструктор завершился. В многопоточной среде другой поток может увидеть объект в **полу-сконструированном** состоянии: часть полей уже присвоена, часть — ещё нет.

Java Memory Model **не** гарантирует безопасную публикацию `this` во время конструирования — кроме `final` полей (для них есть специальные правила freeze).

**Примеры утечки**:
- Регистрация `this` как listener: `this.bus.subscribe(this)`.
- Запуск фонового потока: `new Thread(this).start()`.
- Передача `this` во внутренние классы, которые выходят за пределы конструктора.

**Решение**: статический factory-метод:

```java
public static MyClass create(...) {
    MyClass obj = new MyClass(...);   // полностью сконструирован
    obj.register(bus);                 // потом регистрируется
    return obj;
}
```

### [3-2-q3 | mid] Почему copy-конструктор предпочтительнее `clone()`?
**`clone()`** унаследован от `Object`, возвращает `Object`, требует implements `Cloneable` (маркер-интерфейс без методов), и делает shallow-копию по умолчанию. Если вы забудете вызвать `super.clone()` — бросит `CloneNotSupportedException`.

**Copy-конструктор** принимает экземпляр своего же класса:

```java
public Account(Account other) {
    this(other.owner, new ArrayList<>(other.history));  // контроль deep/shallow
}
```

**Преимущества**:
- Типобезопасно (никакого cast).
- Не нужен `Cloneable`.
- Полный контроль над deep vs shallow копированием.
- Может быть `final`.
- Subclass создаёт свой copy-конструктор.

Рекомендация Josh Bloch (Effective Java): **избегать `Cloneable` в пользу copy-конструкторов** или static factory-методов.
