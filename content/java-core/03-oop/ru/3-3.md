## Title
Inheritance & super

## Summary
Наследование — IS-A отношение: `Dog extends Animal`. Java поддерживает **одиночное** наследование классов и **множественную** реализацию интерфейсов. `super(...)` вызывает конструктор родителя; `super.method()` — переопределённый метод родителя. Диспетчеризация идёт через vtable по runtime-типу объекта — основа runtime-полиморфизма.

## Deep Dive
## Отношения между классами

- **Ассоциация (has-a)** — один класс использует другой.
- **Агрегация** (слабая has-a) — обе стороны существуют независимо. Книга может быть перенесена между библиотеками.
- **Композиция** (сильная has-a) — один класс управляет жизненным циклом другого. Страницы существуют только с книгой.
- **Наследование (is-a)** — `Dog` является `Animal`.
- **Реализация** — класс implements интерфейс.
- **Зависимость** — использование класса в сигнатуре метода / параметре.

## Что наследуется, что нет

Наследуются:
- Public, protected, package-private (если тот же пакет) поля и методы.
- Private поля **существуют** в памяти объекта, но недоступны по имени в потомке.

Не наследуются:
- Конструкторы (нужно вызывать через `super(...)`).
- Private методы.
- Static методы наследуются «логически», но могут быть **скрыты** (hidden), не переопределены.

## Virtual method dispatch

Для instance-методов JVM использует **vtable**: при вызове `ref.method()` смотрим **runtime-тип** объекта, а не объявленный тип ссылки. Это и есть runtime-полиморфизм.

```java
Animal a = new Dog("Rex");
a.speak();  // вызывается Dog.speak(), не Animal.speak()
```

Для static-методов и полей действует **compile-time binding** — используется объявленный тип ссылки.

## Overriding vs hiding

| | Method | Static method / Field |
|--|--------|------------------------|
| Механизм | Override (dynamic binding) | Hide (static binding) |
| По какому типу диспетчеризуется | Runtime-тип объекта | Compile-time тип ссылки |
| `@Override` | Да | Нет |

```java
class Parent { int x = 1; static String name() { return "Parent"; } }
class Child extends Parent { int x = 2; static String name() { return "Child"; } }

Parent p = new Child();
p.x;         // 1 (поля — hiding, по типу ссылки)
p.name();    // "Parent" (static — hiding, по типу ссылки)
```

## Правила переопределения

- **Та же сигнатура** (имя + параметры).
- **Covariant return type** (Java 5+) — можно возвращать более узкий тип.
- Модификатор доступа — **равный или шире**.
- Checked exceptions — **такие же или уже**. RuntimeException — свободно.

> [!tip]
> **Всегда ставьте `@Override`.** Без аннотации опечатка в имени создаст новый метод вместо override — компилятор промолчит, баг всплывёт в рантайме.

## `super`

Две роли:

1. **`super(...)`** — вызов конструктора родителя. Должен быть первым оператором. Если не указан, компилятор вставляет `super()` — и это упадёт, если у родителя нет no-arg.
2. **`super.method()`** — вызов переопределённого метода родителя. Полезно, чтобы **дополнить** поведение:
   ```java
   @Override public String speak() {
       return name + " barks! (" + super.speak() + ")";
   }
   ```

**Нельзя** `super.super.method()` — Java даёт подняться только на один уровень.

## Почему нет множественного наследования классов

**Diamond problem** — если два родителя определяют один метод по-разному, какую реализацию взять? C++ решает через явное разрешение, Java избегает проблемы, запретив множественное наследование классов.

**Default-методы интерфейсов (Java 8+)** возвращают контролируемую форму множественного наследования **поведения**. Если два интерфейса дают конфликтующие default-реализации, компилятор **требует** от реализующего класса override этого метода — неоднозначности нет.

```java
interface A { default void m() { ... } }
interface B { default void m() { ... } }
class C implements A, B {
    @Override public void m() { A.super.m(); }  // явное разрешение
}
```

## `final` и `sealed`

- **`final class`** — нельзя наследовать (`String`, все обёртки примитивов).
- **`final method`** — нельзя переопределить.
- **`sealed class / interface`** (Java 17+) — золотая середина: перечислите `permits`-список разрешённых наследников:

```java
sealed interface Shape permits Circle, Square, Triangle {}
```

Субклассы должны быть `final`, `sealed` (со своим permits) или `non-sealed`. Компилятор знает закрытый набор и делает **exhaustive pattern matching в switch без default**.

## Fragile base class — почему композиция > наследование

> [!production]
> Изменение базового класса может **скрытно сломать** потомков, которые на него полагались. Классический пример (Bloch, Effective Java): `HashSet` → `InstrumentedHashSet` с переопределённым `add` и `addAll`. `addAll` в `HashSet` внутри вызывает `add` → `add` вызывается **дважды** через override, счётчик удваивается. Поведение не документировано как API — это implementation detail.

Правило: «**предпочитай композицию наследованию**» (Bloch). Оберните нужное поведение в поле, а не наследуйте класс. Наследование — только когда вы контролируете базовый класс и IS-A отношение действительно выполняется.

## Code
```java
public class Animal {
    private final String species;

    public Animal(String species) {
        this.species = species;
    }

    public String speak() {
        return species + " makes a sound";
    }

    // final prevents subclasses from overriding
    public final String getSpecies() {
        return species;
    }
}

public class Dog extends Animal {

    private final String name;

    public Dog(String name) {
        super("Canine");  // must be first statement
        this.name = name;
    }

    @Override
    public String speak() {
        // augment parent behavior
        return name + " barks! (" + super.speak() + ")";
    }

    public static void main(String[] args) {
        Animal a = new Dog("Rex");   // polymorphic reference
        System.out.println(a.speak());
        // "Rex barks! (Canine makes a sound)"

        // a.name  -- won't compile; 'name' is not in Animal

        System.out.println(a.getSpecies()); // "Canine" (final method)
    }
}
```

## Tip
Всегда ставьте `@Override`. Без неё опечатка в имени создаст новый метод вместо переопределения — компилятор промолчит, баг найдёте только в рантайме.

## Spring
### concept
Inheritance & super

### springFeature
Spring CGLIB Proxies

### explanation
Spring AOP создаёт прокси, **наследуя** класс вашего бина в рантайме через CGLIB (или использует JDK dynamic proxies, если бин реализует интерфейс).

Последствия:
- Класс **не может быть `final`** — CGLIB не сможет его унаследовать. Бросится ошибка при запуске.
- Методы не могут быть `final` или `private` — прокси не сможет их переопределить для внедрения advice (`@Transactional`, `@Cacheable`, `@Async`, `@Secured`).
- **Self-invocation пропускает прокси**. Вызов `this.method()` изнутри того же класса идёт не через прокси, а напрямую — advice не срабатывает. Решение: вынести метод в другой бин или использовать `AopContext.currentProxy()`.

Понимание механики наследования объясняет эти ограничения.

## Interview
### [3-3-q0 | junior] В чём разница между method overriding и method hiding?
- **Overriding** — для **instance-методов**. JVM диспетчеризует по **runtime-типу** объекта (dynamic binding). Это и есть runtime-полиморфизм.
- **Hiding** — для **static-методов и полей**. Используется **compile-time тип** ссылки (static binding). Если потомок объявит static-метод с той же сигнатурой, он **скрывает** родительский, а не переопределяет.

```java
class Parent { static String tag() { return "P"; } }
class Child extends Parent { static String tag() { return "C"; } }

Parent p = new Child();
p.tag();  // "P" — static binds по типу ссылки
```

Поля аналогично: `p.field` — поле родителя, даже если объект — `Child`.

### [3-3-q1 | mid] Почему Java не поддерживает множественное наследование классов, и как default-методы интерфейсов меняют картину?
**Diamond problem**: если два родителя определяют один метод по-разному, какую реализацию наследовать? Java избегает двусмысленности, запретив множественное наследование классов.

**Default-методы в интерфейсах (Java 8+)** дают контролируемую форму множественного наследования **поведения** (но не состояния — полей у интерфейсов нет). Если два интерфейса дают конфликтующие default, компилятор **требует** override в реализующем классе — двусмысленность невозможна:

```java
interface A { default void m() {} }
interface B { default void m() {} }
class C implements A, B {
    @Override public void m() { A.super.m(); }  // явное разрешение
}
```

Это позволило расширить `Collection` (например, `stream()`) без ломки существующих реализаций.

### [3-3-q2 | senior] Объясните sealed-классы, их связь с pattern matching, и как они улучшают традиционные иерархии.
**Sealed-классы (Java 17)** ограничивают, кто может их наследовать, через `permits`-список. Набор субтипов **закрыт и известен в compile time**.

```java
sealed interface Shape permits Circle, Square, Triangle {}
record Circle(double r) implements Shape {}
record Square(double side) implements Shape {}
record Triangle(double base, double h) implements Shape {}
```

Субклассы обязаны быть `final`, `sealed` (со своим `permits`) или `non-sealed`.

**Pattern matching в `switch`** использует закрытость — компилятор проверяет exhaustiveness без `default`:

```java
double area = switch (s) {
    case Circle c   -> Math.PI * c.r() * c.r();
    case Square sq  -> sq.side() * sq.side();
    case Triangle t -> t.base() * t.h() / 2;
};
```

Добавление нового permitted-варианта — **compile error во всех switch** над этим типом. Пропустить case невозможно.

Sealed-типы решают напряжение между **открытостью** (любой может наследовать) и **контролем** (автор хочет закрытый набор вариантов) — это Java-аналог algebraic data types функциональных языков. Отлично сочетается с рекордами.

### [3-3-q3 | mid] Что такое fragile base class и почему говорят «предпочитай композицию наследованию»?
**Fragile base class** — изменение базового класса может скрытно сломать потомков, которые полагались на **implementation detail** базового, а не на документированный API.

Классика Bloch'а (Effective Java):

```java
class InstrumentedHashSet<E> extends HashSet<E> {
    int added = 0;
    @Override public boolean add(E e) { added++; return super.add(e); }
    @Override public boolean addAll(Collection<? extends E> c) {
        added += c.size(); return super.addAll(c);
    }
}
```

Внутри `HashSet.addAll` вызывает `add` для каждого элемента. Наш override `add` уже инкрементит → счётчик удваивается. Это undocumented implementation detail, на которое полагаться нельзя.

**Композиция** делает зависимости явными. Вместо extends — оборачиваем:

```java
class InstrumentedSet<E> implements Set<E> {
    private final Set<E> delegate;
    int added = 0;
    public boolean add(E e) { added++; return delegate.add(e); }
    public boolean addAll(Collection<? extends E> c) {
        int n = 0; for (E e : c) if (add(e)) n++; return n > 0;
    }
    // ... другие методы делегируют delegate
}
```

Наследование применимо только тогда, когда вы контролируете базовый класс и IS-A отношение по-настоящему выполняется.
