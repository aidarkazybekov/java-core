## Title
Access Modifiers & static

## Summary
Четыре модификатора доступа: `private` (класс), default/package-private (пакет), `protected` (наследники + пакет), `public` (везде). `static` — член принадлежит **классу**, а не экземпляру; живёт в Metaspace, инициализируется один раз через `<clinit>`. Правило: делайте всё максимально приватным и расширяйте только по обоснованию.

## Deep Dive
## Четыре уровня доступа

| Модификатор | Сам класс | Пакет | Наследники (др. пакет) | Везде |
|-------------|-----------|-------|------------------------|-------|
| `private` | ✓ | | | |
| default (нет ключевого слова) | ✓ | ✓ | | |
| `protected` | ✓ | ✓ | ✓ | |
| `public` | ✓ | ✓ | ✓ | ✓ |

**Top-level классы** могут быть только `public` или default. Inner/nested классы — все четыре.

> [!gotcha]
> `protected` член доступен потомку в другом пакете **только через наследование** — не через sibling-инстанс. `parent.protectedField` из метода потомка, когда `parent` — ссылка на другой объект-родитель, **не** компилируется (если мы в другом пакете).

**Правило инкапсуляции**: используйте **самый ограничивающий** модификатор, совместимый с задачей. Поля — почти всегда `private`, доступ через методы (или никак — для immutable).

## Java Module System (JPMS, Java 9+)

Даже `public` класс недоступен вне модуля, если модуль не **экспортирует** пакет:

```java
// module-info.java
module com.example.core {
    exports com.example.core.api;      // доступно всем
    exports com.example.core.internal to com.example.trusted;  // qualified export
}
```

Это добавляет пятый уровень контроля — **модульная видимость**.

## static

**Семантика**: член принадлежит **классу**, а не экземпляру.

- Статическое поле — одно на классhash: хранится в **Metaspace** (метаданные класса), не на куче per-object.
- Статический метод — нет `this`, нет `super`, не может ссылаться на instance-поля напрямую.
- **Статический блок** `static { ... }` — выполняется один раз при инициализации класса (в `<clinit>`).
- Static-члены инициализируются **в текстовом порядке** объявления.

## Static vs instance

| | Static | Instance |
|--|--------|----------|
| Где хранится | Metaspace | Heap (в составе объекта) |
| Когда инициализируется | Один раз при загрузке класса | На каждом `new` |
| Access | `Class.member` или `instance.member` (IDE warn) | Только через `instance` |
| Полиморфизм | Нет — static binding (hiding, не overriding) | Да — virtual dispatch |
| `this` / `super` | Нет | Да |

## Static factory methods

Паттерн из Effective Java: приватный конструктор + публичный static-метод.

```java
public class Duration {
    private Duration(long ns) { ... }
    public static Duration ofSeconds(long s)    { return new Duration(s * 1_000_000_000L); }
    public static Duration ofMillis(long ms)    { return new Duration(ms * 1_000_000L); }
    public static Duration ofNanos(long ns)     { return new Duration(ns); }
}
```

**Преимущества перед конструкторами**:
- Имена описательные: `Duration.ofSeconds(5)` vs `new Duration(5, SECONDS)`.
- Можно **кэшировать** (см. `Integer.valueOf`, `Boolean.valueOf`, `Optional.empty()`).
- Можно возвращать подтип.
- Могут быть `static` в разных классах — фабрика для иерархии (`List.of(...)`, `Set.of(...)`).

JDK: `Integer.valueOf`, `Optional.of` / `empty`, `List.of`, `Collectors.toList`, `EnumSet.of`.

## Static initialization — порядок и ловушки

Static-поля и блоки инициализируются в **текстовом порядке** объявления. Ссылка на поле, объявленное ниже, увидит дефолт:

```java
class Oops {
    static int a = b + 1;  // b = 0 (default) — a = 1
    static int b = 10;
}
```

> [!gotcha]
> **Циклические static-зависимости** между классами могут привести к тому, что класс используется до завершения инициализации → `NullPointerException` или `ExceptionInInitializerError`. Если `<clinit>` бросит исключение — класс помечается **permanently unusable**, любой последующий доступ бросит `NoClassDefFoundError`.

## Initialization-on-Demand Holder

Лучший паттерн для thread-safe lazy singleton без синхронизации:

```java
public class Singleton {
    private Singleton() {}
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

Работает благодаря гарантиям JLS §12.4.1: класс **не инициализируется** до первого активного использования. Первый вызов `getInstance()` триггерит загрузку `Holder` — `<clinit>` запускается. JVM гарантирует, что `<clinit>` выполняется под **class init lock** — thread-safe.

**Лучше, чем double-checked locking** (требует `volatile` + тонкая работа с JMM). Проще, чем eager singleton (`public static final Singleton INSTANCE = new Singleton();`) — действительно ленив.

**Enum singleton** (Bloch) — ещё проще и защищён от reflection/serialization атак:
```java
public enum Singleton { INSTANCE; public void doWork() { ... } }
```

## Code
```java
public class Counter {
    private static int totalCount = 0;
    private int localCount = 0;

    static {
        System.out.println("Counter class loaded, totalCount initialized");
    }

    public void increment() {
        localCount++;
        totalCount++;
    }

    public static int getTotalCount() {
        return totalCount;
    }

    private Counter() {}

    public static Counter create() {
        return new Counter();
    }

    public static void main(String[] args) {
        Counter a = Counter.create();
        Counter b = Counter.create();

        a.increment();
        a.increment();
        b.increment();

        System.out.println(a.localCount);         // 2
        System.out.println(b.localCount);         // 1
        System.out.println(Counter.getTotalCount()); // 3 (shared)
    }
}

// --- Initialization-on-Demand Holder idiom ---
public class Singleton {
    private Singleton() {}

    // Inner class is not loaded until getInstance() is called
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // thread-safe, lazy, no synchronization
    }
}
```

## Tip
Когда интервьюер спрашивает про static, упомяните, что static-методы нельзя легко мокать в классическом Mockito (до 3.4). Это показывает, что вы понимаете тестовые последствия дизайн-решений, а не только семантику языка.

### explanation
Spring singleton-бины — **не** Java `static` singletons. Один инстанс **на ApplicationContext**, а не на classloader. Разные контексты → разные инстансы. Это важно в тестах со `@SpringBootTest` — каждый тест может иметь свой контекст и, значит, свой singleton.

`@Bean`-методы в `@Configuration`-классах — **instance-методы** (CGLIB-проксируются), **не** static. Это осознанное ограничение: static-метод нельзя перехватить для проверки scope и inter-bean-зависимостей. Если сделать `@Bean`-метод `static`, Spring теряет возможность возвращать одну и ту же singleton-ссылку при внутренних вызовах — появляются дубликаты бина.

## Interview
### [3-6-q0 | junior] Какие четыре модификатора доступа в Java и их области видимости?
- **`private`** — только свой класс. Даже потомки не имеют доступа.
- **default** (без ключевого слова) — свой пакет (package-private).
- **`protected`** — свой пакет + потомки (в других пакетах, но только через наследование).
- **`public`** — везде.

**Top-level класс** — только `public` или default. **Nested/inner** — все четыре. С JPMS даже `public` класс невидим вне модуля, если пакет не экспортирован.

Общее правило: **максимум ограничений**. Поля — почти всегда `private`.

### [3-6-q1 | mid] Можно ли переопределить static-метод? Что произойдёт при объявлении такого же метода в подклассе?
Нет, static-методы **не переопределяются** — у них compile-time binding по типу ссылки, не по runtime-типу объекта. Если потомок объявит static-метод с той же сигнатурой — он **скрывает** (hides) метод родителя, не переопределяет.

```java
class Parent { static String tag() { return "P"; } }
class Child extends Parent { static String tag() { return "C"; } }

Parent p = new Child();
p.tag();       // "P" — по типу ссылки, не по объекту
Child.tag();   // "C"
```

Полиморфизм не применим. Поэтому static **нельзя** использовать там, где нужна поздняя связка поведения — это корень многих проблем с мокированием: классический Mockito (до 3.4) не мог мокать static-методы именно потому, что они решаются compile-time.

### [3-6-q2 | senior] Объясните Initialization-on-Demand Holder и почему он предпочтительнее double-checked locking для singleton.
Паттерн использует **приватный static nested-класс** с `static final` инстансом:

```java
public class Singleton {
    private Singleton() {}
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```

По JLS §12.4.1, класс **не инициализируется** до первого активного использования. Первый `getInstance()` триггерит загрузку `Holder`, `<clinit>` создаёт инстанс. JVM гарантирует, что `<clinit>` выполняется под class init lock — **thread-safe by construction**.

**Преимущества перед DCL**:
- Нет необходимости в `volatile` и синхронизации на call-site.
- Невозможно неправильно написать — ошибки в DCL (особенно до Java 5 с ослабленной JMM) были легендарными.
- Ленивый: `INSTANCE` создаётся только при первом вызове `getInstance()`.

**Ещё проще — enum singleton** (Bloch, Effective Java):
```java
public enum Singleton { INSTANCE; public void doWork() { ... } }
```
Thread-safe, serializable правильно, защищён от reflection-атак.
