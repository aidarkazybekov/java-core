## Title
Interfaces vs Abstract Classes

## Summary
**Интерфейс** — контракт: набор методов, которые класс обязан реализовать. С Java 8 поддерживает `default` и `static` методы, с Java 9 — `private`. Все поля — `public static final`. **Абстрактный класс** может иметь состояние (поля), конструкторы и методы любой видимости. Класс реализует **много** интерфейсов, но наследует **один** класс.

## Deep Dive
## Интерфейс

- **Контракт**: сигнатуры методов, которые реализующий класс обязан предоставить.
- Все **поля** неявно `public static final` — это константы, не состояние.
- **Методы по умолчанию** `public abstract`.
- `abstract`-метод не может быть `final` или `private`.
- **Нет конструктора**.

**Что добавили Java 8+:**
- `default` методы — реализация прямо в интерфейсе. Для эволюции API без ломки существующих реализаций (так добавили `stream()` в `Collection`).
- `static` методы — утилиты, относящиеся к интерфейсу.
- (Java 9) `private` методы — шаринг кода между default-ами.

## Абстрактный класс

- Содержит **частичную реализацию**.
- Может иметь **state** (instance fields), конструкторы, static и instance методы.
- Методы могут быть любой видимости (`private`, `protected`, `public`).
- Может быть `abstract` (нельзя инстанцировать) или иметь абстрактные методы без полного покрытия.
- Реализует несколько интерфейсов, но наследуется только от одного класса.

## Ключевые отличия

| | Interface | Abstract class |
|--|-----------|----------------|
| Множественное наследование | Да (implements many) | Нет (extends one) |
| Состояние (instance fields) | Нет | Да |
| Конструктор | Нет | Да |
| Модификаторы методов | `public` по умолчанию | Любые |
| Когда применять | Контракт, capability | Общая реализация + state |

## Когда что выбирать

**Интерфейс** — когда:
- Определяете capability, пересекающую иерархии: `Comparable`, `Serializable`, `AutoCloseable`, `Iterable`.
- Хотите mock/stub для тестов.
- Нужно множественное наследование поведения.
- В Spring — инъекция по интерфейсу = легче тестировать, свободнее заменять реализации.

**Абстрактный класс** — когда:
- Подклассы делят значимую реализацию.
- Нужно общее состояние (поля) и конструкторы для инициализации.
- Template method pattern — вы задаёте скелет алгоритма, потомки заполняют частности.

## Diamond-конфликт default-методов

Если два интерфейса дают конфликтующие default-методы, компилятор **требует** override:

```java
interface A { default String hi() { return "A"; } }
interface B { default String hi() { return "B"; } }

class C implements A, B {
    @Override public String hi() { return A.super.hi(); }  // явное разрешение
}
```

**Правила приоритета** (если неочевидно):
1. **Класс побеждает интерфейс** — метод из class-иерархии приоритетнее default-а.
2. **Более специфичный интерфейс побеждает менее специфичный** — sub-interface > super-interface.
3. Если неоднозначно — класс **обязан** переопределить.

## Функциональные интерфейсы и лямбды

**Functional interface** — интерфейс с **ровно одним** abstract-методом (SAM, Single Abstract Method). Default и static не считаются. Основа для лямбд:

```java
@FunctionalInterface
interface Loggable {
    String toLogString();
    default void log() { System.out.println(toLogString()); }
}

Loggable l = () -> "hello";  // лямбда — инстанс functional interface
```

Аннотация `@FunctionalInterface` не обязательна, но полезна: компилятор проверит, что у интерфейса ровно один abstract-метод — предотвращает случайное добавление второго, ломающего лямбды.

Из `java.util.function`: `Function<T,R>`, `Predicate<T>`, `Consumer<T>`, `Supplier<T>`, `UnaryOperator<T>`, `BinaryOperator<T>`, `BiFunction`, `BiPredicate` и др.

## Interface + Skeletal implementation

> [!tip]
> Паттерн из Effective Java (Item 20): объявите **интерфейс** + **абстрактный класс**, реализующий скелет.

Примеры в JDK: `Collection` + `AbstractCollection`, `List` + `AbstractList`, `Map` + `AbstractMap`.

**Клиенты** зависят от интерфейса. **Реализаторы** наследуют абстрактный класс — и получают iterator, equals, hashCode «из коробки», переопределяя только несколько базовых методов (`get`, `size` для `AbstractList`).

Если класс уже наследует другой класс и не может extend skeletal — реализует интерфейс напрямую, а внутри использует private inner class, делегирующий на skeletal (эмуляция множественного наследования).

## Code
```java
// Interface with default and static methods
public interface Loggable {
    String toLogString();

    default void log() {
        System.out.println("[LOG] " + toLogString());
    }

    static Loggable of(String message) {
        return () -> message;  // lambda since Loggable is functional
    }
}

// Abstract class with shared state and template method
public abstract class AbstractEntity implements Loggable {
    private final long id;
    private final java.time.Instant createdAt;

    protected AbstractEntity(long id) {
        this.id = id;
        this.createdAt = java.time.Instant.now();
    }

    // Template method — subclasses fill in details
    @Override
    public final String toLogString() {
        return getClass().getSimpleName()
            + "[id=" + id + ", created=" + createdAt + "] " + details();
    }

    protected abstract String details();

    public long getId() { return id; }
}

public class User extends AbstractEntity {
    private final String email;

    public User(long id, String email) {
        super(id);
        this.email = email;
    }

    @Override
    protected String details() {
        return "email=" + email;
    }

    public static void main(String[] args) {
        User u = new User(1, "alice@example.com");
        u.log();                           // default from Loggable + template from AbstractEntity

        Loggable quick = Loggable.of("quick message");
        quick.log();
    }
}
```

## Tip
На интервью упоминание паттерна «interface + skeletal implementation» из Effective Java (Item 20) сразу сигнализирует о глубоком понимании и поднимает ответ над общим «интерфейсы — для контрактов».

## Spring
### concept
Interfaces

### springFeature
Spring JDK Dynamic Proxies

### explanation
Когда Spring-бин реализует хотя бы один интерфейс, Spring по умолчанию использует **JDK dynamic proxies** для AOP — создаёт прокси, реализующий те же интерфейсы.

Последствия:
- Прокси совместим по типу только с **интерфейсом**, не с конкретным классом.
- Инъекция бина как `@Autowired MyServiceImpl service;` (по конкретному классу) не совпадёт с прокси → `ClassCastException` или `NoSuchBeanDefinitionException`.
- Если инъекция нужна по классу — Spring падает обратно на **CGLIB**, который наследует класс.

Вывод: **программируйте против интерфейсов** в Spring — это и хорошая практика ООП, и выравнивает поведение с AOP-прокси.

## Interview
### [3-5-q0 | junior] Когда использовать интерфейс, а когда абстрактный класс?
**Интерфейс** — когда нужно определить **capability** или контракт для потенциально несвязанных классов (`Comparable`, `AutoCloseable`, `Serializable`) или получить множественное наследование поведения.

**Абстрактный класс** — когда подклассы делят значительную реализацию или общее изменяемое состояние. Template method pattern — идеальный случай.

Так как класс может implements много интерфейсов, но extends только один класс — интерфейсы гибче для cross-cutting concerns. Современная практика: **начните с интерфейса**, и если найдёте общий код у реализаторов — добавьте абстрактный skeletal-класс, реализующий интерфейс.

### [3-5-q1 | mid] Как Java разрешает конфликт, когда класс реализует два интерфейса с одинаковым default-методом?
Если оба default-метода остаются — компилятор **заставляет** реализатор переопределить метод, иначе ошибка. Внутри override можно делегировать конкретному интерфейсу: `InterfaceA.super.method()`.

**Правила разрешения**:
1. **Класс / абстрактный класс побеждает** — если метод определён в class-иерархии (не только в интерфейсах), он выигрывает у default.
2. **Более специфичный интерфейс побеждает** — если один интерфейс `extends` другой, default в под-интерфейсе выигрывает.
3. **Иначе** — класс обязан переопределить.

```java
interface A { default String hi() { return "A"; } }
interface B { default String hi() { return "B"; } }

class C implements A, B {
    @Override public String hi() { return B.super.hi(); }  // выбираем B
}
```

### [3-5-q2 | senior] Объясните паттерн «interface + skeletal implementation» и как JDK его использует.
Паттерн (Effective Java, Item 20): **публичный интерфейс** + **абстрактный класс**, реализующий скелет.

**Клиенты** зависят от интерфейса для максимальной гибкости. **Реализаторы** наследуют абстрактный класс, получают boilerplate даром (итератор, equals, hashCode) и переопределяют только 2-3 базовых метода.

**Примеры в JDK**:
- `List` + `AbstractList` — `AbstractList` реализует iterator, equals, hashCode, subList. Потомку достаточно `get(int)` и `size()`.
- `Set` + `AbstractSet`, `Map` + `AbstractMap`, `Queue` + `AbstractQueue`.
- `Collection` + `AbstractCollection`.

**Что если класс уже наследует другой класс** и не может extend skeletal? Реализует интерфейс напрямую + private inner class, делегирующий в skeletal — эмуляция множественного наследования.

Преимущества:
- Контракт отделён от реализации.
- Минимум дублирования кода.
- Гибкость для нестандартных реализаторов.
