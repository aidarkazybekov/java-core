## Title
Classes & Objects

## Summary
Класс — шаблон (контракт) с полями и методами; объект — экземпляр класса, выделенный в куче во время исполнения. Ссылочная переменная лежит на стеке, сам объект — в куче. Java **всегда** передаёт параметры по значению — для объектов копируется ссылка, не сам объект. Четыре принципа ООП: абстракция, инкапсуляция, наследование, полиморфизм.

## Deep Dive
## Класс vs объект vs экземпляр

- **Класс** — compile-time конструкция. `.class`-файл содержит constant pool, дескрипторы полей, байт-код методов, access flags. JVM загружает класс через ClassLoader, верифицирует и связывает его **один раз**.
- **Объект** — runtime-инстанс. При `new MyClass()` JVM выделяет память на куче под поля экземпляра плюс заголовок объекта.
- **Экземпляр** — то же, что объект. Ссылка на экземпляр живёт на стеке (или внутри другого объекта, если это поле).

## Layout объекта на HotSpot (64-bit, compressed oops)

| Часть | Размер | Описание |
|-------|--------|----------|
| Mark word | 8 байт | hashCode, GC age, lock state, biased locking |
| Klass pointer | 4 байта | compressed, указывает на метаданные в Metaspace |
| Instance fields | N байт | поля экземпляра с field reordering и padding |
| Array length | 4 байта | только для массивов, после заголовка |
| Padding | 0-7 байт | выравнивание до 8-байтной границы |

Даже пустой `new Object()` занимает **16 байт** (12 заголовок + 4 padding).

Отключение compressed oops (`-XX:-UseCompressedOops`) делает klass pointer 8 байт → overhead растёт.

## Четыре принципа ООП

- **Абстракция** — выделение ключевых характеристик, скрытие деталей реализации. Через `abstract` классы и интерфейсы.
- **Инкапсуляция** — объединение данных и методов, защита состояния через модификаторы доступа. Поля `private`, доступ через getter/setter (или вообще без них — для immutable).
- **Наследование** — новые классы на основе существующих. Java поддерживает только **одиночное** наследование классов (но множественную реализацию интерфейсов).
- **Полиморфизм** — одна сущность в разных формах: overload (compile-time, по сигнатуре) и override (runtime, виртуальная диспетчеризация).

## Identity vs equality

- `==` — сравнение **ссылок** (указатели).
- `.equals()` — сравнение **значения** (если переопределено).

Дефолтный `Object.equals()` эквивалентен `==` — проверяет identity. Это **почти никогда** не то, чего хочет доменный код.

> [!gotcha]
> Контракт `equals()` / `hashCode()`: равные объекты **обязаны** иметь одинаковый `hashCode`. Нарушение приводит к тому, что объект «теряется» в HashMap — положили по ключу, а `get` возвращает null. Всегда переопределяйте обе вместе.

## Передача по значению — всегда

Java **не имеет** pass-by-reference. Для ссылочных типов **копируется указатель** (сам указатель — значение):

```java
void tryReassign(Employee e) {
    e = new Employee("Ghost", 0);   // меняем локальную копию — вызывающий не видит
}

void mutate(Employee e) {
    e.salary = 999_999;              // мутируем объект — вызывающий видит
}
```

**Переприсвоение параметра** не влияет на вызывающего. **Мутация объекта** через скопированную ссылку видна всем, у кого есть ссылка на тот же объект.

## Жизненный цикл объекта

1. **Аллокация** на куче (обычно Eden в Young Gen).
2. **Инициализация**: поля → init-блоки → конструктор, от предка к потомку.
3. **Использование** — через достижимые ссылки.
4. **Unreachable** — когда сильных ссылок не осталось, объект становится eligible for GC.
5. **Reclaim** — GC освобождает память (в будущем цикле).

Между «unreachable» и «reclaim» объект может быть soft/weak/phantom-reachable:

- **SoftReference** — GC собирает, только если нужна память (кэш).
- **WeakReference** — GC собирает при следующем цикле (WeakHashMap).
- **PhantomReference** — объект уже собран, но не освобождён; используется для cleanup actions.

> [!production]
> `finalize()` **deprecated с Java 9**. Используйте `try-with-resources` и `java.lang.ref.Cleaner`. Finalizer выполняется в недетерминированном порядке, может «воскресить» объект (вернуть ссылку в поле), запускается отдельным потоком — источник утечек и непредсказуемого поведения.

## Nested и inner classes

- **Static nested class** — не держит ссылку на enclosing instance. Предпочтительный вариант.
- **Inner class** (non-static) — **неявно хранит** ссылку на enclosing instance. Если inner class переживёт outer (листенер, колбек в другом потоке) — **memory leak**.
- **Local class** — объявлен внутри метода. Может захватывать effectively-final локальные переменные.
- **Anonymous class** — локальный с одноразовой реализацией. До Java 8 — де-факто реализация лямбд; сейчас их вытеснили лямбды для функциональных интерфейсов.

## Code
```java
// Demonstrates class structure, object creation, and reference semantics
public class Employee {
    // Instance fields (stored on the heap per object)
    private final String name;
    private double salary;

    // Static field (stored in metaspace, shared across all instances)
    private static int headcount = 0;

    public Employee(String name, double salary) {
        this.name = name;
        this.salary = salary;
        headcount++;
    }

    // Passing references by value demo
    public static void tryReassign(Employee e) {
        e = new Employee("Ghost", 0); // caller's reference is unchanged
    }

    public static void mutate(Employee e) {
        e.salary = 999_999;           // caller sees the mutation
    }

    public static void main(String[] args) {
        Employee a = new Employee("Alice", 80_000);
        Employee b = a;               // both point to the same heap object

        System.out.println(a == b);   // true  (same reference)

        tryReassign(a);
        System.out.println(a.name);   // "Alice" (unchanged)

        mutate(a);
        System.out.println(a.salary); // 999999.0 (mutated via copied ref)
        System.out.println(b.salary); // 999999.0 (b points to same object)
    }
}
```

## Tip
Когда интервьюер спрашивает «передаётся ли в Java по ссылке» — нарисуйте стек-кучу диаграмму с двумя переменными-ссылками на один и тот же heap-объект. Это сразу показывает, что вы понимаете механику.

### explanation
Spring управляет созданием объектов через IoC-контейнер. Вместо `new` вы объявляете бины (классы), а Spring инстанцирует, конфигурирует и связывает их.

Понимание того, как создаются и ссылаются объекты, критично для scope'ов:
- **singleton** — один инстанс на ApplicationContext; все инъекции получают ту же ссылку.
- **prototype** — новый heap-объект на каждую инъекцию; GC соберёт, когда ссылки отпустят.
- **request / session** — привязка жизненного цикла к HTTP-границам.

Типичный баг: singleton зависит от request-scope бина напрямую — захватывает одну копию навсегда. Решается scoped proxy.

## Interview
### [3-1-q0 | junior] В чём разница между классом и объектом в Java?
- **Класс** — compile-time шаблон, `.class`-файл с определениями полей, байт-кодом методов и метаданными. Загружается ClassLoader'ом один раз.
- **Объект** — runtime инстанс, выделенный в куче при `new`. Из одного класса можно создать много объектов.
- Ссылочная переменная (стек) указывает на объект (куча).

Класс живёт в Metaspace; объект — в Heap; ссылка на объект — в стеке или внутри другого объекта.

### [3-1-q1 | junior] Какие четыре принципа ООП? Объясните каждый.
- **Абстракция** — выделяем существенное, скрываем детали. Интерфейсы и `abstract` классы позволяют работать через контракт, не зная реализации.
- **Инкапсуляция** — объединяем данные и методы, защищаем состояние. Поля `private`, доступ через методы. Доводим до крайности — immutable объекты без сеттеров вообще.
- **Наследование** — новый класс использует функциональность существующего. Java — **одиночное** наследование классов, множественная реализация интерфейсов.
- **Полиморфизм** — одна сущность в разных формах: overload (по сигнатуре, compile-time) и override (виртуальная диспетчеризация, runtime).

### [3-1-q2 | mid] Java передаёт параметры по ссылке или по значению? Объясните на примере.
Java **всегда pass-by-value**. Для примитивов копируется значение. Для объектов копируется **ссылка** (указатель — тоже значение).

Внутри метода переприсвоение параметра (`e = new Employee(...)`) не влияет на вызывающего. Но мутация объекта через скопированную ссылку (`e.salary = 999_999`) видна всем, у кого есть ссылка на тот же объект.

```java
Employee a = new Employee("Alice", 80_000);
tryReassign(a);      // внутри e = new Employee(...) — a не меняется
mutate(a);           // внутри e.salary = ... — a.salary меняется
```

Распространённое заблуждение: «передача ссылки» ≠ pass-by-reference. Java передаёт значение ссылки, а не саму переменную.

### [3-1-q3 | senior] Объясните layout Java-объекта на HotSpot и почему пустой `new Object()` занимает 16 байт.
На HotSpot 64-bit с compressed oops объект имеет:

- **Mark word** — 8 байт (hashCode, GC age, lock state, biased locking).
- **Klass pointer** — 4 байта (compressed; указывает на метаданные в Metaspace).
- **Instance fields** — N байт с field reordering и padding.
- **Array length** — 4 байта, только для массивов.
- **Padding** — выравнивание до 8-байтной границы.

Заголовок: 8 + 4 = **12 байт**. `new Object()` полей не имеет, но дополняется до 16 (8-byte alignment). Отключение compressed oops (`-XX:-UseCompressedOops`) делает klass pointer 8 байт — объект становится ~24 байта.

Для массива добавьте ещё 4 байта на `length`, плюс содержимое.

Это и причина, почему `List<Integer>` такой жирный по сравнению с `int[]` — каждый `Integer` — 16-байтный объект ради 4 байт данных.

### [3-1-q4 | senior] Какая разница между static nested и inner class, и в чём риск inner-классов?
- **Static nested class** — обычный класс, вложенный в другой. **Не** хранит ссылку на enclosing instance. Может быть инстанцирован без объекта outer-класса.
- **Inner (non-static) class** — неявно держит ссылку на enclosing instance через скрытое поле `this$0`. Инстанцируется только через объект outer-класса.

**Риск inner-класса** — memory leak. Если inner class передан как listener/callback и переживёт outer, ссылка `this$0` не даёт GC собрать outer → утекает всё содержимое outer-класса.

Классика: inner-`Runnable`, отправленный в `ExecutorService` с длинным временем жизни. Outer-класс может быть Activity / ApplicationContext / service — и вся графы объектов за ним держится живой.

**Правило**: используйте `static nested class` по умолчанию. Делайте inner, только если реально нужен доступ к полям enclosing instance.
