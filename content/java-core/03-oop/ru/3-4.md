## Title
Polymorphism

## Summary
Полиморфизм — возможность одной сущности принимать разные формы. Три вида в Java: **ad-hoc (статический)** — перегрузка методов; **параметрический** — дженерики; **динамический (субтиповой)** — переопределение методов с виртуальной диспетчеризацией через vtable.

## Deep Dive
## Три вида полиморфизма

- **Ad-hoc (compile-time)** — **перегрузка** методов. Несколько методов с одним именем, разные параметры. Компилятор выбирает нужный по типам аргументов.
- **Параметрический** — дженерики. `List<T>` один и тот же класс, но специализирован под разные типы в compile time.
- **Subtype / Dynamic** — **переопределение**. Код работает против parent-ссылки, JVM выбирает реализацию по runtime-типу объекта.

## Перегрузка vs Переопределение

| | Overload | Override |
|--|----------|----------|
| Когда | Compile-time | Runtime |
| Где | В одном классе (или наследуется) | В подклассе |
| Что меняется | Параметры | Реализация (та же сигнатура) |
| Диспетчеризация | Статическая | Динамическая (vtable) |
| `@Override` | Нет | Да |

**Сигнатура метода** = имя + список параметров (типы, порядок). **Не** входят в сигнатуру: возвращаемый тип, checked-исключения, модификаторы.

## Virtual method dispatch — как JVM это делает

Каждый класс имеет **vtable** (virtual method table), построенную при загрузке класса. Это массив указателей на реализации методов. Override меняет запись в vtable на реализацию потомка.

На call-site:
1. JVM читает klass pointer из заголовка объекта.
2. Находит vtable через метаданные класса в Metaspace.
3. Прыгает на метод по фиксированному индексу.

HotSpot оптимизирует дальше:
- **Monomorphic inline cache** — вызов всегда видит один тип → метод инлайнится напрямую.
- **Bimorphic** — два типа → ветка-if.
- **Megamorphic** — три+ типов → полный vtable lookup.

Deoptimization случается, если загрузился новый подкласс, ломающий monomorphic-предположение.

## Правила разрешения перегрузки

Компилятор ищет **наиболее специфичный** применимый метод. Приоритет (для Java 5+ с autoboxing):

1. **Exact match / widening** (без boxing, без varargs).
2. **Autoboxing** (если widening не подошёл).
3. **Varargs** (в последнюю очередь).

```java
static void f(long x)     {}
static void f(Integer x)  {}
static void f(Object... x) {}

int i = 42;
f(i);  // вызывается f(long) — widening побеждает boxing
```

> [!gotcha]
> Добавление перегрузки может скрытно изменить диспетчеризацию существующих вызовов. Если между `Integer` и `long` появится `Object...` — поведение не поменяется, но появление `Object` перегрузки может поменять resolution для некоторых типов.

## Ковариантные возвращаемые типы (Java 5+)

Override может возвращать **более узкий** тип, чем родитель:

```java
class Shape { Shape copy() { ... } }
class Circle extends Shape {
    @Override Circle copy() { ... }  // OK, covariant return
}
```

**Контравариантных параметров** Java **не** поддерживает: изменение типа параметра создаст перегрузку, не override — типичная ошибка.

## Bridge-методы

Компилятор генерирует **синтетические bridge-методы** для сохранения полиморфизма после **type erasure** дженериков и для covariant return types.

Пример:
```java
class StringList extends ArrayList<String> {
    @Override public String get(int i) { ... }  // возвращает String
}
```

После erasure родительский `get` возвращает `Object`. Компилятор создаёт bridge-метод `Object get(int)`, который делегирует в `String get(int)` — сохраняя vtable-контракт.

Bridge-методы видны в рефлексии: `Method.isBridge()`. Frameworks, обходящие declared methods, должны их учитывать.

## Liskov Substitution Principle

LSP: любой код, работающий с `Parent`, должен продолжать работать, если подставить `Child`. Подкласс **не может**:
- Усиливать precondition (принимать меньше входов).
- Ослаблять postcondition (гарантировать меньше на выходе).
- Бросать неожиданные исключения.

Нарушение LSP — частый баг в иерархиях наследования (`Square extends Rectangle` классический пример).

## Code
```java
public abstract class Shape {
    public abstract double area();

    @Override
    public String toString() {
        return getClass().getSimpleName() + " area=" + area();
    }
}

public class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle extends Shape {
    private final double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override
    public double area() { return w * h; }
}

// --- Overloading resolution demo ---
public class OverloadDemo {
    static String describe(Object o)  { return "Object";  }
    static String describe(String s)  { return "String";  }
    static String describe(long l)    { return "long";    }
    static String describe(Integer i) { return "Integer"; }

    public static void main(String[] args) {
        // Runtime polymorphism
        Shape s = new Circle(5);
        System.out.println(s);         // "Circle area=78.539..."
        s = new Rectangle(3, 4);
        System.out.println(s);         // "Rectangle area=12.0"

        // Overloading resolution
        int x = 42;
        System.out.println(describe(x));          // "long" (widening beats boxing)
        System.out.println(describe("hello"));    // "String" (most specific)
        System.out.println(describe((Object)"a"));// "Object" (explicit cast)
    }
}
```

## Tip
При объяснении полиморфизма на интервью всегда дайте конкретный пример: переменная объявлена parent-типом, хранит child-инстанс, вызываем переопределённый метод. Потом объясните, какая версия вызывается и почему.

## Spring
### concept
Polymorphism

### springFeature
Spring Interface-Based Design

### explanation
Spring активно использует полиморфизм. Вы программируете против **интерфейсов** (`UserService`, `UserRepository`), а Spring инжектит конкретную реализацию в рантайме.

Это runtime-полиморфизм в чистом виде. Следствия:
- Реализации можно заменять (`JpaUserRepository` vs `MongoUserRepository`) без изменения потребителей.
- Моки в тестах работают благодаря тому, что потребители зависят от интерфейса, не от реализации.
- JDK dynamic proxy AOP работает только для бинов, реализующих интерфейс (для остальных — CGLIB, наследование).

## Interview
### [3-4-q0 | junior] В чём разница между overloading и overriding?
- **Overloading (перегрузка)** — compile-time полиморфизм. Одно имя метода, разные параметры. Компилятор разрешает вызов по типам аргументов.
- **Overriding (переопределение)** — runtime полиморфизм. Подкласс переопределяет метод родителя с **той же сигнатурой**. JVM разрешает вызов по runtime-типу объекта через vtable.

Overloading — в одном классе (или наследуется). Overriding — между родителем и потомком.

В сигнатуру метода входят имя и параметры; возвращаемый тип, исключения и модификаторы — **не** входят. Поэтому нельзя перегрузить по возвращаемому типу или списку исключений.

### [3-4-q1 | mid] Как JVM внутри выполняет dynamic method dispatch?
У каждого класса при загрузке строится **vtable** (virtual method table) — массив указателей на реализации методов. Когда подкласс переопределяет метод, его запись в vtable указывает на реализацию потомка.

На call-site:
1. JVM читает **klass pointer** из заголовка объекта.
2. Находит vtable через метаданные класса (Metaspace).
3. Прыгает на метод по фиксированному индексу.

**HotSpot оптимизирует**:
- **Monomorphic inline cache** — call-site всегда видит один тип → метод инлайнится напрямую.
- **Bimorphic** — два типа → `if`-ветка.
- **Megamorphic** — 3+ типов → полный vtable lookup.

Если загрузится новый подкласс, нарушающий monomorphic-предположение, происходит **deoptimization** — откат к интерпретатору и перекомпиляция без этой оптимизации. В графиках это видно как spike latency.

### [3-4-q2 | senior] Что такое bridge-методы и как они связаны с полиморфизмом?
Bridge-методы — **синтетические** методы, которые компилятор генерирует для сохранения полиморфизма после **type erasure** дженериков и для **covariant return types**.

Пример с erasure:
```java
class StringList extends ArrayList<String> {
    @Override public String get(int i) { ... }
}
```

После erasure родительский `get` возвращает `Object`. Чтобы vtable-контракт работал, компилятор создаёт bridge:
```java
// синтетический, добавлен компилятором
public Object get(int i) { return this.<real>get(i); }
```

Bridge-методы можно обнаружить через `Method.isBridge()`. Frameworks, итерирующие `Class.getDeclaredMethods()`, должны их пропускать — иначе при построении прокси или сериализации получается дубликат.

Также генерируются при covariant return (подкласс возвращает более узкий тип).

### [3-4-q3 | senior] Что такое Liskov Substitution Principle и приведите пример его нарушения.
LSP: любой код, работающий с `Parent`, должен корректно работать, если подставить `Child`.

Подкласс **не должен**:
- Усиливать precondition (требовать больше от входа).
- Ослаблять postcondition (гарантировать меньше на выходе).
- Бросать неожиданные исключения.

**Классический пример — Square extends Rectangle:**
```java
class Rectangle { int w, h; void setW(int w){this.w=w;} void setH(int h){this.h=h;} }
class Square extends Rectangle { void setW(int w){this.w=w; this.h=w;} void setH(int h){this.h=h; this.w=h;} }

void test(Rectangle r) { r.setW(5); r.setH(10); assert r.area() == 50; }
test(new Square());  // FAIL: area == 100, не 50
```

Код, полагающийся на независимость `setW` и `setH` (естественное предположение для Rectangle), ломается, когда передают Square.

**Правильно**: Rectangle и Square — параллельные классы с общим интерфейсом `Shape`, а не наследование. IS-A геометрически есть, но IS-A **поведения** — нет.
