## Title
Design Patterns

## Summary
Паттерны — типовые решения повторяющихся задач проектирования (GoF, 1994). Три группы: **порождающие** (Singleton, Factory, Builder, Prototype), **структурные** (Adapter, Decorator, Proxy, Facade, Composite, Flyweight), **поведенческие** (Strategy, Observer, Template Method, State, Chain of Responsibility). Modern Java (лямбды, рекорды, sealed) часто упрощает или заменяет их.

## Deep Dive
## Порождающие (Creational)

**Контролируют создание объектов.**

- **Singleton** — один экземпляр на scope. В Java лучший способ — `enum Singleton { INSTANCE; }` (thread-safe, защищён от reflection, корректно сериализуется). См. также Initialization-on-Demand Holder (3-6).
- **Factory Method** — интерфейс создания, подклассы выбирают тип. JDK-пример: `Collection.iterator()` — каждая коллекция возвращает свой итератор.
- **Abstract Factory** — семейство связанных объектов. `DocumentBuilderFactory`, `XMLInputFactory`.
- **Builder** — пошаговое создание сложного объекта, особенно незаменим для immutable с многими optional-параметрами (Effective Java Item 2). JDK: `StringBuilder`, `Stream.Builder`, `HttpRequest.newBuilder()`.
- **Prototype** — копирование без знания конкретного типа. В Java обычно через copy-конструктор, не через `clone()`.

## Структурные (Structural)

**Композиция объектов.**

- **Adapter** — преобразует один интерфейс в другой. `Arrays.asList(T[])` адаптирует массив к `List`. `InputStreamReader` адаптирует `InputStream` (bytes) к `Reader` (chars).
- **Decorator** — динамически добавляет поведение, не трогая интерфейс. **JDK I/O streams** — эталон: `new BufferedReader(new InputStreamReader(new FileInputStream(...)))`.
- **Proxy** — суррогат, контролирующий доступ к реальному объекту. Варианты:
  - Remote proxy (RMI).
  - Virtual proxy (lazy loading, JPA proxies).
  - Protection proxy (security checks).
  - Smart proxy (добавляет логику — Spring AOP).
- **Facade** — простой интерфейс к сложной подсистеме. Spring `JdbcTemplate` — фасад над JDBC.
- **Composite** — древовидная структура, где листья и контейнеры обрабатываются единообразно. UI-компоненты, AST компиляторов.
- **Flyweight** — разделение общего состояния для экономии памяти. `Integer.valueOf(-128..127)`, `String.intern()`.

## Поведенческие (Behavioural)

**Взаимодействие между объектами.**

- **Strategy** — взаимозаменяемые алгоритмы за одним интерфейсом. JDK: `Comparator`. С лямбдами превращается в один лайнер:
  ```java
  list.sort((a, b) -> a.name().compareTo(b.name()));
  ```
- **Observer** — один-ко-многим подписка на события. Classic `java.util.Observer` (deprecated). Modern: `ApplicationEventPublisher` в Spring, `Flow.Subscriber` (reactive streams).
- **Template Method** — скелет алгоритма в базовом классе + hook-методы, которые переопределяют потомки. JDK: `AbstractList`, `HttpServlet.doGet`. Spring: `JdbcTemplate`, `RestTemplate`.
- **Chain of Responsibility** — запрос проходит по цепочке хэндлеров. **Servlet Filter chain**, **Spring Security FilterChain**, **Spring HandlerInterceptor**.
- **State** — объект меняет поведение в зависимости от внутреннего состояния. Можно реализовать через sealed-иерархию.
- **Command** — объектный запрос с параметрами. `Runnable`, `Callable`. С лямбдами почти исчезает как отдельный паттерн.
- **Iterator** — обход без раскрытия структуры. Java-контракт: `Iterator<T>` с `hasNext()` / `next()`.
- **Memento** — снимок состояния для отмены. Undo/redo в редакторах.
- **Visitor** — double dispatch. В Java 21 заменяется **sealed + switch с pattern matching** — элегантнее и exhaustiveness compile-time.

## Как modern Java изменил паттерны

> [!tip]
> - **Лямбды** схлопывают Strategy, Command, Observer, Callback в одну строку — больше не нужен отдельный класс-обёртка.
> - **Records** (Java 16) убирают Builder для простых value-объектов с малым числом полей.
> - **Sealed + switch pattern matching** (Java 21) реализует Visitor без Visitor-интерфейса, с exhaustiveness-проверкой компилятора.
> - **DI-контейнеры** (Spring, Guice) убирают ручной Singleton и Factory для управляемых объектов.
> - **Optional** и **Stream** — встроенный Null Object и Iterator/Builder в одном.

Паттерны не исчезают — меняется синтаксис их выражения. Понимание паттернов важно как **общий словарь** для обсуждения дизайна.

## Практические замечания на интервью

> [!production]
> - Не просто называйте паттерн — покажите **проблему**, которую он решает, и **trade-off**.
> - Свяжите с **JDK/Spring**: «Strategy — это `Comparator`», «Proxy — это Spring AOP».
> - Осознавайте недостатки: Singleton → скрытое глобальное состояние, сложнее тестировать. Strategy → индирекция.
> - Знайте, когда **не** применять: Builder для 2-3 обязательных полей — overkill. Observer для синхронной коммуникации — ненужная сложность.

## Code
```java
// --- Strategy Pattern (with lambdas) ---
import java.util.*;

@FunctionalInterface
interface PricingStrategy {
    double calculate(double basePrice, int quantity);
}

class OrderCalculator {
    private final PricingStrategy strategy;
    OrderCalculator(PricingStrategy strategy) { this.strategy = strategy; }
    double total(double basePrice, int qty) { return strategy.calculate(basePrice, qty); }
}

// --- Builder Pattern (immutable object) ---
class HttpRequest {
    private final String method, url, body;
    private final Map<String, String> headers;

    private HttpRequest(Builder b) {
        this.method = b.method;
        this.url = b.url;
        this.headers = Map.copyOf(b.headers);
        this.body = b.body;
    }

    static class Builder {
        private final String method, url;
        private Map<String, String> headers = new LinkedHashMap<>();
        private String body;
        Builder(String method, String url) { this.method = method; this.url = url; }
        Builder header(String k, String v) { headers.put(k, v); return this; }
        Builder body(String body) { this.body = body; return this; }
        HttpRequest build() { return new HttpRequest(this); }
    }
}

class PatternDemo {
    public static void main(String[] args) {
        PricingStrategy bulk = (price, qty) -> price * qty * 0.9;
        PricingStrategy retail = (price, qty) -> price * qty;

        var bulkCalc = new OrderCalculator(bulk);
        System.out.println(bulkCalc.total(100, 5));   // 450.0

        var retailCalc = new OrderCalculator(retail);
        System.out.println(retailCalc.total(100, 5)); // 500.0

        HttpRequest req = new HttpRequest.Builder("POST", "/api/orders")
            .header("Content-Type", "application/json")
            .body("{\"item\":\"laptop\"}")
            .build();
    }
}
```

## Tip
На интервью по паттернам всегда связывайте с **реальным примером из JDK или Spring**. «Strategy — это Comparator», «Proxy — основа Spring AOP» — такие связки показывают, что вы видите паттерны в живом коде, а не только в учебнике.

### explanation
Spring — выставка паттернов:

- **Singleton** — дефолтный scope бина.
- **Factory** — `BeanFactory`, `@Bean`-методы, `FactoryBean`.
- **Proxy** — AOP-прокси для `@Transactional`, `@Cacheable`, `@Async`, `@Secured`.
- **Template Method** — `JdbcTemplate`, `RestTemplate`, `JmsTemplate` — скелет операции с callback'ами.
- **Observer** — `ApplicationEventPublisher` + `@EventListener`.
- **Strategy** — pluggable `HandlerMapping`, `ViewResolver`, `ConversionService`.
- **Chain of Responsibility** — `FilterChain` в Spring Security, `HandlerInterceptor` в MVC.
- **Decorator** — `HttpServletRequestWrapper`, обёртки `DataSource`.
- **Facade** — `JdbcTemplate` над голым JDBC.

Понимание GoF — это понимание **архитектуры Spring**. Без них фреймворк кажется магией.

## Interview
### [3-8-q0 | junior] Объясните Singleton и назовите два способа реализации в Java.
Singleton — один экземпляр класса на scope, с глобальной точкой доступа.

**1. Enum singleton** (рекомендация Josh Bloch):
```java
public enum Db { INSTANCE; public void doWork() { ... } }
```
Thread-safe by construction, **защищён от reflection** (JVM не даёт создать второй enum-инстанс), корректно сериализуется.

**2. Initialization-on-Demand Holder**:
```java
public class Singleton {
    private Singleton() {}
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```
Ленивый, thread-safe через гарантии class init lock'а JVM.

**Double-checked locking** с `volatile` — третий вариант, но он error-prone (особенно до Java 5 с ослабленной JMM).

Недостатки Singleton в целом: скрытое глобальное состояние, сложности в тестировании. В Spring singleton-scope бина — альтернатива.

### [3-8-q1 | mid] В чём разница между Decorator и Proxy? Дайте Java-пример каждого.
Оба оборачивают объект, но **намерение** разное.

**Decorator** — добавляет поведение **прозрачно** (тот же интерфейс, расширенный функционал).
Пример — Java I/O streams:
```java
Reader r = new BufferedReader(new InputStreamReader(new FileInputStream(path)));
// FileInputStream — данные; InputStreamReader — bytes→chars; BufferedReader — буферизация
```
Каждый слой добавляет свою ответственность, сохраняя контракт `Reader`.

**Proxy** — **контролирует доступ** к реальному объекту (lazy loading, security, remote access).
Примеры:
- `Collections.unmodifiableList(list)` — proxy, блокирующий мутации.
- JPA lazy-loading proxy — не достаёт данные до первого обращения.
- Spring AOP proxy — перехватывает вызовы для `@Transactional`, `@Cacheable`.

**Структурное отличие**: Proxy часто **создаёт или управляет** реальным объектом внутри. Decorator получает его **извне** как компонент.

### [3-8-q2 | senior] Как лямбды, рекорды и sealed-классы изменили реализацию классических GoF-паттернов в modern Java?
**Лямбды** схлопывают многие behavioural-паттерны в одну строку:
- **Strategy** → лямбда, передаваемая методу. `list.sort((a, b) -> ...)`.
- **Command** → `Runnable` / `Supplier`.
- **Observer** → `Consumer<Event>` как callback.

**Records (Java 16)** убирают Builder-boilerplate для простых value-объектов. Если все поля обязательные и их немного — canonical-конструктор рекорда достаточен:
```java
record Point(int x, int y) {}  // вместо PointBuilder
```

**Sealed-классы (Java 17)** + pattern matching switch (Java 21) реализуют Visitor без отдельного Visitor-интерфейса, с **compile-time exhaustiveness**:
```java
sealed interface Shape permits Circle, Square {}
double area = switch (s) {
    case Circle c -> Math.PI * c.r() * c.r();
    case Square sq -> sq.side() * sq.side();
};  // компилятор проверит все case
```
Abstract Factory может возвращать sealed-тип, потребитель использует pattern matching вместо полиморфной диспетчеризации.

**Паттерны не исчезают** — меняется синтаксис. Из class-based plumbing код становится более декларативным, expression-oriented.
