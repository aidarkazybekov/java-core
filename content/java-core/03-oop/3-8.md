## Title
Design Patterns

## Summary
Patterns are proven solutions to recurring design problems (GoF, 1994). Three groups: **creational** (Singleton, Factory, Builder, Prototype), **structural** (Adapter, Decorator, Proxy, Facade, Composite, Flyweight), **behavioural** (Strategy, Observer, Template Method, State, Chain of Responsibility). Modern Java (lambdas, records, sealed) often simplifies or replaces them.

## Deep Dive
## Creational

**Control object creation.**

- **Singleton** — one instance per scope. In Java the best form is `enum Singleton { INSTANCE; }` (thread-safe, reflection-proof, serialises correctly). Also see Initialization-on-Demand Holder (3-6).
- **Factory Method** — an interface for creation, subclasses pick the type. JDK example: `Collection.iterator()` — every collection returns its own iterator.
- **Abstract Factory** — families of related objects. `DocumentBuilderFactory`, `XMLInputFactory`.
- **Builder** — step-by-step construction of a complex object, essential for immutables with many optional parameters (Effective Java Item 2). JDK: `StringBuilder`, `Stream.Builder`, `HttpRequest.newBuilder()`.
- **Prototype** — copying without knowing the concrete type. In Java usually via a copy constructor, not `clone()`.

## Structural

**Object composition.**

- **Adapter** — converts one interface to another. `Arrays.asList(T[])` adapts an array to `List`. `InputStreamReader` adapts `InputStream` (bytes) to `Reader` (chars).
- **Decorator** — adds behaviour dynamically without changing the interface. **JDK I/O streams** are the canonical example: `new BufferedReader(new InputStreamReader(new FileInputStream(...)))`.
- **Proxy** — a surrogate that controls access to the real object. Variants:
  - Remote proxy (RMI).
  - Virtual proxy (lazy loading, JPA proxies).
  - Protection proxy (security checks).
  - Smart proxy (adds logic — Spring AOP).
- **Facade** — a simple interface to a complex subsystem. Spring `JdbcTemplate` is a facade over JDBC.
- **Composite** — a tree where leaves and containers are handled uniformly. UI components, compiler ASTs.
- **Flyweight** — shared state for memory savings. `Integer.valueOf(-128..127)`, `String.intern()`.

## Behavioural

**Object interaction.**

- **Strategy** — interchangeable algorithms behind one interface. JDK: `Comparator`. With lambdas it collapses to a one-liner:
  ```java
  list.sort((a, b) -> a.name().compareTo(b.name()));
  ```
- **Observer** — one-to-many event subscription. Classic `java.util.Observer` (deprecated). Modern: `ApplicationEventPublisher` in Spring, `Flow.Subscriber` (reactive streams).
- **Template Method** — algorithm skeleton in the base class + hook methods subclasses override. JDK: `AbstractList`, `HttpServlet.doGet`. Spring: `JdbcTemplate`, `RestTemplate`.
- **Chain of Responsibility** — a request travels through a chain of handlers. **Servlet Filter chain**, **Spring Security FilterChain**, **Spring HandlerInterceptor**.
- **State** — an object changes behaviour with its internal state. Can be modelled with a sealed hierarchy.
- **Command** — an object-wrapped request. `Runnable`, `Callable`. With lambdas it almost disappears as a standalone pattern.
- **Iterator** — traversal without exposing structure. Java's `Iterator<T>` with `hasNext()` / `next()`.
- **Memento** — a state snapshot for undo. Editors' undo/redo.
- **Visitor** — double dispatch. In Java 21 replaced by **sealed + switch pattern matching** — more elegant with compile-time exhaustiveness.

## How modern Java reshaped the patterns

> [!tip]
> - **Lambdas** collapse Strategy, Command, Observer, Callback into a single expression — no wrapper class needed.
> - **Records** (Java 16) eliminate Builder for simple value objects with few fields.
> - **Sealed + switch pattern matching** (Java 21) implements Visitor without a Visitor interface, with compiler-checked exhaustiveness.
> - **DI containers** (Spring, Guice) replace manual Singleton and Factory for managed objects.
> - **Optional** and **Stream** — built-in Null Object and Iterator/Builder rolled together.

Patterns don't disappear — their syntactic expression changes. Understanding patterns is still valuable as a **shared vocabulary** for design discussions.

## Interview practice notes

> [!production]
> - Don't just name a pattern — show the **problem** it solves and the **trade-off**.
> - Tie it to **JDK/Spring**: "Strategy is `Comparator`", "Proxy is Spring AOP".
> - Know the downsides: Singleton → hidden global state, harder to test. Strategy → indirection.
> - Know when **not** to apply: Builder for 2-3 required fields is overkill. Observer for synchronous communication is unnecessary complexity.

## Code
```java
// --- Strategy Pattern (with lambdas) ---
import java.util.*;

@FunctionalInterface
interface PricingStrategy {
    double calculate(double basePrice, int quantity);
}

class OrderCalculator {
    private final PricingStrategy strategy;
    OrderCalculator(PricingStrategy strategy) { this.strategy = strategy; }
    double total(double basePrice, int qty) { return strategy.calculate(basePrice, qty); }
}

// --- Builder Pattern (immutable object) ---
class HttpRequest {
    private final String method, url, body;
    private final Map<String, String> headers;

    private HttpRequest(Builder b) {
        this.method = b.method;
        this.url = b.url;
        this.headers = Map.copyOf(b.headers);
        this.body = b.body;
    }

    static class Builder {
        private final String method, url;
        private Map<String, String> headers = new LinkedHashMap<>();
        private String body;
        Builder(String method, String url) { this.method = method; this.url = url; }
        Builder header(String k, String v) { headers.put(k, v); return this; }
        Builder body(String body) { this.body = body; return this; }
        HttpRequest build() { return new HttpRequest(this); }
    }
}

class PatternDemo {
    public static void main(String[] args) {
        PricingStrategy bulk = (price, qty) -> price * qty * 0.9;
        PricingStrategy retail = (price, qty) -> price * qty;

        var bulkCalc = new OrderCalculator(bulk);
        System.out.println(bulkCalc.total(100, 5));   // 450.0

        var retailCalc = new OrderCalculator(retail);
        System.out.println(retailCalc.total(100, 5)); // 500.0

        HttpRequest req = new HttpRequest.Builder("POST", "/api/orders")
            .header("Content-Type", "application/json")
            .body("{\"item\":\"laptop\"}")
            .build();
    }
}
```

In design-pattern interviews, always tie the pattern to a **real JDK or Spring example**. "Strategy is Comparator", "Proxy is the foundation of Spring AOP" — these links show you see patterns in live code, not just textbooks.

### explanation
Spring is a patterns showcase:

- **Singleton** — default bean scope.
- **Factory** — `BeanFactory`, `@Bean` methods, `FactoryBean`.
- **Proxy** — AOP proxies for `@Transactional`, `@Cacheable`, `@Async`, `@Secured`.
- **Template Method** — `JdbcTemplate`, `RestTemplate`, `JmsTemplate` — operation skeletons with callbacks.
- **Observer** — `ApplicationEventPublisher` + `@EventListener`.
- **Strategy** — pluggable `HandlerMapping`, `ViewResolver`, `ConversionService`.
- **Chain of Responsibility** — `FilterChain` in Spring Security, `HandlerInterceptor` in MVC.
- **Decorator** — `HttpServletRequestWrapper`, `DataSource` wrappers.
- **Facade** — `JdbcTemplate` over raw JDBC.

Understanding GoF is understanding **Spring's architecture**. Without them the framework feels like magic.

## Interview
### [3-8-q0 | junior] Explain the Singleton pattern and name two ways to implement it in Java.
Singleton — one instance per scope, with a global access point.

**1. Enum singleton** (Josh Bloch's recommendation):
```java
public enum Db { INSTANCE; public void doWork() { ... } }
```
Thread-safe by construction, **immune to reflection** (JVM won't let you create a second enum instance), serialises correctly.

**2. Initialization-on-Demand Holder**:
```java
public class Singleton {
    private Singleton() {}
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```
Lazy, thread-safe via the JVM's class init lock guarantees.

**Double-checked locking** with `volatile` is a third option but error-prone (especially pre-Java-5 with a looser JMM).

Overall Singleton downsides: hidden global state, harder to test. A Spring singleton-scoped bean is the usual modern alternative.

### [3-8-q1 | mid] What's the difference between Decorator and Proxy? Give a Java example of each.
Both wrap an object, but their **intent** differs.

**Decorator** — adds behaviour **transparently** (same interface, enhanced functionality).
Example — Java I/O streams:
```java
Reader r = new BufferedReader(new InputStreamReader(new FileInputStream(path)));
// FileInputStream — data; InputStreamReader — bytes→chars; BufferedReader — buffering
```
Each layer adds a concern while keeping the `Reader` contract.

**Proxy** — **controls access** to the real object (lazy loading, security, remote access).
Examples:
- `Collections.unmodifiableList(list)` — proxy that blocks mutations.
- JPA lazy-loading proxy — fetches data only on first access.
- Spring AOP proxy — intercepts calls for `@Transactional`, `@Cacheable`.

**Structural difference**: a Proxy often **creates or manages** the real object internally. A Decorator receives it **externally** as a component.

### [3-8-q2 | senior] How have lambdas, records, and sealed classes changed the way classic GoF patterns are implemented in modern Java?
**Lambdas** collapse many behavioural patterns to a single expression:
- **Strategy** → a lambda passed to a method. `list.sort((a, b) -> ...)`.
- **Command** → `Runnable` / `Supplier`.
- **Observer** → `Consumer<Event>` as a callback.

**Records (Java 16)** remove Builder boilerplate for simple value objects. If all fields are required and few — the canonical constructor of a record suffices:
```java
record Point(int x, int y) {}  // no PointBuilder needed
```

**Sealed classes (Java 17)** + switch pattern matching (Java 21) implement Visitor without a Visitor interface, with **compile-time exhaustiveness**:
```java
sealed interface Shape permits Circle, Square {}
double area = switch (s) {
    case Circle c -> Math.PI * c.r() * c.r();
    case Square sq -> sq.side() * sq.side();
};  // compiler verifies all cases
```
An Abstract Factory can return a sealed type, and the consumer uses pattern matching instead of polymorphic dispatch.

**Patterns don't disappear** — the syntax changes. Code shifts from class-based plumbing toward declarative, expression-oriented forms.
