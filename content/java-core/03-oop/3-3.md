## Title
Inheritance & super

## Summary
Inheritance is an IS-A relationship: `Dog extends Animal`. Java supports **single** class inheritance and **multiple** interface implementation. `super(...)` invokes the parent constructor; `super.method()` calls the parent's overridden method. Dispatch goes through the vtable by the object's runtime type — the foundation of runtime polymorphism.

## Deep Dive
## Relationships between classes

- **Association (has-a)** — one class uses another.
- **Aggregation** (weak has-a) — both sides exist independently. A book can move between libraries.
- **Composition** (strong has-a) — one class owns the other's lifecycle. Pages exist only with the book.
- **Inheritance (is-a)** — `Dog` is an `Animal`.
- **Realisation** — a class implements an interface.
- **Dependency** — use of a class in a method signature / parameter.

## What's inherited and what isn't

Inherited:
- Public, protected, package-private (same package) fields and methods.
- Private fields **exist** in the object's memory but aren't accessible by name from the child.

Not inherited:
- Constructors (must be invoked via `super(...)`).
- Private methods.
- Static methods are "logically inherited" but can be **hidden** (not overridden).

## Virtual method dispatch

For instance methods the JVM uses the **vtable**: on `ref.method()` it uses the object's **runtime type**, not the declared reference type. This is runtime polymorphism.

```java
Animal a = new Dog("Rex");
a.speak();  // calls Dog.speak(), not Animal.speak()
```

Static methods and fields use **compile-time binding** — the declared reference type wins.

## Overriding vs hiding

| | Method | Static method / Field |
|--|--------|------------------------|
| Mechanism | Override (dynamic binding) | Hide (static binding) |
| Dispatched by | Runtime type of the object | Compile-time type of the reference |
| `@Override` | Yes | No |

```java
class Parent { int x = 1; static String name() { return "Parent"; } }
class Child extends Parent { int x = 2; static String name() { return "Child"; } }

Parent p = new Child();
p.x;         // 1 (fields hide, by reference type)
p.name();    // "Parent" (static hides, by reference type)
```

## Override rules

- **Same signature** (name + parameters).
- **Covariant return type** (Java 5+) — narrower return allowed.
- Access modifier — **equal or wider**.
- Checked exceptions — **same or narrower**. `RuntimeException` is unrestricted.

> [!tip]
> **Always add `@Override`.** Without it, a typo creates a new method instead of overriding — the compiler stays silent and the bug shows up at runtime.

## `super`

Two roles:

1. **`super(...)`** — invoke the parent constructor. Must be the first statement. If omitted, the compiler inserts `super()` — this fails if the parent has no no-arg constructor.
2. **`super.method()`** — call the parent's overridden method. Useful to **augment** behaviour:
   ```java
   @Override public String speak() {
       return name + " barks! (" + super.speak() + ")";
   }
   ```

You **cannot** `super.super.method()` — Java only lets you go one level up.

## Why Java has no multiple class inheritance

**The diamond problem** — if two parents define the same method differently, which implementation wins? C++ solves it through explicit resolution, Java sidesteps it by forbidding multiple class inheritance.

**Interface default methods (Java 8+)** bring a controlled form of multiple behaviour inheritance. If two interfaces provide conflicting defaults, the compiler **forces** the implementing class to override — no ambiguity remains.

```java
interface A { default void m() { ... } }
interface B { default void m() { ... } }
class C implements A, B {
    @Override public void m() { A.super.m(); }  // explicit resolution
}
```

## `final` and `sealed`

- **`final class`** — cannot be inherited (`String`, all primitive wrappers).
- **`final method`** — cannot be overridden.
- **`sealed class / interface`** (Java 17+) — the middle ground: list the allowed subtypes with `permits`:

```java
sealed interface Shape permits Circle, Square, Triangle {}
```

Subclasses must be `final`, `sealed` (with their own permits), or `non-sealed`. The compiler knows the closed set and does **exhaustive pattern matching in switch with no default**.

## Fragile base class — why composition > inheritance

> [!production]
> A change in the base class can **silently break** subclasses that depend on it. Classic example (Bloch, Effective Java): `HashSet` → `InstrumentedHashSet` overriding `add` and `addAll`. `HashSet.addAll` internally calls `add` → overridden `add` is hit **twice**, the counter doubles. The internal call isn't documented as API — it's an implementation detail.

Rule: **prefer composition over inheritance** (Bloch). Wrap the behaviour you need in a field rather than extending the class. Use inheritance only when you own the base class and IS-A genuinely holds.

## Code
```java
public class Animal {
    private final String species;

    public Animal(String species) {
        this.species = species;
    }

    public String speak() {
        return species + " makes a sound";
    }

    // final prevents subclasses from overriding
    public final String getSpecies() {
        return species;
    }
}

public class Dog extends Animal {

    private final String name;

    public Dog(String name) {
        super("Canine");  // must be first statement
        this.name = name;
    }

    @Override
    public String speak() {
        // augment parent behavior
        return name + " barks! (" + super.speak() + ")";
    }

    public static void main(String[] args) {
        Animal a = new Dog("Rex");   // polymorphic reference
        System.out.println(a.speak());
        // "Rex barks! (Canine makes a sound)"

        // a.name  -- won't compile; 'name' is not in Animal

        System.out.println(a.getSpecies()); // "Canine" (final method)
    }
}
```

## Tip
Always add `@Override`. Without it, a typo in the name silently creates a new method instead of overriding — the compiler stays quiet and the bug only shows up at runtime.

## Spring
### concept
Inheritance & super

### springFeature
Spring CGLIB Proxies

### explanation
Spring AOP creates proxies by **subclassing** your bean's class at runtime with CGLIB (or JDK dynamic proxies if the bean implements an interface).

Consequences:
- The class **cannot be `final`** — CGLIB couldn't extend it. You'll get a startup error.
- Methods can't be `final` or `private` — the proxy couldn't override them to insert advice (`@Transactional`, `@Cacheable`, `@Async`, `@Secured`).
- **Self-invocation bypasses the proxy**. A `this.method()` call from inside the same class goes directly, not through the proxy — advice doesn't fire. Fix: extract the method into another bean, or use `AopContext.currentProxy()`.

Understanding inheritance mechanics explains these constraints.

## Interview
### [3-3-q0 | junior] What's the difference between method overriding and method hiding?
- **Overriding** — for **instance methods**. The JVM dispatches by the object's **runtime type** (dynamic binding). This is runtime polymorphism.
- **Hiding** — for **static methods and fields**. Uses the reference's **compile-time type** (static binding). A child declaring a static method with the same signature **hides** the parent's rather than overriding it.

```java
class Parent { static String tag() { return "P"; } }
class Child extends Parent { static String tag() { return "C"; } }

Parent p = new Child();
p.tag();  // "P" — static binds to the reference type
```

Fields behave the same: `p.field` is the parent's field even if the object is `Child`.

### [3-3-q1 | mid] Why doesn't Java support multiple class inheritance, and how do interface default methods change the picture?
**The diamond problem**: if two parents define the same method differently, which implementation do you inherit? Java avoids the ambiguity by forbidding multiple class inheritance.

**Interface default methods (Java 8+)** give a controlled form of multiple **behaviour** inheritance (not state — interfaces have no fields). If two interfaces provide conflicting defaults, the compiler **forces** an override in the implementing class — ambiguity is impossible:

```java
interface A { default void m() {} }
interface B { default void m() {} }
class C implements A, B {
    @Override public void m() { A.super.m(); }  // explicit resolution
}
```

This was the mechanism that let Java 8 add `stream()` to `Collection` without breaking every existing implementation.

### [3-3-q2 | senior] Explain sealed classes, their link to pattern matching, and how they improve traditional hierarchies.
**Sealed classes (Java 17)** restrict who may extend them via a `permits` list. The set of subtypes is **closed and known at compile time**.

```java
sealed interface Shape permits Circle, Square, Triangle {}
record Circle(double r) implements Shape {}
record Square(double side) implements Shape {}
record Triangle(double base, double h) implements Shape {}
```

Subclasses must be `final`, `sealed` (with their own `permits`), or `non-sealed`.

**Pattern matching in `switch`** uses the closedness — the compiler verifies exhaustiveness with no `default`:

```java
double area = switch (s) {
    case Circle c   -> Math.PI * c.r() * c.r();
    case Square sq  -> sq.side() * sq.side();
    case Triangle t -> t.base() * t.h() / 2;
};
```

Adding a new permitted variant is a **compile error in every switch** over that type. You cannot miss a case.

Sealed types resolve the tension between **openness** (anyone can extend) and **control** (the author wants a fixed set of variants) — Java's take on algebraic data types. Pairs well with records.

### [3-3-q3 | mid] What's the fragile base class problem and why do people say 'prefer composition over inheritance'?
**Fragile base class** — a change in the base class can silently break subclasses that relied on an **implementation detail** rather than the documented API.

Bloch's classic (Effective Java):

```java
class InstrumentedHashSet<E> extends HashSet<E> {
    int added = 0;
    @Override public boolean add(E e) { added++; return super.add(e); }
    @Override public boolean addAll(Collection<? extends E> c) {
        added += c.size(); return super.addAll(c);
    }
}
```

Internally `HashSet.addAll` calls `add` per element. Our overridden `add` already increments → the counter doubles. This is an undocumented implementation detail you cannot rely on.

**Composition** makes dependencies explicit. Instead of extending, wrap:

```java
class InstrumentedSet<E> implements Set<E> {
    private final Set<E> delegate;
    int added = 0;
    public boolean add(E e) { added++; return delegate.add(e); }
    public boolean addAll(Collection<? extends E> c) {
        int n = 0; for (E e : c) if (add(e)) n++; return n > 0;
    }
    // ... other methods delegate to delegate
}
```

Use inheritance only when you own the base class and IS-A genuinely holds.
