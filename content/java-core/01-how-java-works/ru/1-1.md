## Title
JVM Architecture

## Summary
JVM (Java Virtual Machine) -- виртуальная машина, исполняющая байт-код Java. Она является ядром платформы Java: загрузка классов, управление памятью (Heap, Metaspace, Stack), сборка мусора и JIT-компиляция -- все это происходит внутри JVM. Понимание архитектуры JVM критично для диагностики проблем производительности и утечек памяти.

## Deep Dive
## Три подсистемы JVM

JVM — абстрактная вычислительная машина, исполняющая байт-код Java. Состоит из трёх подсистем:

- **Class Loader Subsystem** — загрузка (`.class` → метаданные в Metaspace), связывание (верификация → подготовка static-полей → резолвинг символических ссылок) и инициализация (`<clinit>`).
- **Runtime Data Areas** — области памяти во время исполнения.
- **Execution Engine** — интерпретатор + JIT + сборщик мусора.

## Runtime Data Areas

**Общие между всеми потоками (shared):**

- **Heap** — все объекты и массивы. Делится на Young Generation (Eden + S0 + S1) и Old Generation.
- **Method Area (Metaspace с Java 8)** — метаданные классов, Runtime Constant Pool, `static`-поля, байт-код методов. Находится в native-памяти, не ограничена `-Xmx`.

**Per-thread (у каждого потока свои):**

- **JVM Stack** — фреймы методов (локальные переменные + operand stack + return address).
- **PC Register** — адрес текущей инструкции байт-кода.
- **Native Method Stack** — для вызовов через JNI.

> [!gotcha]
> Локальная переменная-ссылка живёт в стеке, но сам объект, на который она указывает, — в куче. Поэтому `static`-поле — НЕ thread-safe автоматически: оно лежит в Metaspace (shared), и все потоки пишут в одну ячейку.

## Execution Engine

- **Интерпретатор** — выполняет байт-код по инструкции. Старт мгновенный, но медленный.
- **JIT-компилятор (HotSpot)** — отслеживает горячие методы и компилирует в native-код. Tiered compilation: C1 (быстро, Tier 1–3) → C2 (агрессивные оптимизации: inlining, escape analysis, loop unrolling, Tier 4).
- **OSR (On-Stack Replacement)** — может скомпилировать метод прямо во время его исполнения (важно для длинного цикла в `main`).
- **Deoptimization** — если оптимистическое предположение сломано (например, появился новый подкласс), JVM откатывается к интерпретатору.

Именно из-за tiered compilation Java-приложения «прогреваются»: первые запросы — холодные и медленные.

## Области памяти, о которых часто забывают

`-Xmx` ограничивает только heap. Реальное RSS-потребление процесса складывается из:

- Heap (`-Xmx`)
- Metaspace (`-XX:MaxMetaspaceSize` — ОБЯЗАТЕЛЬНО задавать в проде)
- CodeCache для JIT-кода (`-XX:ReservedCodeCacheSize`, по умолчанию 240MB)
- Direct buffers (`-XX:MaxDirectMemorySize`, используется NIO/Netty)
- Стеки потоков (native-память: N потоков × `-Xss`)
- GC bookkeeping, card tables, symbol tables

Поэтому RSS процесса обычно в 1.5–2× больше `-Xmx`.

> [!production]
> Обязательный минимум в проде: `-Xmx`, `-XX:MaxMetaspaceSize`, `-XX:+HeapDumpOnOutOfMemoryError`, `-XX:HeapDumpPath`. Для отладки native-памяти: `-XX:NativeMemoryTracking=summary` + `jcmd <pid> VM.native_memory summary`.

## Code
```java
// Exploring JVM memory areas at runtime
public class JvmArchitectureDemo {
    // Static field -> stored in Method Area (Metaspace)
    private static final String APP_NAME = "JVM Demo";

    // Instance field -> stored on the Heap (as part of the object)
    private int[] largeArray = new int[1_000_000];

    public static void main(String[] args) {
        // 'demo' reference is on the stack; the object is on the heap
        JvmArchitectureDemo demo = new JvmArchitectureDemo();

        // Runtime memory inspection
        Runtime rt = Runtime.getRuntime();
        System.out.println("Max heap:   " + rt.maxMemory() / 1024 / 1024 + " MB");
        System.out.println("Total heap: " + rt.totalMemory() / 1024 / 1024 + " MB");
        System.out.println("Free heap:  " + rt.freeMemory() / 1024 / 1024 + " MB");
        System.out.println("Used heap:  " +
            (rt.totalMemory() - rt.freeMemory()) / 1024 / 1024 + " MB");

        // Thread stack info
        System.out.println("\nActive threads: " + Thread.activeCount());
        System.out.println("Current thread stack depth: " +
            Thread.currentThread().getStackTrace().length);

        // Demonstrate StackOverflowError
        try {
            infiniteRecursion(0);
        } catch (StackOverflowError e) {
            System.out.println("\nStackOverflowError caught!");
            System.out.println("Stack frames before overflow vary by JVM and -Xss setting");
        }
    }

    // Each call adds a frame to the thread's JVM Stack
    private static void infiniteRecursion(int depth) {
        infiniteRecursion(depth + 1);
    }
}
```

## Tip
Когда слышите вопрос 'где X живет в памяти?', думайте так: статика/метаданные классов -> Metaspace, объекты -> Heap, локальные переменные-ссылки -> Stack (но объект, на который они указывают, все равно в Heap).

### explanation
Spring singleton-бины живут как объекты в Heap в течение всей жизни приложения, на них держит ссылку `ApplicationContext`. Prototype-бины тоже лежат в Heap, но становятся доступными для GC сразу после того, как ваш код отпустит ссылку.

Почему это важно для диагностики Spring Boot:
- Много прокси-классов (AOP, CGLIB) раздувают Metaspace -- каждая прокси-обертка это отдельный сгенерированный класс.
- Слишком много singleton-бинов с большим state'ом исчерпают Heap.
- `@Cacheable` без политики вытеснения -- классическая утечка: коллекция растет, ссылки держатся из singleton-сервиса, GC не может ничего освободить.
- ApplicationContext, не закрытый правильно при перезагрузке (актуально для embedded Tomcat + hot-reload), не даст выгрузиться classloader'у -> Metaspace leak.

## Interview
### [1-1-q0 | junior] Какие основные области данных времени выполнения есть в JVM и какие из них общие для всех потоков?
JVM имеет пять областей данных времени выполнения.

Общие между потоками (shared):
• Heap -- здесь живут все объекты (new Foo() -> Heap). Делится на Young (Eden + S0 + S1) и Old Generation.
• Method Area / Metaspace -- метаданные классов, Runtime Constant Pool, static поля, байт-код методов.

Per-thread (у каждого потока свои):
• JVM Stack -- стек фреймов методов (локальные переменные + operand stack + return address).
• PC Register -- адрес текущей инструкции байт-кода.
• Native Method Stack -- для вызовов через JNI.

Почему это важно: в shared-областях нужна синхронизация (два потока могут одновременно писать в один объект на Heap -- отсюда race conditions), а per-thread области безопасны по умолчанию. Поэтому локальные примитивы thread-safe, а поля объекта на Heap -- нет.

Частая ловушка: static поле 'thread-safe потому что static' -- неверно. Static живет в Metaspace (shared), то есть к нему тоже одновременно обращаются все потоки.

### [1-1-q1 | junior] Где в памяти JVM живут: примитивная локальная переменная, объект, ссылка на объект, static поле, литерал строки? Разберите на примере.
Рассмотрим код:

```java
class Order {
  static int counter = 0;        // (1)
  String code = "A-001";         // (2)
  public void process() {
    int qty = 5;                 // (3)
    Order other = new Order();   // (4)(5)
  }
}
```

(1) `counter` -- static -> Metaspace (поле класса, одно на всю JVM).
(2) `code` -- instance поле -> живет внутри объекта Order на Heap.
(3) `qty` -- локальный примитив -> JVM Stack текущего потока, внутри фрейма process().
(4) `other` -- локальная ссылка -> JVM Stack (4 или 8 байт -- сам указатель).
(5) `new Order()` -- сам объект -> Heap (в Eden при первом создании).

Строковый литерал `"A-001"` -> String Pool. В Java 7+ пул перенесен в Heap; в Metaspace хранятся только имена классов и метаданные.

Ключевой момент: ссылка и объект -- это два разных места в памяти. Когда метод завершается, стек-фрейм (с `qty` и `other`) автоматически уничтожается, но объект Order на Heap останется, пока до него есть reachable-ссылки -- затем GC его заберет.

### [1-1-q2 | mid] В чем разница между StackOverflowError и OutOfMemoryError? Можно ли получить OOM из-за стека? Как это диагностировать в проде?
**StackOverflowError**: поток превысил глубину своего стека (обычно бесконечная или слишком глубокая рекурсия). Размер регулируется `-Xss` (по умолчанию ~512KB-1MB).

```java
void recurse() { recurse(); } // SOE через ~10-20K фреймов
```

**OutOfMemoryError** -- JVM не может выделить память. Под этим названием скрывается несколько разных ошибок:
• `Java heap space` -- закончился -Xmx. Часто -- утечка памяти (коллекция растет без ограничений).
• `GC overhead limit exceeded` -- JVM тратит >98% времени на GC и освобождает <2% памяти.
• `Metaspace` -- слишком много загруженных классов (classloader leak в app-серверах).
• `unable to create new native thread` -- OS не может выделить память под стек нового потока. Это именно OOM из-за стека, но не StackOverflow! Если у вас тысячи потоков, каждый со стеком по 1MB, это уже гигабайты native-памяти.
• `Direct buffer memory` -- исчерпан `-XX:MaxDirectMemorySize` (NIO, Netty).

Диагностика в проде:
1. Всегда запускайте с `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/logs/heap`.
2. Анализ дампа: Eclipse MAT, VisualVM, JProfiler -- смотрите Leak Suspects.
3. Live-диагностика: `jcmd <pid> GC.heap_info`, `jstat -gc <pid> 1s`, `jmap -histo:live <pid>`.
4. Для 'unable to create thread': `ulimit -a`, `cat /proc/<pid>/status | grep Threads`.

### [1-1-q3 | mid] Что такое JIT-компиляция? Почему Java-приложения 'прогреваются', и как это влияет на SLA и бенчмарки?
Execution Engine JVM работает в двух режимах:

**Интерпретатор** -- читает байт-код по инструкции и выполняет. Запускается мгновенно, но медленный (каждая итерация цикла -- диспетчеризация).

**JIT-компилятор** (HotSpot) -- отслеживает 'горячие' методы (счетчик вызовов > порога, по умолчанию ~10K) и компилирует их в native-код. Есть tiered compilation:
• Tier 0: интерпретатор.
• Tier 1-3: C1 compiler (быстрая, простая компиляция).
• Tier 4: C2 compiler (агрессивные оптимизации: inlining, escape analysis, loop unrolling).

Отсюда warm-up: первые запросы идут через интерпретатор -- медленно. По мере прогрева C1 -> C2 компилируют горячие пути, и latency падает в 10-100 раз.

Последствия для прода:
• Первые N секунд после старта сервис 'холодный'. Kubernetes readiness probe с низким timeout может убить pod до прогрева.
• Canary-деплой: новая инстанция медленнее старых -- может завалить P99 latency.
• Микробенчмарки: НИКОГДА не меряйте производительность без warm-up итераций. Используйте JMH -- он делает warm-up автоматически.
• Решения: AOT-компиляция (GraalVM native-image), CDS (Class Data Sharing), прогрев по checklist'у перед включением трафика.

Флаги для наблюдения: `-XX:+PrintCompilation`, `-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining`.

### [1-1-q4 | senior] Почему в Java 8 PermGen заменили на Metaspace? Какие production-последствия это имеет, и какие флаги обязательно настраивать?
**PermGen (до Java 8)** -- фиксированный регион внутри Heap для метаданных классов, interned strings и static-ов. Проблема: жесткий лимит `-XX:MaxPermSize` (по умолчанию 64-82MB). В app-серверах (Tomcat, JBoss) с частыми hot-deploy'ами classloader'ы не всегда выгружались -> PermGen быстро переполнялся -> 'OutOfMemoryError: PermGen space'. Классика -- постоянно после ~10-20 передеплоев.

**Metaspace (Java 8+)** -- метаданные классов перенесены в native memory (вне Heap). Ключевые изменения:
• По умолчанию размер не ограничен -- растет, пока есть RAM у процесса.
• Interned strings переехали в обычный Heap (можно дампить, GC-ить).
• Class metadata тоже может освобождаться при выгрузке classloader'а.

Production-последствия, почему это опасно:
• Старый PermGen OOM падал громко и рано. Новый Metaspace leak тихо ест native memory, пока не убьет всю ноду (OOM killer в Linux).
• Без лимита на Metaspace один баг с classloader'ом может съесть всю память хоста и положить соседние сервисы.

Обязательные флаги в проде:
```
-XX:MaxMetaspaceSize=512m          # верхняя граница -- must have
-XX:MetaspaceSize=256m             # начальный размер, снижает ре-трешолды
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/logs/heap
```

Мониторинг: `jcmd <pid> VM.metaspace` или `jstat -gcmetacapacity <pid>`. Если Metaspace растет линейно со временем без стабилизации -- classloader leak. Типичные причины: ThreadLocal, который держит ссылку на класс; неправильная отписка от JDBC-драйверов; Spring-контексты, которые не закрываются.

### [1-1-q5 | senior] Продовский сервис получает OutOfMemoryError: Java heap space через 6 часов после старта. Heap растет медленно и монотонно. Как вы пошагово диагностируете утечку памяти?
План действий:

**1. Убедиться, что дамп уже есть.** Сервис должен запускаться с `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/logs/heap/`. Если нет -- немедленно добавить и дождаться следующего падения (или снять дамп вручную до падения: `jcmd <pid> GC.heap_dump /tmp/leak.hprof`).

**2. Подтвердить, что это именно leak, а не нормальный рост.** Смотрим GC-логи (`-Xlog:gc*:file=gc.log` в Java 9+). Признаки утечки:
• После Full GC used heap не падает до прежнего уровня (монотонный рост baseline).
• Интервалы между GC сокращаются, длительность растет.
• Пик Old Gen после GC приближается к max heap.

**3. Анализ дампа в Eclipse MAT:**
• Leak Suspects Report -- автоматически показывает подозрительные объекты.
• Dominator Tree -- какой объект удерживает больше всего retained heap.
• Histogram -- топ классов по количеству и размеру. Обычные подозреваемые: HashMap$Node, char[]/byte[], ArrayList.
• Path to GC Roots (exclude weak/soft) для подозрительных объектов -- покажет, кто держит ссылку.

**4. Типичные виновники:**
• Статическая коллекция используется как кеш без eviction (Map кидает записи, но не удаляет).
• ThreadLocal в thread-pool'е -- threads живут вечно, ThreadLocal-значения накапливаются.
• Listener'ы / observer'ы без отписки.
• Неправильно закрытые ресурсы (Statement, ResultSet, InputStream).
• Внутренние классы (inner class) неявно держат ссылку на enclosing instance.

**5. Быстрая live-проверка без дампа:**
```
jcmd <pid> GC.class_histogram | head -30   # топ-классы
jstat -gcutil <pid> 1s                     # метрики GC в реальном времени
jmap -histo:live <pid> | head -30          # после форсированного GC
```

**6. Фикс + proof.** После правки запустить нагрузочный тест на несколько часов с мониторингом Old Gen -- baseline должен стабилизироваться, а не расти.

### [1-1-q6 | senior] Вы выставили `-Xmx2g`, но `top` показывает, что процесс Java занимает 3.5GB RSS. Куда ушло еще 1.5GB и как это диагностировать?
`-Xmx` ограничивает ТОЛЬКО Java heap. Реальное потребление памяти JVM-процессом = heap + множество других регионов, каждый со своим бюджетом:

**Из чего складывается RSS:**
- **Heap** (`-Xmx`) -- то, что мы задали.
- **Metaspace** -- метаданные классов в native memory. Без `-XX:MaxMetaspaceSize` может расти неограниченно.
- **CodeCache** -- скомпилированный JIT-код. Default `-XX:ReservedCodeCacheSize=240m`. Может забиться в long-running сервисах с большим code path.
- **Direct Memory** -- `ByteBuffer.allocateDirect()`, Netty ByteBufs, некоторые compression-либы. Ограничение: `-XX:MaxDirectMemorySize` (по умолчанию = `-Xmx`).
- **Thread stacks** -- native memory: N потоков × `-Xss` (по умолчанию ~1MB). 1000 потоков -- это уже ~1GB.
- **GC структуры** -- card tables, remembered sets (для G1/ZGC), marking bitmaps.
- **Symbol Table, String Table** -- native.
- **JNI-аллокации** -- нативные либы (JDBC-драйверы, libsnappy, jemalloc) могут аллоцировать вне контроля JVM.
- **Glibc / malloc fragmentation** -- аллокатор может держать "free" блоки, не возвращая их ОС.

**Диагностика:**

```
# 1. Включить Native Memory Tracking (нужен рестарт)
-XX:NativeMemoryTracking=summary

# 2. Смотреть breakdown
jcmd <pid> VM.native_memory summary
jcmd <pid> VM.native_memory summary.diff   # разница с baseline

# 3. Состояние всех регионов
jcmd <pid> VM.info
jcmd <pid> VM.flags                        # все активные флаги
jstat -gcutil <pid>                        # использование GC

# 4. Потоки
jcmd <pid> Thread.print | grep -c '^"'    # количество потоков

# 5. Для direct memory (JMX bean)
jcmd <pid> GC.run                          # форсировать GC
# и смотреть java.nio:type=BufferPool через JMX
```

**Типичные виновники "невидимой" памяти:**
- Netty / Kafka client -- direct buffers не ограничены.
- JDBC-пул с тысячами connection'ов -- каждый держит native ресурсы.
- Неправильная настройка thread-pool'а (Tomcat maxThreads=500+) -- стеки тихо сжирают GB.
- Glibc arena fragmentation -- решается `MALLOC_ARENA_MAX=2` env variable или переходом на jemalloc.

**Правило для прода:** rough-бюджет RSS = `-Xmx` × 1.5-2×. Если больше -- разбирайтесь с NMT.
