## Title
ArrayList vs LinkedList

## Summary
Коллекции -- это структура данных, набор каких-либо объектов. Интерфейс List определяет упорядоченную коллекцию с допустимыми дубликатами. ArrayList -- инкапсулирует обычный массив с динамически изменяемой длиной. LinkedList -- состоит из узлов со ссылками на следующий и предыдущий узел (также реализует Deque).

## Deep Dive
Коллекции -- это структура данных, набор каких-либо объектов. Данными могут быть числа, строки, объекты пользовательских классов и т.п.

На верхушке иерархии стоит интерфейс Iterable. Который содержит в себе Iterator (отвечает за переход к следующему элементу), forEach(), Spliterator.

Интерфейсы Collection и Map разделяют коллекции на две части по типу хранения: последовательные наборы элементов и наборы пар «ключ-значение».

Интерфейс List (список) определяет упорядоченную коллекцию, в которой допустимы дублирующие значения и гарантирует порядок добавления.

- ArrayList -- инкапсулирует обычный массив, длина которого динамически изменяется. Элементы пронумерованы, к ним можно обратиться по индексу. Примечание: нет механизма автоматического сжатия capacity.
- LinkedList -- состоит из узлов, каждый из которых содержит как собственные данные, так и две ссылки на следующий и предыдущий узел. Также реализует интерфейс Deque.
- Vector -- реализация динамического массива объектов, методы которой синхронизированы. Считается устаревшим.
- Stack -- реализация стека LIFO.

Queue (очередь) предназначена для хранения элементов с предопределенным способом вставки и извлечения FIFO. Методы: offer, poll, peek / add, remove, element.
- PriorityQueue -- предоставляет возможность управлять порядком элементов при помощи Comparator, либо сохраняет элементы с использованием natural ordering.
- ArrayDeque -- реализация интерфейса Deque, двусторонняя очередь.

## Code
```java
import java.util.*;

public class ListComparison {
    public static void main(String[] args) {
        // ArrayList: backed by Object[] array
        List<String> arrayList = new ArrayList<>(100); // preallocate
        arrayList.add("A");          // amortized O(1)
        arrayList.get(0);            // O(1) random access
        arrayList.add(0, "B");       // O(n) - shifts elements
        arrayList.remove(0);         // O(n) - shifts elements

        // LinkedList: doubly-linked nodes
        LinkedList<String> linkedList = new LinkedList<>();
        linkedList.add("A");         // O(1) at tail
        linkedList.addFirst("B");    // O(1) at head
        linkedList.get(5);           // O(n) - must traverse!
        linkedList.removeFirst();    // O(1) at head

        // ArrayDeque: usually better than LinkedList for stack/queue
        Deque<String> stack = new ArrayDeque<>();
        stack.push("A");
        stack.push("B");
        stack.pop(); // "B"

        // Performance: ArrayList with preallocated capacity
        int n = 1_000_000;
        List<Integer> preAllocated = new ArrayList<>(n);
        for (int i = 0; i < n; i++) {
            preAllocated.add(i); // no resizing needed
        }

        // Iteration with ListIterator (LinkedList's strength)
        ListIterator<String> it = linkedList.listIterator();
        while (it.hasNext()) {
            String val = it.next();
            if (val.equals("target")) {
                it.remove(); // O(1) removal during iteration
                it.add("replacement"); // O(1) insertion
            }
        }
    }
}
```

## Tip
In interviews, confidently say 'ArrayList is the default choice for List, and ArrayDeque for stack/queue. LinkedList has almost no practical advantage due to cache locality.' This shows real-world understanding.

## Interview
### [6-1-q0 | junior] What are the time complexities for get(), add(), and remove() in ArrayList vs LinkedList?
ArrayList: get() is O(1), add() at end is amortized O(1), add()/remove() in middle is O(n) due to shifting. LinkedList: get() is O(n) because it must traverse, add()/remove() at head or tail is O(1), but add()/remove() by index is O(n) because finding the position requires traversal. The key nuance: even though LinkedList removal is O(1) once you have the node, finding that node is O(n).

### [6-1-q1 | mid] Why is ArrayList generally preferred over LinkedList even for frequent insertions?
CPU cache locality. ArrayList stores elements in contiguous memory, enabling efficient CPU cache prefetching -- sequential memory access is orders of magnitude faster than random access. LinkedList nodes are scattered across the heap, causing constant cache misses during traversal. Additionally, each LinkedList node requires ~32 bytes of overhead (object header + two pointers), significantly increasing memory usage. Benchmarks show ArrayList outperforms LinkedList for most operations including mid-list insertion until very large sizes, because the cost of shifting a contiguous array is hidden by hardware optimizations.

### [6-1-q2 | senior] How does ArrayList grow internally, and how would you optimize for a known large dataset?
When the internal array is full, ArrayList creates a new array of size oldCapacity + (oldCapacity >> 1) -- a 50% increase. It then copies all elements via System.arraycopy() (a native, highly optimized memcpy). For known sizes, pass the capacity to the constructor: new ArrayList<>(1_000_000). This avoids multiple resize-and-copy operations. Without preallocation, growing from 10 to 1M elements requires approximately 30 resizes, each involving a full array copy. Also consider using trimToSize() after bulk loading to reclaim unused capacity. For primitive types, prefer specialized collections (Eclipse Collections IntArrayList or arrays) to avoid boxing overhead.
